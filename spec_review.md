# Spec Review 标准

本文档定义 WebEnvoy 仓库的正式 FR / 架构规约审查标准。

本文档中的 “spec” 默认指**整套正式 FR 套件**，而不是只指 `spec.md`。当前最小正式套件至少包含：

- `spec.md`
- `plan.md`
- `TODO.md`

Spec review 的目标不是检查代码风格，而是回答两个问题：

1. 这个事项是否被正确定义成了一个值得进入开发的正式契约
2. 该契约是否已经具备进入实现阶段的边界、验收与风险清晰度

## 适用范围

以下情况默认需要走 spec review：

- 新建或修改 `docs/dev/specs/FR-XXXX-*/`
- 修改 `docs/dev/architecture/**` 且会影响实现准入或架构边界
- 修改 `vision.md`、根级 `AGENTS.md`、`docs/dev/AGENTS.md` 中与研发漏斗、实现准入、正式契约相关的规则

以下情况通常不需要独立 spec review：

- 纯文案修正、错别字、失效链接
- 不改变正式契约语义的说明性补充
- 已通过 spec review 的 FR 下，仅更新实现进度的 `TODO.md`

## 审查基线

进行 spec review 时，至少对照以下材料：

- `vision.md`
- 根级 `AGENTS.md`
- `docs/dev/AGENTS.md`
- 相关架构文档
- 对应 `docs/dev/roadmap.md`
- 当前 GitHub Issue / Sprint / PR 描述

如果规约与上述基线冲突，应直接阻断，不做“先合后补”。

## 准入要求

一个核心事项进入 spec review 前，至少应满足：

- GitHub 上已有对应 Issue，且该事项已被纳入正确阶段或 backlog
- 当前事项成熟度至少达到 `spec-ready`；若仍处于 `spike`，应先收敛证据边界再进入正式评审
- 正式 FR 目录已经建立
- 目录内至少包含 `spec.md`、`plan.md`、`TODO.md`
- 当前 PR 以规约审查为主，不混入实现代码

### 正式套件最低内容要求

#### `spec.md`

至少应清楚覆盖：

- 背景
- 目标
- 非目标
- 功能需求
- `Given / When / Then` 验收场景
- 异常与边界场景
- 验收标准
- 依赖与前置条件

#### `plan.md`

至少应清楚覆盖：

- 实施目标
- 分阶段拆分
- 实现约束
- 测试与验证策略
- TDD 范围
- 并行 / 串行关系
- 进入实现前条件

`plan.md` 的最低要求不是泛泛“写个计划”，而是必须正面回答这 7 个问题：

1. 这次实现具体交付什么
2. 每个阶段产出什么，先后依赖是什么
3. 哪些边界不能碰
4. 怎么证明实现完成
5. 哪些模块先写测试，哪些不强制
6. 哪些 issue / 工作项可并行，哪些是前绪阻塞
7. spec review 通过后，才允许进入哪些实现动作

推荐直接使用以下标题，避免审查时靠猜：

- `## 实施目标`
- `## 分阶段拆分`
- `## 实现约束`
- `## 测试与验证策略`
- `## TDD 范围`
- `## 并行 / 串行关系`
- `## 进入实现前条件`

#### `TODO.md`

至少应支持：

- spec review 阶段待办
- review blockers 跟踪
- 进入实现前动作

#### 状态型 runtime 事项补充要求

若事项涉及会话 / 锁 / 进程活性 / 断连 / 恢复 / 并发隔离等状态型 runtime 问题，正式套件还必须额外说明：

- 健康矩阵
  - 哪些信号决定 `healthy`、`disconnected`、`recoverable`、`blocked`
- 恢复路径
  - 同 run 重试、幂等 `start/stop`、孤儿进程回收、失败后状态收敛
- 最小验证矩阵
  - 至少覆盖控制进程死 / 浏览器活、锁仍持有但控制链断开、ready 信号陈旧、并发争抢等状态组合

若无法从 `plan.md`、`risks.md` 或相关契约中判断这些内容，应视为未达到进入实现的清晰度。

### 按需文档触发规则

不是所有 FR 都要补齐所有可选文档。只有在它们能显著减少歧义、锁定边界或暴露风险时，才应纳入正式套件。

#### `contracts/`

以下情况默认必须出现：

- 新增或修改稳定的机器接口、跨进程协议或跨模块契约
- 影响 CLI 输入输出、退出码、错误结构、Native Messaging、扩展与运行时通信
- 适配器接口、结构化返回、共享 payload 会被多个模块或后续 FR 依赖

以下情况通常不应单独创建：

- 仅限单文件内部实现细节
- 不形成稳定边界的临时辅助结构
- 只是在 `spec.md` 中一句话即可说清的局部参数

最低要求：

- 至少说明边界名称、生产者 / 消费者、版本或兼容策略
- 明确输入、输出、错误或状态返回
- 说明字段约束、必填项、空值与异常返回
- 给出最小示例，能支撑开发与测试对齐

若创建 `contracts/`，其中每份文档都必须承担一个明确边界，不能只放空目录或占位文件。

#### `data-model.md`

以下情况默认必须出现：

- 新增或修改持久化模型、SQLite schema、迁移、索引或清理策略
- 引入身份、会话、任务、运行记录、配置、缓存等共享实体
- 同一数据结构会被多个模块、契约或测试共同依赖

以下情况通常不应单独创建：

- 纯展示型文档，没有稳定数据实体
- 仅限局部函数内的临时对象
- 不涉及持久化、共享状态或跨模块语义的数据拼装

最低要求：

- 列出核心实体或表，以及它们的职责
- 说明关键字段、必填 / 可空、唯一性或约束
- 说明状态流转、生命周期或保留策略
- 如涉及迁移、兼容、回填或清理，必须写清影响面

#### `research.md`

以下情况默认必须出现：

- 正式契约依赖未稳定的第三方行为、反检测约束、签名机制或浏览器能力验证
- 存在两个以上可行方案，且需要记录为何选择当前方案
- 仍有关键未知项，但已决定进入正式 spec，需要把未知项与证据分开管理

以下情况通常不应单独创建：

- 只是把背景材料复制进仓库
- 研究结论已全部沉淀进 `spec.md` / `plan.md`，且没有剩余不确定性
- 试图用研究笔记替代正式契约

最低要求：

- 研究问题与上下文
- 关键信息来源或实验输入
- 结论与对当前 FR 的影响
- 未解决问题、失效条件或后续验证动作

`research.md` 必须服务于正式套件，不能成为“以后再看”的资料堆。

#### Spike 类事项的证据分层

若 FR 的主要任务是先冻结研究结论或第三方行为边界，再决定是否进入实现，审查时应显式区分以下证据层：

- `primary`
  - 第一手、可复核、且与 WebEnvoy 正式 runtime / profile / contract 边界一致的首选路径
- `candidate`
  - 看起来成立，但仍缺少复现、字段冻结或正式边界对齐的候选路径
- `fallback`
  - 可保留为降级或保底路径，但不自动等于实现准入
- `admission_ready`
  - 后续实现 FR 可直接消费的正式输入边界

审查规则：

- `fallback` 证据可以作为风险收口，但不能替代 `admission_ready`
- 依赖仓库外临时浏览器现场、手工 profile、不可复核外部环境的结论，默认不得直接判为 `admission_ready`
- 若文档没有写清哪些证据只是研究结果、哪些证据才是实现输入，应直接 `REQUEST_CHANGES`

#### `risks.md`

以下情况默认必须出现：

- 涉及账号、安全、风控、真实写入、删除、迁移、并发竞争或不可逆动作
- 修改公共契约、执行链路、数据读写或会影响多个 FR 的共享基座
- 需要明确 stop-ship 条件、降级策略、人工介入或回滚路径

以下情况通常不应单独创建：

- 低风险、局部、可完全由 `plan.md` 覆盖的文档性调整
- 没有显著副作用、也不触及共享基座的纯只读微特性

最低要求：

- 列出关键风险、触发条件与影响
- 说明当前缓解手段、观察信号与剩余风险
- 说明失败时如何停下、回滚、降级或转人工

`risks.md` 不要求做成完整风控体系，但必须让实现者和评审者知道什么不能失手。

## 必查维度

### 1. 范围与阶段一致性

- 是否属于当前阶段目标
- 是否提前把后续 Phase 的内容混入当前 FR
- 是否把 backlog 讨论稿误写成正式契约
- 当前事项成熟度是否与目标动作一致；仍处于 `spike` 的事项是否被误写成进入实现的承诺

### 2. 问题定义是否成立

- 背景是否清楚说明为什么要做
- 目标是否聚焦在用户价值或运行时能力，而非技术堆砌
- 非目标是否明确，足以限制范围蔓延

### 3. 契约完整性

- `spec.md` 是否说清做什么、做到什么程度、如何验收
- `plan.md` 是否围绕 7 个必答问题形成了可执行计划，而不是泛泛路线图
- `TODO.md` 是否足以支持跨会话恢复
- 按风险和复杂度应出现的附加文档是否已经出现
- 已出现的附加文档是否真的承载信息，而不是空壳占位

### 3.1 价值与功能连线

- 每个主要功能点是否都能直接支撑当前业务目标或阶段目标
- 是否存在与当前阶段目标弱相关的功能点
- 如某功能被删除或延期，主闭环是否仍成立

对目标支撑很弱、但会显著增加实现或维护面的功能，应标记为疑似 Scope Creep。

### 4. 架构一致性

- 是否与系统设计、执行边界、CLI-first、浏览器内执行红线一致
- 是否误引入上层系统能力
- 是否把 L1/L2/L3 的边界写乱

### 5. 依赖与并行关系

- 是否明确硬依赖
- 是否明确可并行项
- 是否避免把阻塞项和后续验证项混成同一承诺
- 是否把“前绪阻塞”和“实现阶段可并行切片”区分清楚

### 5.1 关闭语义与实现承诺

- 当前 PR / Issue 描述是否把 `spike` 或 `spec-ready` 事项提前写成完成闭环
- 是否错误使用 `Fixes #...` 制造提前关闭语义
- 是否把“证据已冻结”误写成“实现已完成”或“可直接开始实现”

### 6. 主流程闭环

- Happy path 是否从前置条件、动作、结果、观测形成闭环
- 是否存在“只能触发，不能确认结果”的单向路径
- 是否存在“能创建不能撤销/终止/恢复”的逻辑死胡同

### 7. 异常与边界场景

审查时不要只检查文档写了什么，还要反向生成至少以下 4 类问题：

1. 网络与并发：弱网、断网、重复提交、并发操作同一资源
2. 空与极值：空状态、极大/极小输入、超长文本、极端数量
3. 权限与状态异常：无权限、Session 失效、账号异常、人工介入
4. 外部依赖故障：第三方接口超时、错误返回、返回结构异常

若正式套件没有说明这些关键异常如何表现、如何返回、是否需要人工介入，应视为缺口。

### 7.1 Spike 证据与正式输入边界

- 是否区分 `primary`、`candidate`、`fallback`、`admission_ready`
- 是否把 `fallback` 误写成正式实现准入
- `admission_ready` 证据是否真正绑定 WebEnvoy 的正式 runtime / profile / contract 边界
- 是否仍依赖仓库外不可复核现场才能成立

### 8. 验收与验证

- 验收标准是否可验证
- `GWT` 场景是否足以支撑 QA 与开发对齐，而不是只写抽象口号
- 是否定义了与风险相匹配的验证证据
- 是否避免只写“完成实现”这类不可审查的验收口径

### 9. 测试策略与 TDD 范围

- `plan.md` 是否说明测试分层，而不是只写“补测试”
- 是否明确哪些模块默认采用 TDD，哪些部分暂不强制
- TDD 范围是否与模块稳定性、公共契约和风险等级相匹配

### 9.1 进入实现前条件

- `plan.md` 是否明确写出 spec review 通过后，哪些动作才能开始
- 是否避免在文档层默认跳过 review 直接编码
- 是否把进入实现前条件写成可判断、可执行的门槛

### 10. 风险与回滚

- 是否识别高风险目录、公共契约、数据读写、安全、账号、执行链路等风险
- 是否说明失败时如何停下、回滚或重新收口

### 10.1 状态型 runtime 健康矩阵与恢复

- 若事项涉及状态型 runtime，是否已写清健康判定矩阵
- 是否定义断连、锁占用、孤儿进程、ready 信号陈旧时的收敛路径
- 是否给出与状态复杂度相匹配的最小测试矩阵

### 11. 按需文档充分性

- 涉及稳定接口时，是否缺少 `contracts/`
- 涉及共享或持久化实体时，是否缺少 `data-model.md`
- 依赖关键未知项或第三方验证时，是否缺少 `research.md`
- 涉及高风险链路时，是否缺少 `risks.md`
- 这些文档是否最小但足够，而不是模板化堆砌

## 明确不允许

- spec PR 与实现代码混在一起提交
- 在 spec review 之前就把相关事项当作 Sprint 实施承诺
- 在 `spike` 或 `spec-ready` 阶段使用 `Fixes #...` 提前关闭实现事项
- 用研究结论替代正式架构或正式 spec
- 以“后面再补”为由跳过边界、验收、依赖和风险定义
- 要求审查 Agent 显式输出内部 Chain-of-Thought
- 默认把所有 spec review 都升级为多 Agent 流程

## 审查结论

### `APPROVE`

仅在以下条件同时满足时给出：

- 事项属于正确阶段
- 正式契约边界清晰
- 验收标准明确且可验证
- 依赖与风险已足够支撑进入实现

### `REQUEST_CHANGES`

出现以下任一情况时应直接给出：

- 范围与阶段不一致
- 缺失关键契约文件或关键章节
- `plan.md` 缺少最低结构，无法判断实施顺序、验证方法或进入实现前门槛
- 触发条件已经成立，但缺失必要的附加文档
- Spike 类事项缺少证据分层，或把 `fallback` 错当成 `admission_ready`
- 验收标准模糊，无法判断何时完成
- 状态型 runtime 事项缺少健康矩阵、恢复路径或最低回归矩阵
- 与愿景、架构或既有正式契约冲突
- 混入实现代码，破坏“先 spec review，再实现”的顺序

## 审查输出建议

每次 spec review 至少包含：

- 结论：`APPROVE` 或 `REQUEST_CHANGES`
- 是否允许进入实现：`ready_for_implementation = true|false`
- 范围判断：是否属于当前阶段
- Findings：阻止进入实现的问题
- 疑似 Scope Creep：建议延期、下放或删除的功能
- Edge Cases Needed：必须补齐的异常与边界场景
- 缺失项：缺少哪些契约、验证、依赖说明
- 进入实现前动作：必须完成的事项

## 审查组织方式

- 默认使用单 Agent 的结构化审查即可
- 只有在 `P0/P1`、跨模块、公共契约或高风险规约时，才建议引入多 Agent 分工复核
- 即使采用多 Agent，最终也必须回收成一份统一结论，不拆散成多份无法执行的评论

## 与代码审查的关系

- spec review 回答“这件事是否已经被正确定义，值得进入开发”
- code review 回答“这段实现是否值得进入主干”

两者不能互相替代。
