# Roadmap

> 本文档是 WebEnvoy 的阶段交付路线图。每个 Phase 代表一组有内聚目标的交付集合，Phase 内按关注点分类平铺 FR 列表，不设史诗（Epic）层。
>
> **状态与 GitHub Issue 保持同步，本地文档不追踪进度**。FR 的详细规约请前往对应目录 `docs/dev/specs/FR-XXXX-*/`。
>
> **阶段门约束**：每个 Phase 开工前，应先补齐对应 FR spec 与前置 Spike 结论；未满足前置条件时，不应并行推进后续高风险阶段。

---

## Phase 1：核心运行时基座与首个平台侦察

> 目标：完成最小可运行底座，验证四组件主链路（CLI / Extension Background / Content Script / SQLite）与“浏览器内执行”原则，并跑通首个高价值平台的关键侦察链路。
>
> 指标：CLI 到浏览器内执行链路稳定可复现；小红书关键 API、签名与读写约束完成 Spike 验证；形成可继续编码的正式 FR 规约。

**[基础运行时]**
> 建立 CLI 入口、Native Messaging 通道、Extension 主链路、SQLite 持久化与最小信息披露控制，确保系统具备真正可运行的骨架，而不是只停留在文档层。

**[最小账号基座]**
> 只实现运行所必需的 Named Profile、登录恢复、独占锁、基础代理绑定与浏览器生命周期管理；不在本阶段引入完整养号系统、复杂节律引擎或大规模账号矩阵调度。

**[首个平台侦察 · 小红书]**
> 完成 Spike A：确认核心 API、签名函数、关键追踪字段、富文本编辑器结构与上传路径，为后续适配器与写链路实现提供确定性输入。

---

## Phase 2：知识转化与快速 CLI 化建模

> 目标：将参考项目中的知识、架构与关键算法转化为 WebEnvoy 自有资产，优先建设“10 分钟 CLI 化”所依赖的侦察产物、规则抽象与人工微调接口，而不是把它留到后期再补。
>
> 指标：形成统一的侦察快照格式、规则抽象边界、算法移植清单与人工固化工作流草案；首个平台的侦察结果可以沉淀为可复用模板，而非一次性经验。

**[参考项目知识转化]**
> 系统梳理参考项目中与 API 发现、签名定位、语义树压缩、请求筛选、规则维护相关的方法论，提炼为 WebEnvoy 可持续复用的设计输入与检查清单。

**[侦察产物建模]**
> 定义“平台侦察”的标准化输出，包括网络样本、关键 JS 变量、AX Tree 片段、签名调用点、错误码样本与人工备注位，作为后续 CLI 化与规则固化的中间表示。

**[规则抽象与人工微调接口]**
> 提前定义 `rules.yaml` 的收敛边界、候选规则到正式规则的转化流程，以及哪些点由人工确认、哪些点可以由工具辅助生成。

---

## Phase 3：单平台生产化闭环

> 目标：在前两阶段的基座与建模资产之上，把一个头部高防平台做成可调用、可诊断、可恢复的生产化能力，优先验证 WebEnvoy 的 L3 路线是否真正成立。
>
> 指标：首个平台核心读链路稳定，关键写链路可用，L3 专用引擎在限定场景下稳定操作成功率接近 95%；失败时能输出可读诊断与明确人工介入点。

**[侦察工具链]**
> 实现 `recon` / `dispatch` 所需的最小侦察与调度原语，支持观察请求头、关键 JS 变量、风控探针状态，并为规则维护提供诊断证据。

**[平台适配 · 小红书]**
> 基于已完成的 Spike 结果实现首个 L3 适配器，优先覆盖搜索、详情、核心发布或互动中的一条最短写链路，而不是一次性追求全量能力。

**[错误处理与可恢复性]**
> 打通登录失效、验证码、风控暂停、断连恢复等最小闭环，让系统在失败时可定位、可暂停、可恢复，而不是静默报错。

---

## Phase 4：第二平台复用验证

> 目标：在不推翻 Phase 3 架构的前提下接入第二个高防平台，验证适配器协议、规则文件、账号模型和侦察工具链是否具备横向复用价值。
>
> 指标：抖音适配开发明显快于首个平台；同一套运行时、规则装载和诊断机制无需重构即可复用；形成“新增平台”的标准化开发流程。

**[平台适配 · 抖音]**
> 完成核心 API、签名与最短读写闭环，验证 WebEnvoy 不是“小红书专用实现”，而是真正的平台执行底座。

**[适配器协议固化]**
> 根据前两个平台的实际差异收敛 `rules.yaml`、适配器接口与错误码映射，避免在通用层建设前留下平台特化债务。

**[账号与限流策略补强]**
> 在两个平台的真实运行反馈基础上，补齐冷却、退避、基础配额控制与平台差异化策略，但仍不把完整“无人值守资产管家”一次性做满。

---

## Phase 5：L2 通用读取底座

> 目标：在既有运行时之上构建真正面向未知网站的 L2 通用读取层，使 WebEnvoy 对“无适配器网站”也具备可靠的读取与基础交互能力。
>
> **关键区分**：L2 通用层 ≠ 已知平台适配器集合。L3 解决的是“已知高价值平台的极速与高成功率”，L2 解决的是“任意未知网站的首次可用性”。
>
> 指标：L2 层对未知网站首次读取成功率 ≥ 80%；启用请求拦截优化后任意 URL 内容提取响应时间 ≤ 2s。

**[L2 通用读取层]**
> 基于 Playwright 构建 AI 友好的通用读取原语，核心是 AX Tree 压缩与 RefMap 元素引用系统。移植 agent-browser 的 `processAriaTree` + `compactTree` + RefMap 短引用算法；实现通用 CLI 原语 `navigate` / `snapshot` / `click` / `type` / `intercept` / `extract`。

**[请求拦截与提取优化]**
> 集成图片/广告/字体等资源拦截策略，并补齐正文 Markdown 提取、登录墙检测、验证码检测等通用读路径能力。

**[前置 Spike 收口]**
> Phase 5 开工前必须完成 AX Tree 压缩算法研究与请求拦截可行性验证，避免一边实现一边确认基础假设。

---

## Phase 6：L1 兜底与适配器固化流水线

> 目标：在 L2 稳定后补上最末级兜底能力，并把“探测 → 验证 → 人工微调 → 固化”为 L3 规则的闭环做成可重复流程，正式逼近“10 分钟 CLI 化”。
>
> 指标：L2 失效场景具备明确 L1 兜底路径；单点平台从侦察到规则固化的流转周期 ≤ 10 分钟。

**[L1 VLM 漫游兜底]**
> 截图 + 坐标 + 物理点击，覆盖 AX Tree 无法解析、强 Canvas 化或 DOM 语义严重缺失的边缘站点。

**[适配器固化流水线]**
> 将 L2 漫游路径的探索结果提炼为可复用的 L3 适配器规则，跑通“探测 → 验证 → 人工微调 → 固化”闭环，但不做运行时自动切换。

**[多账号资产管家增强]**
> 在已有账号基座上逐步加入 Profile Seeding、行为人格、长尾节律与更完整的代理调度，把“账号长期存活”建设为独立能力，而不是前置阻塞所有主路径。

---

## Phase 7：逆向生态工厂与横向扩张

> 目标：将 WebEnvoy 升级为可持续自我进化的“逆向工厂”，并通过结构化插件规范与沙盒机制扩展长尾平台生态。
>
> 指标：侦察工具泛化能力达到通用标准，支持基于 JSON 或轻量 TS 脚本在外部仓库动态挂载适配器（Adapter as a Plugin）。

**[逆向侦察增强]**
> 扩展通用流量审计、变量侦察、规则诊断与 CLI 化固化辅助能力，持续压缩新平台逆向周期。

**[生态与沙盒体系]**
> 建立插件装载、执行隔离、版本兼容与安全边界，避免第三方适配器污染核心运行时。

**[长尾平台支持]**
> 在核心平台和通用层都稳定之后，再扩大更多 L3 平台覆盖，把横向扩张建立在成熟底座之上。
