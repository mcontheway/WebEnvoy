# Roadmap

> 本文档是 WebEnvoy 的阶段交付路线图。Roadmap 的主线按用户逐步拿到的价值来组织，而不是按技术模块平铺。
>
> **状态与 GitHub Issue 保持同步，本地文档不追踪进度**。FR 的详细规约请前往对应目录 `docs/dev/specs/FR-XXXX-*/`。
>
> **阶段门约束**：每个 Phase 开工前，应先补齐对应 FR spec 与前置 Spike 结论；未满足前置条件时，不应并行推进后续高风险阶段。

---

## 阅读说明

### 1. Phase 主要讲用户价值，也允许纳入产品完整性阶段

本 Roadmap 的大部分 Phase 用来回答用户逐步拿到什么价值；但当某个能力对产品完整性不可缺少时，也会被纳入正式主线阶段。

也就是说，这里的 Phase 主要回答以下问题：

1. WebEnvoy 什么时候先成为一个真正好用的网页执行工具？
2. 什么时候用户能把一次成功经验留下来反复使用？
3. 什么时候这些能力能被别人导入、验证、升级和修复？
4. 什么时候产品完整性需要被正式补齐？
5. 什么时候能力开始在用户之间流动，形成网络？

### 2. L3 / L2 / L1 是执行策略，不是 Phase 名称

- **L3**：优先解决重点平台的高成功率与高效率
- **L2**：解决未知网站的首次可用性，并为能力封装提供基础
- **L1**：只在前两者无法满足时作为兜底，不作为前期主线

它们会跨多个 Phase 出现，但每个 Phase 的投入重点不同：


| Phase   | 执行能力重点 |
| ------- | --- |
| Phase 1 | 以 **L3 最小闭环** 为主，同时建立最小能力封装骨架 |
| Phase 2 | 以 **L2 首次可用 + 跨 L3/L2 的统一封装验证** 为主，不做完整 L1 |
| Phase 3 | 重点转向能力交付：导入、版本、兼容性、变化感知与快速修复 |
| Phase 4 | **L1 兜底能力** 成为主线正式阶段，但实施顺序可与 Phase 3 前后插入或并行推进 |
| Phase 5 | 执行策略不再大扩张，重点转向能力分享、导入、管理与流通 |


### 3. 当前集成策略是 CLI-first

- 当前阶段默认以 CLI 作为第一集成面
- 暂不把 SDK / API 作为主线交付
- CLI 必须按可集成契约设计：命令语义稳定、输入输出机器可读、错误结构化、运行标识清晰

### 4. 当前主线不包含独立上层运行系统

WebEnvoy 当前主线专注于：

- 网页执行
- 能力沉淀
- 能力交付
- 能力流动

不把身份管理、账号矩阵、任务调度、长期运营等独立上层系统能力写入当前 Roadmap 主线。

### 5. 核心能力面是读 / 写 / 下载

无论底层走 L3、L2 还是 L1，WebEnvoy 面向用户交付的核心网页能力都应能落在以下三类之一：

- **读**：读取页面、接口或内存中的结构化信息
- **写**：在页面中填写、点击、提交，完成真实交互
- **下载**：把网页中的文件、媒体或可落盘结果安全取回

它们在各阶段的关注重点不同：

- **Phase 1**：先把读做稳，并验证最小交互动作；下载不作为首阶段阻塞项，但不能在能力模型里缺位
- **Phase 2**：让读 / 写 / 下载都能被快速封装为可复用能力
- **Phase 3**：让读 / 写 / 下载能力都能被导入、验证、升级和修复
- **Phase 4 / 5**：让这三类能力在兜底与流通场景下仍保持同一套产品模型

## Phase 1：先变成好用工具

> 目标：只证明一件事成立: Agent 已经能通过 CLI 稳定调用 WebEnvoy，在一个高价值平台上完成一次真实网页读取，并具备最小交互、最小观察和最小错误返回；同时建立最小能力封装骨架，为后续统一封装 L3 / L2 / L1 打底。
>
> 指标：CLI 到浏览器内执行链路稳定可复现；首个平台至少一条核心读链路稳定可用；至少一种最小页面交互动作已可验证；结果与错误均能结构化返回；能力入口、参数输入、结果输出与运行标识具备最小统一骨架。

### 本阶段只做 5 个必需件

**[1. CLI 最小入口与可集成契约]**

> 建立能被稳定调用的 CLI 入口，让命令、参数、结果、错误和运行标识都具备最小契约，而不是继续停留在临时脚本层。

**[2. 浏览器内通信闭环]**

> 打通 CLI、Native Messaging、Extension 与浏览器内执行链路，确保 WebEnvoy 真能进入真实网页环境完成一次调用。

**[3. 最小身份 / 会话承载]**

> 只保留完成一次可靠网页执行所必需的能力，例如指定执行身份、恢复既有登录态、执行隔离、并发保护与状态回传。

**[4. 首个平台核心读闭环]**

> 以一个高价值平台为目标，优先打穿一条稳定可复现的核心读链路，并验证至少一种最小页面交互动作。重点是先证明“浏览器内执行 + L3 路线”成立，而不是一开始追求完整平台写链路。

**[5. 最小观察、错误分类与结构化诊断]**

> 为首个平台最小闭环提供必要的执行过程可见性，让用户至少能看见页面状态、关键请求、失败位置和基础错误类型，而不是面对黑盒 success / fail。

### 本阶段同时形成

**[最小能力封装骨架]**

> 先建立一套最小但统一的能力壳，让 Phase 1 跑通的首个 L3 样本不是孤例命令，而是已经具备能力入口、参数输入、结果输出、错误表达与运行标识的雏形。

### 本阶段非阻塞项

- 最小运行记录只要求达到“可追踪、可排错、可复现”的最低可用程度；不要求在 Phase 1 一次性做完整 SQLite 基座建设
- 最小写链路可以作为 Phase 1 后段验证项，但不阻塞“首个读闭环成立”
- 下载能力在能力模型里不能缺位，但不作为首阶段阻塞项

### 明确不做

- 不做完整 L2 通用层
- 不做 L1 视觉 / 物理兜底
- 不做 SDK / API
- 不做能力分享、导入和版本化
- 不做独立上层运行系统能力

### 阶段门

- Agent 已能通过 CLI 稳定调用一次真实网页执行
- 首个平台的最短路径不是一次性脚本，而是可重复运行的命令闭环
- 最小能力封装骨架已经形成，不会把首个 L3 样本做成未来无法复用的特例
- 出错时至少能判断是页面变化、请求失败还是执行中断

## Phase 2：再变成可复用能力

> 目标：让用户不只“跑通一次”，而是能把成功经验快速留下来，自己反复使用；同时让未知网站具备首次可用性。
>
> 指标：用户可以把一次成功路径快速封装成可复用能力；未知网站具备最小读取与基础交互能力；同一套封装机制已能同时承载首个 L3 样本与新的 L2 样本；读 / 写 / 下载三类能力已具备统一封装方向。

### 本阶段建设

**[L2 首次可用能力]**

> 建立面向未知网站的最小读取与基础交互能力，让用户在无现成能力时仍能先做成一次，而不是被完全卡死在未知页面上。

**[快速封装能力]**

> 把“一次成功路径”变成可以留下来的能力描述，而不是只存在于临时脚本或操作者记忆里。该封装机制不只服务 L2，也必须能承载 Phase 1 已经形成的 L3 样本，目标是尽量逼近“10 分钟封装一次成功经验”的工作流。

**[最小验证与重放]**

> 用户把能力留下来后，应该能再次运行、再次验证，知道它是否仍然可用，而不是每次都重新手工试。

**[首个平台的复用收敛]**

> 将首个平台第一阶段打穿的最短路径整理成可复用形态，为后续交付、版本化和分享奠定基础。

**[跨 L3 / L2 的统一封装验证]**

> 明确同一套能力封装方式可以同时承载专用路径与通用路径，避免 CLI 化被误写成某一层专属能力。

**[读 / 写 / 下载统一表达]**

> 明确三类核心能力在封装时如何统一表达，避免把下载能力留成后面临时拼接的特殊分支。

### 明确不做

- 不做完整能力分发网络
- 不做复杂版本兼容体系
- 不做完整 L1 兜底
- 不做上层系统集成接口扩张

### 阶段门

- 用户能把一次成功经验快速留下，并自己重复调用
- 未知网站已经具备“先做成一次”的最小能力
- 同一套封装机制已经能同时承载至少一个 L3 能力与一个 L2 能力
- 封装出来的能力不再依赖临时人工记忆才能复用

## Phase 3：再变成可交付能力

> 目标：让这些能力不只“自己能用”，而是“别人导入后也能用”；同时让网站变化、接口变动、能力升级成为可感知、可判断、可修复的问题。
>
> 指标：能力可被导入、验证、升级与禁用；兼容性变化可被发现；失效后能输出明确诊断并支持快速修复；至少两个平台验证同一套交付机制不是特例；读 / 写 / 下载能力都能落入同一套交付模型。

### 本阶段建设

**[能力导入与安装]**

> 让一个能力从“本地留存”升级为“可交付给别人使用”。重点是导入后能跑、能验证、能启用 / 禁用，而不是停留在文件拷贝层面。

**[版本与兼容性]**

> 为能力建立最小版本化与兼容性表达，让用户知道自己装的是什么版本、升级后会不会受影响、当前网站变化是否已经超出能力适用范围。

**[变化感知与快速修复]**

> 当网站改版、接口变动、页面结构变化时，系统要能帮助用户尽快定位哪里坏了，并支持快速修复，而不是把用户重新扔回从零摸索。

**[第二平台验证]**

> 用第二个平台验证“可封装、可导入、可修复”的机制不是首个平台特例，从而证明 WebEnvoy 不是单点定制，而是真正的能力底座。

### 明确不做

- 不做大而全的官方能力库
- 不把官方示例扩张成官方包办全部站点
- 不做独立上层运行系统能力

### 阶段门

- 别人导入一个能力后，能知道怎么验证、怎么启用、怎么升级
- 网站变化后，用户能更快判断是能力失效还是使用方式有误
- 第二个平台验证完成，证明交付机制具备横向复用价值

---

## Phase 4：补上 L1 兜底能力

> 目标：把 L1 作为产品完整性的一部分正式补齐，使 WebEnvoy 在 L3 / L2 无法满足时仍然存在明确兜底路径。
>
> 指标：已经定义哪些场景必须进入 L1；至少一类真实边缘场景具备可工作的 L1 路径；L1 能被同一套能力封装机制承载，而不是成为体系外特殊分支。
>
> **顺序说明**：本阶段是主线正式阶段，但不要求死板地排在 Phase 3 之后。只要前置条件满足，可在 Phase 3 前后插入，或与 Phase 3 并行推进。

### 本阶段建设

**[L1 兜底路径]**

> 建立面向极端场景的最后兜底能力，例如 DOM 语义严重缺失、强 Canvas 化或前两层执行路径明确失效的页面。

**[L1 场景边界]**

> 明确什么情况下应该进入 L1，什么情况下不应轻易进入，避免把高成本兜底路径滥用成默认路径。

**[L1 与统一封装对齐]**

> 证明 L1 虽然更重，但仍能被同一套能力封装、调用、验证和交付机制承载，而不是在产品里另起一套平行体系。

### 明确不做

- 不把 L1 升级成默认执行路径
- 不为追求“看起来完整”而引入没有真实场景支撑的重型能力

### 阶段门

- 至少一类真实阻塞场景已经被 L1 成功兜住
- 用户能理解什么时候该用 L1，什么时候不该用
- L1 没有破坏前面已建立的统一能力模型

---

## Phase 5：最后变成能力网络

> 目标：让能力开始在用户之间流动，形成分享、导入、管理与演进的使用网络。
>
> 指标：私有能力与共享能力可并存；用户可以发现、导入、升级和禁用能力；能力来源、适用范围与信任边界可表达；官方以示例和打样为主，而不是包办全部供给。

### 本阶段建设

**[分享与导入网络]**

> 让用户不仅能自己封装能力，还能分享给别人，或导入别人已经验证过的能力，从而减少重复劳动。

**[来源与信任表达]**

> 当能力开始流动时，用户需要知道它来自哪里、适用于什么、是否值得安装，而不是面对一堆来源不明的脚本。

**[安装、升级与管理]**

> 能力一旦进入流通，就必须有最基本的安装、升级、禁用、替换和清理机制，避免能力越积越多却不可控。

**[官方角色收敛]**

> 官方重点提供示例、模板、最佳实践与少量高价值参考能力，而不是承担“所有能力都由官方产出和维护”的职责。

### 明确不做

- 不把能力网络扩展为独立上层运行系统
- 不把分享网络等同于中心化官方商店

### 阶段门

- 能力已经不只存在于单个操作者本地，而开始在用户之间真实流动
- 用户能区分私有能力、共享能力与官方示例能力
- 能力网络不会反过来破坏执行底座的清晰边界
