# 架构原则 (Architecture Principles)
> WebEnvoy 核心架构决策与约束
> 本文档记录 WebEnvoy 的技术底座、执行哲学及核心架构约束。这些决策比愿景更具体，是研发过程中的"交通规则"。

---

## 1. 执行哲学：专用斧头优于全能小刀

我们拒绝陷入"追求 100% 通用化"的理论陷阱。

*   **核心逻辑**：如果有一把极其趁手、专门劈柴的斧头（专用 L3 逆向库），AI 就不应该被迫拿着一把多功能小刀（L1 漫游）去低效砍树。
*   **研发策略**：优先针对高价值平台（如社媒、电商后台）打造"特供插件"，以极速和稳健性（95%+ 成功率）作为核心竞争力，再辅以通用漫游方案作为兜底。

## 2. 交互原语：两层命令体系

WebEnvoy 存在两个层级的命令接口，分别服务于不同的执行路径，通过 CLI 命令前缀路由：

### L3 平台级原语（Phase 1，高成功率，特定平台）

针对已建立适配器的平台，所有平台适配器（PlatformAdapter）必须遵循统一的协议契约：

| 命令 | 职责 | 核心要求 |
|---|---|---|
| `search` | 检索信息 | 返回精简后的结构化列表，严控 DOM 噪音 |
| `detail` | 深度读取 | 获取完整数据，包含隐藏字段或内存状态 |
| `publish` | 写入内容 | 处理富文本同步、多媒体上传等复杂交互 |
| `interact` | 社交互动 | 完成点赞、评论、关注、私信等行为指标 |
| `recon` | **链路侦察** | 返回当前网络请求头、JS 变量或风控探针状态 |
| `dispatch` | **介入调度** | 修改后续执行参数，干预底层的指纹或请求策略 |

CLI 调用格式：`webenvoy xhs.search --query "露营"`（带平台前缀）

### L2 通用级原语（Phase 2，任意网站）

基于 AX Tree + Playwright 构建，适用于任意未知网站的通用读写：

| 命令 | 职责 |
|---|---|
| `navigate` | 导航到任意 URL |
| `snapshot` | 获取 AX Tree 压缩快照（RefMap 格式） |
| `click` | 点击元素（引用 `@e1` 格式） |
| `type` | 向元素输入文本 |
| `intercept` | 设置请求拦截规则 |
| `extract` | 提取正文内容（Markdown 格式） |

CLI 调用格式：`webenvoy navigate --url "https://..."` （无平台前缀）

> 两层接口均通过同一个 CLI 进程入口，由命令格式决定路由方向。L3 优先级高于 L2，L2 高于 L1 VLM 兜底。

## 3. 技术路线红线：拒绝异构缝合，坚持版权自主

我们拒绝直接"拼装"不同技术栈的庞然大物作为核心基石。

*   **拒绝缝合**：坚决不通过 Submodule 或命令行直接挂靠大规模 Python 爬虫，防止出现"维护成本灾难"和技术栈割裂。
*   **知识转化法**：竞品是"武功秘籍"而非"代雇佣兵"。我们可以学习其逆向思路、截取其客观接口定义，但核心代码逻辑必须使用自主技术栈（如 TypeScript/Node/Go）重新实现。
*   **合规基座**：确保代码资产 100% 自主，消除未来商业发行中的法务红线。

## 4. 性能与成本：渐进式披露 (Progressive Disclosure)

我们追求极简高效，反对滥用 LLM 上下文。

*   **入口选择**：坚持 **CLI（命令行界面）** 作为首选入口，而非建立重型长连接常驻服务（如 MCP）。
*   **节流哲学**：交互过程中仅向 AI 披露当前步骤最必要的上下文。拒绝一次性推送全量 DOM 或复杂的协议声明。
*   **Token 预算**：将 Token 消耗视为核心成本指标，优先通过"后端预处理/过滤"来减少大模型的感知负担。

## 5. QoS (Quality of Service) 成功率量化指标

我们对不同层级的执行路径设定了明确的质量底线，任何合并进主分支的代码应以此为基准测试：

*   **L3 专用引擎（API/内存态直读）**：稳定操作成功率必须达到 **95% 以上**。
*   **L2 通用读取（AX Tree + Playwright 原语）**：针对任意未知网站，成功率应达到 **80% 以上**。
*   **L1 VLM 漫游兜底**：作为最后手段，在 L2 失效时成功率预期为 **60% 左右**。

## 6. 核心目标：自我进化与快速 CLI 化 (Self-Evolution)

仿效 `bb-browser` 的实现路径，WebEnvoy 不仅是一个执行器，更是一套**"逆向工厂"**。

*   **侦察工具链 (Recon Tools)**：
    *   **语义树降噪**：内置对 A11y Tree 的过滤算法，仅保留关键交互节点，为 LLM 提供极致干净的操作上下文。
    *   **流量实时审计**：通过底层协议（CDP 等）实时监听并结构化网络流量，快速识别关键 API 及其载体。
*   **适配器固化工作流**：
    *   系统应具备将"L1/L2 漫游中的成功路径"**半自动提炼**为"L3 固定脚本"的能力（需人工验证后确认固化，不做全自动替换）。
    *   **逆向周期指标**：我们的工具栈应支持开发者/AI 在 10 分钟内完成对一个新平台核心接口的"探测 -> 验证 -> 固化"。
*   **策略感知与诊断建议**：执行器在任务失败或成功率下降时，应输出可读的诊断信息并**建议**升级/降级路径（如"建议从 L2 固化为 L3，当前 L2 成功率 73%，低于阈值"）。**策略切换本身由人工在 `rules.yaml` 中确认**，不做运行时自动切换。

---

## 7. 开发者警示：防止范围蔓延 (Scope Creep)

*   **不做 Agent 大脑**：不在这里写长篇的逻辑推理和任务拆解，只写执行。
*   **不做人类浏览器**：不考虑书签、历史记录等"人类舒适性"功能。
*   **不做零配置承诺**：我们承认并积极维护针对特定站点的微调脚本，不追求虚幻的"一招鲜吃遍天"。
