# #208 最小页面交互正式验证说明

> 关联 Issue：`#208`
>
> 当前用途：作为后续 `#208` 实现 PR 的唯一正式验证依据。
>
> 关闭语义：本说明文档本身不关闭 `#208`；仅后续真正完成 live 验证闭环的实现 PR 才可 `Fixes #208`。

## 背景

Phase 1 的正式指标不是“已经具备完整写能力”，而是“至少一种最小页面交互动作已可验证”。

`FR-0008` 已冻结 `#208` 可消费的候选动作输入，`FR-0011` 已冻结 `#208` 的 gate-only 前置与最小可观测性边界；但这两者都不等于 `#208` 已完成真实页面交互正式验证。

当前需要单独冻结的是：`#208` 到底验证哪一个动作、验证到什么程度、以什么信号判定成功或失败、需要什么证据、什么情况下才允许关闭 issue。

## 目标

- 冻结 `#208` 的唯一正式验证动作。
- 冻结该动作的验证范围、成功信号、失败信号与最小复现实验。
- 冻结后续实现 PR 必须提交的最小证据与关闭语义。

## 非目标

- 不开放完整写能力。
- 不冻结正式 `xhs.editor_input` / `xhs.interact` 命令 contract。
- 不推进图片上传、提交、发布确认、多步骤发布链路。
- 不扩张到 Phase 2 能力封装、L2 通用层或上传能力。
- 不把 gate-only contract tests 当作真实页面交互验证完成。

## 唯一正式验证动作

- `action_id`：`editor_input`
- 页面：`creator.xiaohongshu.com/publish`
- 动作定义：在发布页富文本编辑器上执行“聚焦并输入少量文本”的一次真实、可逆页面交互。
- 当前动作层级：`reversible_interaction`

补充约束：

- `editor_input` 在这里表示 `#208` 的正式验证对象，不等于正式稳定命令名。
- 后续实现 PR 可以围绕 `issue_208` 范围内的验证路径交付真实交互证据，但不得在该 PR 中顺手宣称正式 command contract 已冻结。
- 若后续需要对外冻结稳定机器接口，仍需单独起正式 command contract 规约。

## 验证范围

本事项只验证以下范围：

- 页面已进入目标发布页。
- 富文本编辑器可被定位并获得焦点。
- 输入少量文本后，页面上出现稳定可见的文本结果。
- 输入结果在最小失焦/重新观测后仍保留。

本事项明确不验证以下范围：

- 图片上传。
- 提交、发布、确认弹窗或任何不可逆动作。
- 草稿保存、发布 API、上传 API 或任意 fallback 写链路。
- 能力封装、复用、分享、版本化。

## 运行前置

后续实现 PR 只有在以下前置同时满足时，才能把本事项视为真实验证，而不是 gate-only：

- 已满足 `FR-0010` / `FR-0011` 冻结的门禁、审批、审计与风险状态前置。
- 目标域与目标页明确为 `creator.xiaohongshu.com/publish`。
- 使用已登录且可复核的目标 profile / session。
- 执行路径明确标识为 `issue_208` 范围内的正式验证，而不是通用写命令开放。

## 成功信号

后续实现 PR 至少需要同时满足以下成功信号：

1. 编辑器成功获得焦点，且输入动作实际发生在目标编辑区域。
2. 少量文本在页面上可见，未被即时清空或回滚。
3. 在一次最小失焦或重新观测后，文本仍然保留。
4. 本次验证未触发图片上传、提交、发布确认或其他不可逆副作用。
5. 结构化结果能够明确标识“本次完成的是 `editor_input` 真实验证”，而不是 gate-only success。

## 失败信号

出现以下任一情况，都不得判定 `#208` 已完成：

1. 请求仍停留在 gate-only，未发生真实页面交互。
2. 门禁或审批未满足，导致执行被阻断。
3. 编辑器未获得焦点，或输入落在错误节点。
4. 输入后文本未出现、立即丢失、失焦后回空，或页面框架状态未同步。
5. 页面出现风控提示、账号异常、验证码或其他需中止的风险信号。
6. DOM 结构漂移导致动作依赖旧路径，无法证明当前页面仍满足正式验证口径。
7. 验证过程中误触发上传、提交、发布确认或其他超范围动作。

## 最小复现实验

后续实现 PR 的最小 live 实验固定为：

1. 打开已登录的 `creator.xiaohongshu.com/publish` 页面。
2. 确认编辑器处于可输入前状态，且页面不存在待提交副作用。
3. 对目标编辑器执行一次聚焦。
4. 输入少量测试文本。
5. 读取页面可见结果，并执行一次最小失焦或重新观测。
6. 确认文本仍保留，且未触发上传、提交、发布确认。

最小 replay 要求：

- 在同一正式验证 PR 中，至少提供一次按上述步骤重新执行的复现实验。
- replay 仍只允许围绕 `editor_input` 单动作，不得借机扩成上传或提交链路。

## 证据要求

后续真正用于关闭 `#208` 的实现 PR，至少需要提交以下证据：

1. 指向本说明文档的引用，证明本 PR 以此作为唯一正式验证口径。
2. 一组真实执行证据，能证明已从 gate-only 进入真实页面交互。
3. 一组最小 replay 证据，能证明该动作不是单次偶然成功。
4. 对成功信号与失败信号的逐项对照说明。
5. 对“未触发上传/提交/发布确认/完整写链路”的边界说明。
6. 本地验证记录与必要测试结果；若环境依赖导致部分自动化验证无法执行，必须在 PR 中明确写出未完成项与原因。

可接受的证据载体包括但不限于：

- 结构化运行结果与 run id
- 关键页面观测结果
- 最小截图或日志摘录
- 对 replay 的复现记录

## 关闭语义

- 本说明文档合入时，PR 只能使用 `Refs #208`。
- 只有后续实现 PR 同时满足“真实交互已发生 + 最小 replay 已完成 + 证据齐备 + 未越出范围”时，才可使用 `Fixes #208`。
- 若后续 PR 只补 gate-only 观测、contract test、候选动作整理、上传/提交流程研究或正式命令 contract 冻结，均不得关闭 `#208`。
