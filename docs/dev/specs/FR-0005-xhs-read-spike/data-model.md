# FR-0005 数据模型说明

## 结论

FR-0005 不引入新的持久化实体，也不修改现有共享数据模型。

本 FR 的职责是冻结“小红书核心读链路 Spike A”的结构化侦察结论，而不是定义新的 SQLite 表、运行时会话对象或跨模块共享 schema。

## 本 FR 实际产出的结构化对象

虽然没有新增持久化/共享实体，但本 FR 会为后续实现冻结三类结构化结论：

1. `endpoint_catalog`
2. `signature_path`
3. `field_lifecycle_matrix`

这些对象的边界定义见 [contracts/xhs-read-spike.md](./contracts/xhs-read-spike.md)。

## 为什么这里不创建持久化实体

1. Spike 阶段的目标是确认端点、签名和字段生命周期，而不是落库运行数据。
2. 后续实现如果需要把侦察结果或运行记录持久化，应由对应实现 FR 显式决定是否复用现有基座，或在需要时修改数据模型。
3. 当前 Phase 1 的最小持久化基座由 `#144` 负责，FR-0005 不应提前抢占其数据真相源职责。

## 与后续实现的关系

后续小红书 L3 读适配实现只消费本 FR 冻结的结构化结论，不直接继承本 FR 的“数据模型”文件作为运行时 schema。

如果后续实现需要：

- 落库存储侦察样本
- 持久化端点版本快照
- 存储字段生命周期基线

则必须在对应实现 FR 中重新评估并显式定义数据模型变更，而不是在 FR-0005 中提前虚构实体。

## 非目标

- 不定义新的 SQLite 表
- 不定义新的跨进程 payload schema
- 不定义新的账号、会话、任务或能力持久化实体
