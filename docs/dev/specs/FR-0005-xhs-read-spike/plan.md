# FR-0005 实施计划

## 实施目标

在不写平台实现代码的前提下，交付一套可评审、可复用的 Spike A 正式规约与证据结构，明确小红书核心读链路的端点、签名、字段生命周期，并把结论约束成后续 L3 读适配可直接消费的输入。

## 分阶段拆分

### 阶段 1：侦察设计与样本矩阵冻结

- 产出：场景矩阵（搜索/详情/主页 + 异常边界）
- 依赖：FR-0002/0003 链路已可执行最小侦察

### 阶段 2：证据采样与结论收敛

- 产出：端点清单、签名调用链、关键字段生命周期矩阵
- 依赖：阶段 1 样本矩阵；同一场景多次复现结果

### 阶段 3：契约化输出

- 产出：`contracts/` 下的机器边界文档 + `research.md` 结论 + `risks.md`
- 依赖：阶段 2 已收敛结论

### 阶段 4：评审收口

- 产出：可进入 spec review 的 `spec.md`、`plan.md`、`TODO.md`
- 依赖：阶段 3 完成，阻断项清零或转明确待决

## 实现约束

1. 本 FR 仅做规约和侦察结论，不实现平台业务命令。
2. 不改动 FR-0001/0002/0003 既有契约，不引入新 CLI 外壳。
3. 不在本 FR 内新增或修改持久化 schema。
4. 不将一次性实验记录当作正式结论；结论必须可复现。
5. 风控、封禁、验证码等高风险场景必须记录为风险，不得忽略。

## 测试与验证策略

1. 文档验证：
   - `bash scripts/docs-guard.sh`
   - `bash scripts/spec-guard.sh`
2. 结论验证：
   - 每类端点至少 2 轮独立复现记录
   - 签名链路至少 1 轮“成功 + 失败分支”证据
3. 评审验证：
   - 使用 `spec_review.md` 检查价值对齐、闭环、异常边界、可执行性

## TDD 范围

1. 本 FR 本身为规约与侦察，不直接引入实现代码，故不要求新增单元测试代码。
2. 对后续实现的 TDD 先验要求在本 FR 中预先冻结：
   - 端点请求构造器与签名参数序列化：必须 TDD
   - 响应解析与错误分类映射：必须 TDD
   - 字段生命周期判定与规则化校验：必须 TDD
3. 人工侦察与抓样过程不适用 TDD，但必须有可复现实验步骤。
