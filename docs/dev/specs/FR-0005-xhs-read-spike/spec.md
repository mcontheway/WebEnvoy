# FR-0005 小红书 Spike A：核心读链路与签名确认

## 背景

Phase 1 的第 4 个必需件是“首个平台核心读闭环”。`#145` 的职责不是直接交付完整平台适配器，而是先把小红书读链路里最关键且最不确定的三类输入固定下来：

- API primary 读路径与请求形态
- page-state fallback 读路径与最小可冻结状态
- 页面内签名函数调用路径
- 关键追踪字段与生命周期

没有这层 Spike 证据，后续 L3 读能力会退化成猜测式实现，无法进入稳定迭代。

## 目标

1. 确认小红书核心读链路最小读路径集（搜索、详情、用户主页），Spike 输出允许 `API primary + page-state fallback` 并存；若 API 成功样本受风控/环境限制阻断，可冻结 fallback 证据，但 fallback 不等于实现准入。
2. 确认页面签名函数的最小调用路径，证明可在浏览器内执行链路中稳定触发。
3. 形成可复用的“侦察证据包”与结构化结论，作为后续 L3 读适配 FR 的输入基线；该输入基线不等同于“实现已就绪”。
4. 与 FR-0001/0002/0003 的运行时契约对齐，不引入第二套命令或消息壳。

## 非目标

- 不实现完整小红书读适配器。
- 不实现发布、互动、下载等写入能力。
- 不扩张到多平台并行侦察框架。
- 不在本 FR 内引入新的 SQLite schema 变更。
- 不把 page-state fallback 证据升级为默认实现路径或实现准入结论。

## 功能需求

### 1. 核心端点侦察范围冻结（API primary + page-state fallback）

- 必须覆盖三类读取场景：搜索列表、笔记详情、用户主页。
- 每类场景至少沉淀一个可复现读路径与关键字段，并显式标注 `primary` 或 `fallback`。
- 对 API 路径，必须记录请求方法、路径、最小必需 headers、关键 query/body 字段。
- 对 page-state fallback 路径，必须记录导航 URL 形态、关键 URL 参数、可直接提取的页面状态（如 `__INITIAL_STATE__` / 关键 store）。
- 如果某场景最终只拿到了页面级备用读路径，必须显式标注对应 API 路径仍为候选或失败样本。
- page-state fallback 的最小冻结内容只允许包含：
  - 路径模板与方法（例如 `GET /explore/<noteId>?xsec_token=...`、`GET /user/profile/<userId>?xsec_token=...`）
  - 关键 URL 参数（`noteId/userId`、`xsec_token`、`xsec_source`）
  - 最小页面状态探针（`window.__INITIAL_STATE__ === object`，以及已在 `research.md` 观测到的最小键路径）
  - 成功/失败信号与复现动作
- page-state fallback 的最小冻结内容不得扩张为以下实现承诺：稳定 DOM 选择器、完整字段覆盖率、长期兼容策略、默认运行时路由。

### 2. 签名链路最小可证实

- 必须确认页面签名函数的调用入口、最小输入、输出结构。
- 必须给出“可复现调用”证据，不接受仅靠静态猜测。
- 如果存在版本漂移风险，必须标注检测点和失效症状。

### 3. 关键追踪字段生命周期

- 对请求中影响风控或路由的关键字段，必须标注来源（页面状态/运行时生成/固定值）。
- 必须标注字段生命周期：会话级稳定、请求级变化、页面刷新后变化。
- 必须明确哪些字段属于后续适配器强依赖，哪些仅为可选增强。

### 4. 结构化侦察输出

- Spike 输出必须是结构化结论，而不是散乱笔记。
- 输出应能直接被后续 FR 引用，至少包含：
  - 端点清单
  - 签名链路结论
  - 关键字段矩阵
  - 失效信号与验证步骤
- 输出必须能区分“可作为侦察证据的 fallback 结论”与“可放行实现的 primary 结论”。

### 5. 与现有运行时契约对齐

- 侦察执行必须依赖 FR-0002/0003 已定义的浏览器内链路与会话边界。
- 不允许定义与 FR-0001 CLI 契约冲突的新外层输入输出模型。

### 6. 证据来源一致性

- 正式结论必须与 `research.md` 的“浏览器内第一手复核证据 + 仓库内历史基线”一致。
- 不引入仓库外不可复核引用作为冻结结论依据。

## GWT 验收场景

### 场景 1：核心端点可以被稳定识别

Given Native Messaging 与最小会话链路可用
When 对小红书执行搜索、详情、主页三类读取侦察
Then 每类场景都能给出至少一个 `primary|fallback` 已标注的证据端点
And API primary 路径记录方法、路径与关键请求字段
And page-state fallback 路径仅记录最小冻结内容（URL 模板、关键参数、最小状态探针、成功/失败信号）
And fallback 证据不会被标记为“实现已放行”

### 场景 2：签名调用路径可复现

Given 页面处于登录态并可访问对应功能页  
When 在浏览器内上下文执行签名链路验证  
Then 能得到可复现的签名调用入口和输出  
And 失败时能区分“调用入口失效”和“参数不匹配”  

### 场景 3：关键字段生命周期被明确标注

Given 已捕获核心读链路请求样本  
When 对关键追踪字段做多次会话/请求对比  
Then 每个关键字段都有来源与生命周期归类  
And 能明确后续适配器的强依赖字段集合  

### 场景 4：输出可直接服务后续 FR

Given Spike 侦察已经完成  
When 评审者检查输出文档  
Then 能直接提取下一轮实现 FR planning 所需的端点和签名结论
And 不需要二次逆向就能识别“哪些是 fallback-only 证据，且尚不构成实现准入”

## 异常与边界场景

### 1. 网络与并发

- 侦察期间出现短时超时或 429 时，必须记录失败证据，不得用成功样本覆盖失败事实。
- 同一端点多次请求返回不一致时，必须标注触发条件和复现实验。

### 2. 空与极值

- 搜索空结果、无效 note_id、无作品用户主页都要有样本与结论。
- 超长关键词或特殊字符导致签名输入变化时，必须给出边界说明。

### 3. 权限与状态异常

- 未登录、会话过期、风控拦截（验证码/滑块）必须单独记录，不得混写为普通请求失败。
- 账号权限不足导致端点行为变化时，必须标记为环境前置限制。

### 4. 外部依赖故障

- 页面脚本更新导致签名入口变化时，必须提供最小排查路径。
- Extension 注入失败或 content-script 不可达时，必须明确这属于运行时链路问题，而非平台接口问题。

## 验收标准

1. 三类核心读场景均有“结构化读路径证据 + 关键字段”结论；只有满足 `reproduced_multi_round`（且满足实现前置）的路径，才可被视为“已验证读路径”或进入实现准入输入。
2. 签名函数调用路径可复现，并有失效信号说明。
3. 关键追踪字段的来源与生命周期矩阵完整。
4. page-state fallback 仅冻结最小证据内容，不扩张为实现承诺；若某场景仅有 fallback 证据，默认仍不满足实现准入。
5. 侦察结论可直接作为后续 L3 读适配 FR 的输入基线，但不自动代表实现准入条件已满足。
6. 文档中没有引入与 FR-0001/0002/0003 冲突的契约，且未引入仓库外不可复核引用。
