# FR-0004 风险

## 关键风险

| 风险 | 触发条件 | 影响 | 缓解手段 | 观察信号 | 回滚 / 降级方式 |
|---|---|---|---|---|---|
| 诊断字段泄露敏感信息 | 直接把 Cookie、Token、完整请求体或响应体带回给调用方 | 安全和合规风险，可能放大信息披露面 | 只允许结构化元数据和短证据；对敏感字段做默认裁剪和脱敏 | 输出中出现敏感原文或高熵长串 | 立即回退到最小错误壳，只保留外层错误码与短诊断 |
| URL 携带敏感参数 | 页面状态或关键请求 URL 直接回传 query、fragment、签名或回调参数 | 一次性凭证、OAuth code 或签名链接可能被固化到对外诊断 | URL 默认只保留 `scheme://host/path`，敏感 query 参数必须按白名单或替换规则处理 | 对外输出中出现 `?token=`、`?code=`、`#...` 或签名串 | 立即净化 URL 并回退到不含 query/fragment 的最小地址 |
| 诊断载荷过大 | 页面状态、关键请求或证据过多 | Native Messaging 传输膨胀、分片、超时或失败 | 设置字段上限、证据上限和截断标记；保留主失败位置 | 响应长度逼近 1MB 或出现分片 | 丢弃次要证据，只保留 `failure_site` 和单条主诊断 |
| 误分类导致错误排障 | 页面变化、请求失败、执行中断被归到同一类 | 调用方无法正确判断是否重试或修复规则 | 冻结最小分类集合；允许 `unknown`，但不能乱映射 | 同类失败在不同场景下表现不一致 | 保持外层错误码不变，优先返回保守分类 |
| Scope Creep 演变成完整观测平台 | 开始引入回放、全量 trace、长期存储或暂停 / 恢复 | 破坏 Phase 1 收敛点，拖慢后续 FR | 在 spec 中明确非目标；仅保留最小观察面 | 文档或实现开始出现 replay / checkpoint / analytics | 从 FR-0004 剥离新增能力，回到最小诊断契约 |
| 观测失败反过来干扰主流程 | 采集失败抛异常，覆盖原始业务失败 | 调用方看到的是二次错误，不是原始问题 | 观测层必须是 best effort；失败时降级为最小诊断 | 原本应返回业务错误，却变成观测异常 | 记录最小错误壳后直接收口，不让观测失败吞掉主错误 |

## Stop-Ship 条件

出现以下任一情况时，不应进入实现或合并：

- 响应中出现敏感明文数据
- 诊断字段无法被机器稳定解析
- 诊断载荷没有上限或没有截断策略
- 本 FR 被改写成完整暂停 / 恢复或回放系统
- FR-0001 的外层错误壳被改写

## 剩余风险

- `page_kind` 与 `failure_site` 的粒度后续可能需要扩展，但扩展必须保持向后兼容
- 不同平台的页面状态可观测性不完全一致，部分场景只能返回降级诊断
