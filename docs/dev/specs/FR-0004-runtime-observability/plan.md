# FR-0004 实施计划

## 实施目标

这次实现只交付 Phase 1 所需的最小观察、错误分类与结构化诊断能力。

实现完成后，应至少形成以下结果：

- 成功响应可携带最小页面状态与关键请求摘要
- 失败响应可携带最小结构化诊断
- 页面变化、请求失败、执行中断、运行时不可用具备可区分的诊断分类
- 诊断字段复用 FR-0001 外层错误壳，不引入第二套失败返回协议
- 诊断输出受 URL 净化、敏感信息脱敏与消息体积约束

本 FR 不交付完整暂停 / 恢复系统、回放系统、长期观测平台或独立分析服务。

## 分阶段拆分

### 阶段 A：冻结诊断契约与分类

产出：

- 最小观测字段定义
- 诊断分类定义
- 失败位置与责任组件的枚举边界
- 与 FR-0001 外层壳的兼容约束
- 与 FR-0002 既有 transport 承载的兼容约束
- URL 净化、脱敏与截断上限

依赖：

- FR-0001 已冻结
- `#355` 的通信链路边界已明确

### 阶段 B：打通观测数据的最小采集面

产出：

- 页面状态采集口径
- 关键请求摘要采集口径
- 失败位置归因口径
- 敏感信息裁剪与长度截断规则
- 总预算超限时的裁剪顺序与截断标记

依赖：

- 阶段 A 完成
- Phase 1 首个平台读闭环有可观测事件源

### 阶段 C：接入统一错误壳与结果壳

产出：

- 成功响应携带观察面
- 错误响应携带诊断对象
- 运行时不可用与执行失败的区分输出
- 与 FR-0001 的退出码、状态码语义对齐
- 不改写 FR-0002 已冻结的 link-layer envelope

依赖：

- 阶段 B 完成

### 阶段 D：测试与收口

产出：

- 分类映射测试
- 观察字段结构测试
- 截断 / 脱敏测试
- 成功与失败契约测试

依赖：

- 阶段 C 完成

最小闭环：

- 到阶段 C 结束时，应能在首个平台流程里同时看到“结果 + 观察面”与“失败 + 结构化诊断”

## 实现约束

- 必须继续沿用 FR-0001 的外层 `status / error.code / error.retryable / exit code` 约束。
- 不能把完整 pause / resume、重放或长期观测平台塞进本 FR。
- 不能返回 Cookie、Authorization、Token、完整响应体或其他高敏感原文。
- 不能让诊断输出无限增长，必须可截断、可降级、可丢弃次要证据。
- 不能把页面观测做成一套和 CLI 外层并列的新协议。
- 不能在本 FR 内引入持久化分析基座或新的共享数据模型。
- 不能要求 FR-0002 为 FR-0004 新增专用握手阶段、专用分片协议或额外 transport 状态。

## 测试与验证策略

单元测试：

- 诊断分类映射
- 失败位置选择
- 页面状态字段标准化
- 关键请求摘要字段标准化
- URL 净化测试
- 脱敏与截断规则

契约测试：

- 成功响应仍遵循 FR-0001 外层壳
- 错误响应仍遵循 FR-0001 外层壳
- `observability` 与 `error.diagnosis` 结构稳定可解析
- 关键失败面可被机器断言区分
- 诊断字段不会突破 FR-0002 已冻结的承载边界

集成测试：

- 首个平台成功闭环输出最小观察面
- 页面变化场景输出“页面变化”诊断
- 请求失败场景输出“请求失败”诊断
- 执行中断场景输出“执行中断”诊断
- 运行时不可用场景输出“运行时不可用”诊断

人工验证：

- 在真实浏览器执行一次 Phase 1 样本，确认输出可读、可解析、可定位问题
- 确认诊断字段没有泄露敏感原文
- 确认发生裁剪时有显式截断标记，而不是静默丢字段

完成证据至少包括：

- 诊断相关单测通过
- CLI 契约测试通过
- `docs-guard` 或等价文档校验通过
- 核心失败场景的人工 smoke 通过

## TDD 范围

默认先写测试的模块：

- 诊断分类映射
- 失败位置选择
- 观察面序列化
- 脱敏与截断
- 成功 / 错误响应拼装

暂不强制先写测试的部分：

- 真实站点上的观测采样细节
- 人工调试日志格式
- 只用于排障的临时提示文案

这样划分的原因：

- 这些稳定逻辑后续会被 Phase 1、Phase 2 反复复用
- 真实站点采样依赖外部环境，适合先把契约定下来，再按契约实现

## 并行 / 串行关系

前绪阻塞：

- FR-0001 外层 CLI 契约必须已冻结
- `#355` 对运行时通信链路的前置工作必须可承接诊断数据
- Phase 1 首个平台的观察事件源必须明确

可并行项：

- 诊断分类与页面状态字段可以先行冻结
- 关键请求摘要与失败位置映射可以并行设计
- 测试用例与契约文档可以与实现方案并行对齐

串行关系：

- 先冻结观察字段，再实现采集
- 先冻结分类与失败位置，再做输出拼装
- 先完成契约测试，再放开后续 FR 消费

## 进入实现前条件

以下条件是未来进入 `feat/FR-0004-*` 实现工作的门禁，当前未满足前不得进入：

- spec-only 变更先进入 Draft PR 或等价评审上下文
- spec review 结论达到 `APPROVE`
- `ready_for_implementation = true`
- `contracts/runtime-observability.md` 与 `risks.md` 随套件一起完成评审
- `TODO.md` 中的 review blockers 清空

在这些条件满足前，明确禁止：

- 编写 FR-0004 的运行时代码
- 把完整暂停 / 恢复系统混入当前实现切片
- 破坏 FR-0001 已冻结的外层错误壳与退出码

在 formal review 通过并完成 closeout 后，本计划只保留后续实现消费边界；不在此文件补写外部 PR、guardian 或 merge 历史事实。
