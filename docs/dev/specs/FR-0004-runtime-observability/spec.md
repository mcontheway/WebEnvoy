# FR-0004 最小观察、错误分类与结构化诊断能力

## 背景

Phase 1 已经明确要先打通首个 L3 最小闭环，但如果命令只返回黑盒式 `success / fail`，就算流程跑通，也无法判断问题出在页面变化、关键请求失败，还是执行链路中断。

`#154` 吸收了旧的“侦察建模”诉求中，首阶段真正需要的部分，目标不是提前建立完整暂停 / 恢复系统，而是在不改变 FR-0001 外层错误壳和退出码的前提下，补上最小观察、最小错误分类和结构化诊断能力。

## 目标

1. 在 FR-0001 冻结的 CLI 外层契约之内，补充最小可复用的运行时观察面。
2. 让首个平台的 Phase 1 闭环至少能够返回页面状态、关键请求和失败位置，而不是只返回成败结论。
3. 让运行时失败具备最小结构化诊断，能够区分页面变化、请求失败、执行中断和运行时不可用等首阶段关键失败面。
4. 让这套诊断结构足以支撑 Phase 1 调试与 Phase 2 封装，但不扩张为完整 trace、replay 或 pause / resume 系统。

## 非目标

- 不建立完整暂停 / 恢复系统。
- 不建立长期运行的观测平台、链路追踪平台或会话回放系统。
- 不把原始请求体、Cookie、Token、全部响应体直接暴露给调用方。
- 不改写 FR-0001 已冻结的外层成功 / 错误壳、命令语法和退出码语义。
- 不把这套诊断能力扩张成独立的运维观测基座。

## 功能需求

### 1. 最小观察面

- 对 Phase 1 的首个平台闭环，系统必须能返回最小页面状态。
- 最小页面状态至少应覆盖当前页面的可判定信息，例如页面类型、URL、标题、加载状态或其他等价的稳定标识。
- 系统必须能返回当前命令窗口内的关键请求摘要。
- 关键请求摘要只保留可用于诊断的结构化元数据，不返回敏感明文负载。
- 页面状态与关键请求中的 URL 只允许返回净化后的规范化地址，必须默认去除 `query`、`fragment` 和签名类敏感参数，不得原样暴露。

### 2. 失败位置

- 当命令失败时，系统必须能返回失败位置。
- 失败位置至少要能区分页面加载、请求、交互、收敛等待、运行时连接这类首阶段关键位置。
- 失败位置必须能对应到具体责任面，例如 CLI、Extension、Content Script、页面或网络。

### 3. 最小错误分类

- 在不改变 FR-0001 外层错误码集合的前提下，系统必须补充更细的运行时诊断分类。
- Phase 1 至少要能把以下失败面区分开：
  - 页面变化
  - 请求失败
  - 执行中断
  - 运行时不可用
  - 未知 / 无法判定
- 错误分类用于诊断，不替代 FR-0001 的外层 `error.code`。

### 4. 结构化诊断

- 命令失败时，必须返回可机器解析的结构化诊断对象。
- 诊断对象至少应包含：
  - 诊断分类
  - 失败位置
  - 责任组件
  - 证据摘要
- 证据摘要必须短、可截断、可过滤，不得把完整页面内容或完整请求体直接塞回去。

### 5. 兼容 FR-0001

- 所有成功 / 失败响应仍必须遵循 FR-0001 冻结的外层响应壳与退出码。
- 本 FR 只能在 FR-0001 允许的响应结构内做加法，不能发明另一套外层失败返回。
- 任何后续 FR 若要消费这套诊断能力，必须复用这里定义的最小结构。

## GWT 验收场景

### 场景 1：首个平台闭环成功时返回最小观察面
Given 一个 Phase 1 的已实现读闭环执行成功  
When 命令结束并返回结果  
Then 响应仍遵循 FR-0001 的成功外层结构  
And 响应包含最小页面状态与关键请求摘要  
And 这些观察字段可以被机器稳定解析  

### 场景 2：页面变化导致失败时能定位到页面层
Given 命令执行过程中页面结构与预期不一致  
When 处理器判定当前流程无法继续  
Then 响应仍遵循 FR-0001 的错误外层结构  
And 诊断对象能把失败归类为“页面变化”  
And 失败位置能指向页面层或其对应的执行阶段  
And 调用方能据此区分这是页面改版，不是参数错误  

### 场景 3：关键请求失败时能定位到请求层
Given 命令执行依赖的关键请求返回异常、超时或非预期状态  
When 命令失败  
Then 响应仍遵循 FR-0001 的错误外层结构  
And 诊断对象能把失败归类为“请求失败”  
And 关键请求摘要中能找到对应请求的结构化线索  

### 场景 4：执行中断时能定位到执行链路
Given 命令在执行过程中被浏览器断连、页面中断或流程提前终止  
When 命令结束  
Then 响应仍遵循 FR-0001 的错误外层结构  
And 诊断对象能把失败归类为“执行中断”  
And 失败位置能指向执行链路中的具体阶段  

### 场景 5：运行时不可用时不伪造页面诊断
Given 底层运行时尚未就绪或无法连接  
When 命令尝试进入执行阶段  
Then 响应仍遵循 FR-0001 的错误外层结构  
And 诊断对象能把失败归类为“运行时不可用”  
And 系统不应伪造不存在的页面状态或关键请求

## 异常与边界场景

- 页面尚未完成加载时，页面状态可以不完整，但必须明确标记为部分可观测，而不是伪装成完整成功状态。
- 关键请求为空时，允许返回空列表，但必须说明当前命令没有可用请求证据，而不是返回空壳。
- 当同一轮执行里出现多个失败信号时，必须保留最接近根因的主失败位置，其余信号只能作为补充证据。
- 任何诊断字段都不得包含 Cookie、Authorization、Token、完整响应体或其他高敏感原文。
- 任何诊断字段中的 URL 都必须先净化再返回，默认移除 `query` 与 `fragment`，不得原样透出签名、回调或一次性凭证。
- 响应体必须受 Native Messaging 传输上限约束，诊断内容过长时必须截断并显式标记。
- 若观测采集失败，系统仍应返回 FR-0001 外层错误壳，并把诊断降级为最小可用状态，而不是抛出未结构化异常。

## 验收标准

1. FR-0004 能清楚说明 Phase 1 为什么需要最小观察与结构化诊断。
2. 成功响应和失败响应都能在 FR-0001 外层壳内携带统一的最小观测结构。
3. 页面变化、请求失败、执行中断、运行时不可用这几类失败面可以被清楚区分。
4. 诊断信息足以支撑 Phase 1 排障与 Phase 2 的能力封装，而不需要额外脑补。
5. 套件没有引入完整暂停 / 恢复、回放或长期观测平台的范围蔓延。
6. 任何新增字段都保持向后兼容，后续 FR 可以读取但不能重定义外层失败语义。
