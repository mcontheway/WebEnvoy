# FR-0007 实施计划

## 实施目标

本 FR 的实现目标是把“能力执行”从一次性命令提升为可复用的最小封装壳，且不改写 FR-0001 外层 CLI 契约。

实现完成后，至少形成以下交付：

- 能力调用最小入口（复用既有 `webenvoy <command> --params` 模式）
- 能力输入最小壳：`ability` + `input` + `options`
- 能力输出最小壳：`summary.capability_result`
- 能力错误最小壳：`error.details`（含 `ability_id` / `stage` / `reason`）
- `run_id` 在能力层贯穿成功与失败路径
- 面向首个 L3 样本的能力壳契约测试

本 FR 不交付平台业务逻辑、能力分享/导入系统、版本分发系统。

## 分阶段拆分

### 阶段 A：契约冻结与工程映射

产出：

- 能力壳字段字典与命令映射规则
- `contracts/ability-shell.md` 与实现模块映射关系

依赖：

- FR-0001 契约冻结
- FR-0007 spec review 通过

### 阶段 B：输入壳与校验层落地

产出：

- `params.ability` / `params.input` / `params.options` 解析与校验
- 参数错误到外层错误壳的稳定映射

依赖：

- 阶段 A 完成

### 阶段 C：输出壳与错误细节落地

产出：

- `summary.capability_result` 统一组装
- `error.details` 统一映射（`ability_id` / `stage` / `reason`）

依赖：

- 阶段 B 完成

### 阶段 D：首个 L3 样本挂接与回归

产出：

- 在 `#145` 样本命令中接入能力壳
- 成功、参数错误、执行失败、输出映射失败四类路径回归

依赖：

- 阶段 C 完成
- FR-0002 已冻结的最小通信契约与既有实现链路可被复用

### 阶段 E：测试收口与实现准入证据

产出：

- 能力壳契约测试报告
- 与 FR-0004 诊断字段对齐说明
- 不破坏 FR-0001 退出码/外层壳的回归证据

依赖：

- 阶段 D 完成

最小闭环：

- 到阶段 D 结束，首个 L3 样本已按统一能力壳被调用，不再是一次性命令特例。

## 实现约束

- 必须保持 CLI-first，不引入 SDK / API / daemon。
- 禁止改写 FR-0001 的外层调用语法、`stdout` 单 JSON 规则和最小退出码语义。
- 禁止把 FR-0004 的 `observability` / `error.diagnosis` 结构重写进能力壳；能力层只补 `summary.capability_result` 与 `error.details`。
- 禁止让 FR-0006 的持久化结果反向决定能力响应是否成功返回；落库只能消费能力壳，不能重塑能力壳。
- 不把能力壳扩张为能力市场、导入、签名、版本发布机制。
- 不在本 FR 内新增持久化 schema；如后续需要，另开数据模型规约。
- 不允许同一能力在同一主版本内出现字段语义漂移。

## 测试与验证策略

单元测试：

- 能力输入壳解析与字段校验
- 能力输出壳组装
- `error.details` 阶段映射（`input_validation` / `execution` / `output_mapping`）
- `run_id` 透传与一致性断言

集成 / 契约测试：

- 真实 CLI 调用下的能力壳成功返回
- 缺失 `ability`、缺失 `input`、字段类型错误的参数失败返回
- 执行失败与输出映射失败的结构化错误返回
- 不破坏 FR-0001 外层错误壳与退出码
- 与 FR-0004 `observability` / `error.diagnosis` 共存时不发生字段冲突
- 与 FR-0006 运行记录映射对齐时，只复用 `run_id` 与最小能力字段，不新增响应层专属持久化字段

人工验证：

- 以同一 `ability.id` 重复调用，确认输入/输出字段稳定
- 对接 FR-0004 的最小诊断输出，确认可通过 `run_id` 串联定位

完成证据至少包括：

- 能力壳相关自动化测试通过
- `docs-guard` 与 `spec-guard` 对 FR-0007 套件通过
- 首个 L3 样本能力壳回归记录可追踪
- 与 FR-0004 / FR-0006 的共享字段边界已有文档级对齐结论

## TDD 范围

默认先写测试的模块：

- 能力输入壳校验器
- 能力输出壳映射器
- 能力错误细节映射器
- `run_id` 与能力壳关联逻辑

暂不强制先写测试的部分：

- 能力文案字段或展示层提示文本
- 非契约性调试日志内容

划分原因：

- 前者属于跨 FR 共享契约，必须先锁定可回归行为。
- 后者属于低风险展示信息，不应阻塞能力壳主闭环。

## 并行 / 串行关系

前绪阻塞：

- FR-0007 spec review 通过前，不进入能力壳实现。
- FR-0001 外层 CLI 契约若未稳定，不应推进能力壳编码。

实现阶段串行主链路：

- 阶段 A → B → C → D → E

可并行事项：

- FR-0002 formal closeout 之后的通信契约 / 实现链路与 `#356` 可并行承接本 FR，只要不改写能力壳边界。
- FR-0004 可并行补充诊断字段，但必须复用 `run_id` 与外层错误壳。
- FR-0006 可并行推进其自身规约；若涉及能力结果 / 错误字段映射，只能消费本 FR 已冻结字段，不能反向定义响应层。

## 进入实现前条件

满足以下条件后，才允许进入 `feat/FR-0007-*` 实现分支：

- 当前 FR-0007 套件已进入 Draft PR 或等价评审上下文
- spec review 结论为 `APPROVE`
- `ready_for_implementation = true`
- `contracts/ability-shell.md` 与 `risks.md` 已完成评审
- `TODO.md` 中 spec review 阶段阻断项已清空

在以上条件满足前，禁止：

- 编写 FR-0007 运行时代码
- 在 `#145` 中以临时字段绕过能力壳契约
- 将 FR-0007 契约变更与实现代码混入同一 PR
