## 实施目标

在不新增实现代码的前提下，为 XHS 读写路径建立统一风险审查与保护门禁规约，形成可供对应 issue 线程、后续 formal 套件与 `#201` exit review 引用的治理基线，并把历史风险结论固定下来。

## 分阶段拆分

### 阶段 A：范围与边界冻结（规约阶段）

- 冻结读路径（`#209` 历史闭环与其后续扩展）与写路径（`#208` 历史边界与其后续事项）的审查范围。
- 冻结读域/写域分离规则。
- 输出不可做事项（不做绕过风控、不做完整发布能力）。

### 阶段 B：门禁与恢复条件定义（规约阶段）

- 定义默认停高风险 live 实验策略。
- 定义人工确认门禁与 dry-run/侦察门禁。
- 定义恢复 live 的最小证据条件与审批条件。

### 阶段 C：消费关系收口（规约阶段）

- 输出 `#208/#209` 的 issue 线程、后续 formal 套件与恢复讨论可直接引用的治理要求。
- 输出 `#201` / Phase 1 exit review 可引用的历史安全结论。

### 阶段 D：定位张力与差距清单收口（规约阶段）

- 明确插件事实中枢与 CLI-first 叙事之间的治理张力。
- 形成“文档已设计 / Phase 1 已落地 / 仍未落地”差距清单。
- 将差距项与当前风险预警建立可复核映射，作为后续事项统一口径输入。

## 实现约束

1. 本 FR 仅做正式规约，不写运行时代码。
2. 不扩张到账号矩阵、调度系统或运营系统。
3. 不把 `#209` 回滚为未完成事项，但限制其后续高风险 live 扩展。
4. 不把“插件定位分析”扩张成新架构重构任务；仅输出可被门禁消费的治理口径与差距清单。
5. 本 FR 的目标是风险收敛与账号安全，不承接隐蔽性增强、绕过风控或其他对抗策略设计。

## 测试与验证策略

1. 文档结构验证：`spec.md`、`plan.md`、`TODO.md` 完整，且附加文档满足风险事项触发条件。
2. 一致性验证：`#213`、FR-0009、PR 描述三者范围一致，均为“读写风险审查”。
3. 门禁可引用性验证：评审者可直接据文档判断：
   - `#208` 的后续 live / replay / 恢复讨论需要补齐哪些治理前置
   - `#209` 的后续 live 扩展不能被既有读闭环自动推导为已放行
4. 静态门禁验证：`docs-guard`、`spec-guard`、`git diff --check` 必须通过。
5. 风险证据可复核性验证：代码级风险发现必须具备明确文件路径与可复查解释。
6. 差距清单可引用性验证：`#201/#208/#209` 的 issue 同步与后续 formal 套件能直接引用 gap 项与对应风险含义。
7. 审批与审计留痕验证：评审者可直接据文档判断责任人、审批记录载体与审计留痕方式。

## TDD 范围

- 本 FR 为规约文档事项，不包含运行时代码变更，故不要求单元测试先行。
- 若后续进入门禁实现阶段，再由实现 FR 明确：
  - 契约测试（门禁决策输入输出）
  - 错误模型测试（阻断与放行判定）
  - 配置/模式切换测试（dry-run vs live）

## 并行 / 串行关系

- 串行：
  - `FR-0009` spec review 通过前，`#208` 的后续事项不得继续高风险 live 正式验证。
  - `FR-0009` spec review 通过前，`#209` 的后续事项不得新增高风险 live 扩展实验。
  - `FR-0009` 的 formal closeout 必须先完成，之后才能把其治理结论同步到 `#201/#208/#209`。
  - `FR-0009` 的结论写回必须晚于 canonical formal-review PR 合入与 guardian `APPROVE`。
- 可并行：
  - 不触发高风险 live 的文档梳理、风险证据整理与侦察输入归档可并行推进。

## 进入实现前条件

1. FR-0009 完成 spec review 且无阻断项。
2. `risk_gate_contract` 四对象完整（见 `contracts/risk-gates.md`）：
   - `path_scope`
   - `risk_state`
   - `execution_mode_gate`
   - `resume_requirements`
3. 明确 `#208` 的 live 恢复讨论需要引用的条件与禁止动作清单。
4. 明确 `#209` 后续 live 扩展需要引用的审批条件与审计要求。
5. 相关事项（`#201`、`#208`、`#209` 的 issue comment）已同步引用 FR-0009 作为治理基线。
6. 已完成“插件定位张力 + 设计-实现 gap 清单”并通过 spec review 认定可消费。
7. 已明确人工确认责任人、审批记录载体与审计留痕方式，且责任边界不外溢到运行时持久化。

## Spike Readiness 分级（规约准入）

- `evidence_readiness`：BLOCKED（当前有风险预警事实，但缺统一归档的复核证据集）
- `primary_candidate`：PASS（以“默认停高风险 live + 人工确认 + dry-run”为主候选）
- `fallback`：PASS（保留纯侦察/文档化推进，不做 live 扩张）
- `admission_ready`：BLOCKED（需先完成 spec review 与前置条件收口）
