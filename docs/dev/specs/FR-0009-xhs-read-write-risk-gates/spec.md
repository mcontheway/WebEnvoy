# FR-0009 XHS 读写路径风险审查与保护门禁

## 背景

`#209` 已经把 `xhs.search` 的读路径打通并合入；`#208` 对应的最小页面交互验证边界也已在下游 formal 套件中冻结。当前更重要的问题不再是“单条路径能否跑通”，而是“后续任何 live 扩展、replay、恢复与写路径验证，是否都已被统一纳入可控风险边界”。

同时，小红书读写路径不在同一域名：

- 读域：`www.xiaohongshu.com`
- 写域：`creator.xiaohongshu.com/publish/...`

如果继续把 live 实验当作默认操作，会放大账号风险，也会把后续真实交互验证、replay 与恢复判断混在一起。本 FR 用于先冻结风险审查对象、保护门禁与恢复条件，再允许后续读写事项进入 live 验证。

## 目标

1. 明确 XHS 读写路径的统一风险审查范围，覆盖 `#209` 已落地读路径与 `#208` 留下的页面交互 / 写路径边界。
2. 明确读域/写域分离要求及其共同风险约束，防止执行入口、登录态与注入范围混淆。
3. 建立高风险 live 实验默认停用策略，并定义人工确认和 dry-run/侦察门禁。
4. 输出可被 `#208/#209` 的后续 live 扩展、replay、恢复事项，以及 `#201` 的历史 exit review 语义直接消费的安全前置结论。
5. 明确“插件已是事实执行枢纽，但当前产品叙事仍以 CLI-first 为主”的定位张力，并将其纳入门禁治理范围。
6. 明确本 FR 的目标是风险收敛与账号安全，不是更隐蔽执行或绕过平台风控。


## 非目标

- 不交付完整发布能力或完整写闭环。
- 不把本 FR 扩展为账号矩阵、运营系统或平台策略自动博弈系统。
- 不回滚 `#211`，但其后续 live 扩展必须服从本 FR 门禁。
- 不把本 FR 解释为隐蔽性增强、风控绕过或对抗策略设计。

## 功能需求

### 1. 读写审查范围冻结

- 必须同时审查：
  - `#209` 的读路径（`xhs.search` 及其 live 执行面）
  - `#208` 留下的写路径 / 页面交互验证边界，以及其后续 live、replay、恢复事项
- 必须输出每条路径的：
  - 域名范围
  - 登录前置
  - 注入范围
  - tab 选择边界
  - live 风险等级

### 2. 读域/写域分离与共同约束

- 必须显式区分读域与写域，不允许以“主站可用”推导“发布域可用”。
- 对两域必须同时给出共同风险约束：
  - 会话来源一致性
  - 自动化特征暴露面
  - 风险触发后的停机与恢复流程

### 3. 高风险 live 实验默认停用

- 在本 FR 审查通过前：
  - 不得以 `#208` 已有 formal 冻结边界为理由恢复新的高风险 live 正式验证
  - 不得以 `#209` 已形成读闭环为理由扩展新的高风险 live 实验面
- 允许的默认执行形态应为 dry-run / 侦察，不产生不可逆写动作。
- 如需恢复 limited / allowed live，必须同时满足：
  - formal spec review 已通过
  - 风险复核已完成
  - `#208` 的动作边界与 `#209` 的扩展边界均已显式冻结
  - 审批已被真实记录，且其正式载体与后续执行契约一致，不存在口头放行

### 4. 人工确认与门禁机制

- 必须定义人工确认门禁，至少覆盖：
  - 执行目标域确认
  - 账号风险状态确认
  - 动作类型确认（读、写、不可逆写）
- 必须定义人工确认流程中的最小责任与留痕：
  - 规约阶段默认责任人是发起本次 live 恢复 / 扩展请求的实现负责人
  - formal closeout 的 review / sync 记录可保留在 GitHub issue / PR 中，供历史追溯与 review 复核
  - 后续 limited / allowed live 的审批与审计正式载体由执行层契约承接；本 FR 不把 GitHub review/comment 升格为未来运行时审批通道
- 必须定义 dry-run/侦察门禁，至少覆盖：
  - 默认模式
  - 升级到 live 的前置条件
  - 审计记录要求

### 5. 结论交付与消费边界

- 必须输出给 `#208` 及其后续承接事项的前置结论：
  - 是否允许恢复 live 最小交互验证
  - 允许的动作边界与禁止动作
- 必须输出给 `#209` 后续 live 扩展事项的前置结论：
  - 是否允许新增 live 扩展实验面
  - 允许的模式边界、审批要求与审计要求
- 必须输出给 `#201` 历史 exit review 语义的补充结论：
  - Phase 1 出站是否仍受风险门禁阻断

### 6. 设计-实现差距清单必须显式化

- 必须给出“文档已设计 / Phase 1 已落地 / 仍未落地”三段式 gap 清单，至少覆盖：
  - 读路径执行模式
  - 写路径候选成熟度
  - 行为层反风控（Layer 3/4）
  - 插件中枢治理
  - 读写域边界治理
- 必须说明每条 gap 与当前风险预警之间的关系，避免只列抽象差距不列风险含义。

## GWT 验收场景

### 场景 1：读写路径都被纳入同一风险审查口径

Given `#209` 已落地且 `#208` 的后续验证边界已被冻结
When 评审者检查 FR-0009 输出
Then 能同时看到读路径与写路径的风险审查对象
And 不存在“只审写不审读”或“只审读不审写”的缺口

### 场景 2：读域与写域分离被明确且可执行

Given 小红书存在主站读域与发布写域
When 评审者检查域名与执行边界条目
Then 文档明确区分 `www.xiaohongshu.com` 与 `creator.xiaohongshu.com`
And 对两域给出共同风险约束与禁止混推规则

### 场景 3：高风险 live 实验默认停用生效

Given 风险审查尚未通过
When 评审者检查执行策略
Then `#208` 后续承接事项的 live 正式验证被明确阻断
And `#209` 后续高风险 live 扩展被明确阻断
And 默认允许形态为 dry-run / 侦察

### 场景 4：人工确认与 dry-run 门禁可被后续事项消费

Given 后续事项需要继续读写验证
When 评审者检查门禁输出
Then 能得到可直接消费的人工确认步骤与门禁条件
And `#208/#209` 的后续事项可以据此定义进入 live 的前置条件

### 场景 5：插件定位与差距清单可被直接引用

Given `xhs.search` 已在插件链路中落地
When 评审者检查 FR-0009 的研究与规约输出
Then 能明确识别“插件事实中枢”与“CLI-first 叙事”之间的治理张力
And 能看到“文档已设计 / Phase 1 已落地 / 仍未落地”的差距清单
And 每条差距都说明了与当前风险预警的直接关联

## 异常与边界场景

1. 账号风控告警：出现风控告警后，默认进入“暂停高风险 live 实验”状态，不得继续放大实验面。
2. 域名误判：把发布域当作主站子路径处理时，必须判为边界错误并阻断。
3. 登录态失配：读域可用不代表写域可用；任一域登录失配都应阻断 live 执行。
4. 不可逆写动作：任何可能触发不可逆写入的路径，在门禁未放行前一律禁止。
5. 证据不足：只有单次样本或不可复核样本时，不得放行为 live 恢复结论。
6. 定位漂移：若后续事项把插件层风险面降格为“CLI 参数问题”，必须判定为治理边界错误并阻断。
7. 审批留痕缺失：若后续执行契约要求的审批 / 审计载体未明确定义或不可复核，同样不得放行为 limited / allowed live。

## 验收标准

1. FR-0009 套件完整并可审（spec/plan/TODO + 风险相关附加文档）。
2. 明确覆盖读路径与写路径，不遗漏 `#209` 或 `#208` 任一侧。
3. 明确读域/写域分离规则与共同风险约束。
4. 明确“默认停高风险 live 实验 + 人工确认 + dry-run/侦察门禁”。
5. 明确 `#208/#209` 的后续事项在本 FR 审查通过前不得恢复或扩张高风险 live。
6. 研究输出中有代码级风险发现清单，并附可复核文件路径与风险解释。
7. 研究输出中有插件定位张力与设计-实现 gap 清单，且能被 `#201/#208/#209` 直接引用。
