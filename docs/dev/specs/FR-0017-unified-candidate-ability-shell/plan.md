# FR-0017 实施计划

## 实施目标

为 FR-0017 owning Work Item 冻结统一候选能力描述与能力壳的正式输入，使后续实现 PR 可以围绕一套稳定的候选能力模型落地，而不再从 roadmap、issue 正文或临时样本里手工拼装字段与边界。

## 分阶段拆分

### 阶段 A：定位与继承边界冻结

- 产出：
  - `spec.md` 中的 Phase 2 定位
  - 与 `FR-0007`、`FR-0004`、`FR-0006` 的继承边界
- 重点：
  - 明确本 FR 只冻结候选能力描述，不承担验证、导入或交付

### 阶段 B：稳定对象与数据模型冻结

- 产出：
  - `contracts/candidate-ability-shell.md`
  - `data-model.md`
- 重点：
  - 冻结 `candidate_ability_descriptor`
  - 冻结与最小能力壳的接线方式
  - 明确最小生命周期与来源证据字段

### 阶段 C：L3 / L2 / 三类能力面的统一表达冻结

- 产出：
  - `spec.md` 中的 L3/L2 承载边界
  - 读 / 写 / 下载统一表达边界
- 重点：
  - 证明同一套描述可同时承载至少一个 L3 样本与一个 L2 样本

### 阶段 D：下游 handoff 收口

- 产出：
  - `TODO.md` 中进入实现前条件
  - 指向 `FR-0018/0019/0021` 的下游衔接说明
- 重点：
  - 明确验证/重放和 L2 首次可用如何消费本 FR

## 实现约束

1. 本 FR 只提交规约文档，不提交运行时代码。
2. 不重定义 `FR-0007` 的最小能力输入/输出/错误壳。
3. 不把候选能力描述提前扩写成导入格式、分享协议或版本治理对象。
4. 不把验证结果、回放结果或可信判断对象混入本 FR。
5. 不因为 L3/L2 差异而引入第二套候选能力描述协议。
6. 不把下载能力从统一模型中排除。

## 测试与验证策略

- 规约阶段门禁：
  - `bash scripts/docs-guard.sh`
  - `bash scripts/spec-guard.sh`
  - `git diff --check`
- 审查重点：
  - 与 `FR-0007` 的继承边界是否清楚
  - 读 / 写 / 下载统一表达是否成立
  - L3/L2 是否继续复用同一套描述对象
  - 是否把验证或交付语义提前混入本 FR
- 实现前验证要求：
  - 候选能力描述对象能直接映射到实现配置与测试夹具
  - 最小生命周期与来源证据字段能直接被后续验证/重放消费

## TDD 范围

- 本 FR 处于规约阶段，不直接新增实现测试。
- 后续实现 PR 默认纳入 TDD 的模块：
  - 候选能力描述解析与校验
  - 能力壳映射
  - 生命周期状态校验
  - L3/L2 样本到描述对象的转换
- 不强制 TDD 的部分：
  - 仅用于样本采集的临时人工实验

## 并行 / 串行关系

### 串行前置

- `FR-0007` 已完成 formal closeout，可作为本 FR 的上游最小能力壳基线。
- 本 FR spec review 未通过前，不进入能力封装实现 PR。

### 可并行

- `FR-0018` 的验证/重放方向可以并行讨论，但最终对象必须消费本 FR 冻结的能力描述。
- `FR-0019` 的 L2 首次可用方向可以并行收口，但其输出对象不得绕开本 FR。

### 串行后置

- `FR-0018` 正式进入实现前，必须先引用 FR-0017 的稳定能力描述对象。
- `FR-0019` 若要把首次成功路径沉淀为候选能力，必须先消费 FR-0017 的描述边界。
- `FR-0021` 若要进入统一能力模型，必须先继承本 FR 冻结的描述方式。

## 进入实现前条件

1. FR-0017 规约 PR 完成 spec review 且无阻断项。
2. `contracts/candidate-ability-shell.md` 中的稳定对象语义无歧义。
3. `data-model.md` 已明确哪些字段属于候选能力正式描述，哪些字段不属于。
4. reviewer 明确认可本 FR 没有提前混入验证、导入、分享或版本治理语义。
5. 后续实现 issue / PR 已明确：
  - FR-0017 owning Work Item 负责承接本 FR 的能力壳实现
  - `FR-0018/0019/0021` 只能消费本 FR，而不是并行改写它
