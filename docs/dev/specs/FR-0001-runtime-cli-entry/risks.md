# FR-0001 风险

FR-0001 定义的是后续 Sprint 1 事项共同依赖的 CLI 外层契约，因此需要单独识别哪些地方不能失手。

## 关键风险

| 风险 | 触发条件 | 影响 | 缓解手段 | 观察信号 | 回滚 / 收口方式 |
|---|---|---|---|---|---|
| `stdout` 被人类日志污染 | 入口、帮助、调试信息直接打印到 `stdout` | 上层 Agent / 脚本无法稳定解析 JSON | 冻结 `stdout` 单 JSON 规则；增加污染断言 | CLI 输出无法被 JSON 解析 | 阻断合并，回到统一输出层收口 |
| 退出码与错误码漂移 | 不同命令各自定义失败码 | 调用方无法基于退出码自动处理失败 | 冻结最小退出码表；集中常量管理 | 同类错误在不同命令返回不同退出码 | 以 FR-0001 契约为准统一回收 |
| Scope Creep 把完整命令面塞进 FR-0001 | 为“未来方便”提前承诺安装、服务化、完整平台命令 | Phase 1 入口项失焦，拖慢 `#142`、`#143`、`#145` | 明确非目标；只冻结命名空间与外层壳 | `spec.md` 或实现开始出现 SDK / daemon / 全命令目录 | 从 FR-0001 剥离到后续 FR |
| 后续 FR 绕过统一命令上下文 | `#142`、`#143`、`#145` 自行发明 `run_id`、错误壳或命令格式 | Sprint 1 出现多套 CLI 契约，后续难以收口 | 在 spec review 明确承接关系；要求复用 `contracts/cli-entry.md` | 后续 FR 文档或实现出现第二套外层字段 | 阻断 review，要求回到 FR-0001 契约 |
| 运行时不可用与执行失败混成一类 | 处理器缺少错误边界治理 | 调用方无法区分“稍后重试”与“逻辑失败” | 冻结最小错误类别；让 `runtime_unavailable` 单独成类 | 失败路径统一返回 `ERR_EXECUTION_FAILED` | 回到错误分类层重新拆分 |

## Stop-Ship 条件

出现以下任一情况时，不应进入实现或合并：

- 还不能保证 `stdout` 单 JSON 输出
- 还不能稳定区分未知命令、未实现命令、参数错误
- 还没有冻结最小退出码语义
- 后续事项需要通过破坏 FR-0001 契约才能接入

## 剩余风险

- 具体命令 argv 语法在实现阶段仍需收口，但不得影响这里冻结的逻辑命令上下文
- `#154` 的诊断字段后续可能扩展；扩展允许，但不得破坏 FR-0001 的外层错误壳
