# FR-0001 实施计划

## 实施目标

这次实现只交付 CLI 最小入口与可集成契约骨架，不交付真实平台能力。

本 FR 的实现完成后，应至少形成以下产物：

- 单一可执行入口 `webenvoy`
- 冻结的最小 argv 调用语法：`webenvoy <command> [--params '<json>'] [--profile <profile>] [--run-id <run_id>]`
- 标准化命令上下文：`run_id`、`command`、`profile`、`params`、`cwd`
- 稳定的命令注册 / 路由层
- 统一的成功 / 错误响应壳
- 稳定退出码映射
- 最小帮助或诊断命令，以及已注册但未实现命令的占位处理器
- 覆盖 CLI 契约的最小测试与验证证据

本 FR 不交付 Native Messaging、Profile 运行时、SQLite 持久化、平台读取逻辑或完整命令面。

## 分阶段拆分

### 阶段 A：工程骨架与入口建立

产出：

- Node / TypeScript 最小工程骨架
- `webenvoy` 入口与基础执行方式
- CLI 启动层与命令调度层分离

依赖：

- 以当前 spec / contract 冻结结果为准
- 不依赖 `#142`、`#143`、`#145` 的实现完成

### 阶段 B：CLI 契约核心落地

产出：

- 最小 argv 解析规则
- 统一运行上下文模型
- 成功 / 错误响应格式化器
- 退出码常量与错误分类常量
- `stdout` / `stderr` 边界治理

依赖：

- 阶段 A 完成

### 阶段 C：路由与占位处理器

产出：

- 命令注册表
- `runtime.<verb>` 与 `<platform>.<verb>` 的最小命名空间承载
- 未知命令与未实现命令的稳定区分

依赖：

- 阶段 B 完成

### 阶段 D：最小已实现命令与入口可调用验证

产出：

- 至少一个已实现的最小运行时命令
- 帮助 / 诊断路径可调用
- `run_id` 贯穿成功与失败路径
- `ERR_EXECUTION_FAILED` 与 `ERR_RUNTIME_UNAVAILABLE` 都有可断言的失败路径

依赖：

- 阶段 C 完成

### 阶段 E：测试、收口与交接

产出：

- CLI 契约测试
- 退出码断言
- `stdout` 污染断言
- 对后续 `#142`、`#143`、`#145`、`#154` 的承接说明

依赖：

- 阶段 D 完成

最小闭环：

- 到阶段 D 结束时，应已形成“一个命令可成功、五类冻结失败语义都可稳定断言”的最小 CLI 闭环

## 实现约束

- 必须保持 CLI-first，不引入 SDK、HTTP API 或 daemon 常驻服务。
- `stdout` 只允许输出单个 JSON 对象；帮助信息、调试日志和堆栈不得污染 `stdout`。
- 只冻结最小命令上下文、错误壳和退出码，不提前铺开完整命令目录。
- 只冻结最小 argv 语法，不引入第二套外层调用方式。
- 不在本 FR 内接入 Native Messaging、Extension、Playwright、SQLite 的真实运行逻辑。
- 不为后续事项预埋需要马上兑现的复杂插件系统或能力分发机制。
- 后续 FR 允许新增命令和字段，但不能破坏本 FR 冻结的外层契约。

## 测试与验证策略

单元测试：

- argv 解析与标准化
- 命令上下文标准化
- 成功 / 错误响应格式化
- 错误码到退出码映射
- 未知命令 / 未实现命令分流
- `ERR_EXECUTION_FAILED` 与 `ERR_RUNTIME_UNAVAILABLE` 分流

集成 / 契约测试：

- 入口执行到已实现命令的完整调用链
- `<command>` 位置参数解析
- `--params` JSON 入口解析
- `--profile` 与 `--run-id` 的透传
- `stdout` 单 JSON 输出
- `stderr` 不污染 `stdout`
- 非零退出码断言
- `ERR_EXECUTION_FAILED` 输出断言

人工验证：

- 本地直接调用 `webenvoy`
- 帮助 / 诊断路径 smoke test
- 观察最小 argv 语法是否足以覆盖调用方输入
- 观察调用方是否能稳定解析 JSON 输出

完成证据至少包括：

- CLI 基础测试通过
- `docs-guard` 通过
- `spec-guard` 对 FR-0001 套件通过

## TDD 范围

默认先写测试的模块：

- argv 解析与标准化
- 命令上下文标准化
- 成功 / 错误响应壳
- 退出码映射
- 命令注册 / 路由分发
- 已知失败类型的结构化输出

暂不强制先写测试的部分：

- 构建脚本与本地打包细节
- 帮助文案文本内容
- 纯人工 smoke 命令演示

这样划分的原因：

- 前者属于公共契约和稳定逻辑，后续 `#142`、`#143`、`#145`、`#154` 都会依赖
- 后者更偏工程包装或展示层细节，不值得在 FR-0001 把测试体系做重

## 并行 / 串行关系

前绪阻塞：

- FR-0001 的 spec review 通过，是实现 `#141` 的前绪阻塞
- 在 CLI 外层契约和最小 argv 语法未冻结前，不应开始任何会固化命令输入输出格式的实现代码

实现阶段串行关系：

- 阶段 A → B → C → D → E 为主串行链路
- 阶段 B 完成前，不应让后续事项各自定义 argv 语法、错误壳、退出码或 `run_id`

可并行事项：

- `#142`、`#143`、`#145`、`#154` 可以继续进行各自的 spec 起草 / 修订
- `#154` 可在 spec 层并行对齐错误分类与诊断字段，但不得改写 FR-0001 的外层错误壳
- `#144` 可独立推进其自身规约，但如要记录运行标识，应复用 FR-0001 的 `run_id` 语义

FR-0001 契约冻结后的并行实现边界：

- `#142` 可以在既有路由上挂接 Native Messaging / runtime 相关命令
- `#143` 可以在既有上下文上挂接 `profile` 相关命令与状态检查
- `#145` 可以在 `<platform>.<verb>` 命名空间下挂接首个平台读命令
- `#154` 可以在既有错误壳下补强诊断与可观测性

## 进入实现前条件

只有在以下条件满足后，才允许进入 `feat/FR-0001-*` 的实现工作：

- 当前 spec-only 变更已经进入 Draft PR 或等价评审上下文
- spec review 结论为 `APPROVE`
- `ready_for_implementation = true`
- `contracts/cli-entry.md` 与 `risks.md` 已随套件一起评审
- `TODO.md` 中的 spec review 阶段阻断项已清空

在这些条件满足前，明确禁止：

- 编写 FR-0001 的运行时代码
- 在 `#142`、`#143`、`#145` 中硬编码新的 CLI 外层契约
- 把 spec 修订和实现代码混在同一个 PR
