# FR-0018 风险与回滚

## 主要风险

### 1. “已验证”被误解为“可交付”

- 风险：最小健康状态被直接解读成能力已经可以导入、分享或长期交付。
- 缓解：
  - 明确 `health_state` 只用于当前能力可信判断
  - 明确导入/交付属于后续 Phase 3/5

### 2. 重放被误写成自动修复

- 风险：实现阶段把 replay 当作自动调参与重新学习入口。
- 缓解：
  - `ability_replay_request` 只冻结最小再运行入口

### 3. 建立第二套运行真相源

- 风险：验证结果另建一套与 `run_id` / artifact 无关的运行记录。
- 缓解：
  - 强制引用 `run_id`
  - `artifact_refs` 只作为补充 evidence refs，不提前假定上游 artifact carrier 已冻结

### 4. 失败大类过细或过粗

- 风险：过细会退化成错误码镜像，过粗又失去用户价值。
- 缓解：
  - 冻结有限的大类集合，并把细粒度错误继续留给下层诊断

## 回滚原则

- 若 reviewer 认为健康状态集合过宽，应优先缩减状态集合，不要并行引入第二套状态模型。
- 若 reviewer 认为某些失败大类证据不足，应先回退到更稳定的大类，而不是扩写更多半成品字段。
