# FR-0018 实施计划

## 实施目标

为 FR-0018 owning Work Item 冻结能力最小验证、最小重放与最小可信判断的正式输入，使后续实现 PR 可以围绕统一的验证请求、重放请求、最近一次验证结果和最小健康状态落地。

## 分阶段拆分

### 阶段 A：定位与继承边界冻结

- 产出：
  - `spec.md` 中与 `FR-0017`、`FR-0004`、`FR-0006` 的继承边界
- 重点：
  - 明确本 FR 不承担导入、版本治理或自动修复

### 阶段 B：稳定对象与数据模型冻结

- 产出：
  - `contracts/ability-validation.md`
  - `data-model.md`
- 重点：
  - 冻结验证请求、重放请求与健康视图
  - 冻结最近一次验证结果与失败大类

### 阶段 C：下游 handoff 与验证入口冻结

- 产出：
  - `TODO.md` 中进入实现前条件
  - 与 `FR-0019` 和后续交付线的边界说明
- 重点：
  - 明确 L2 样本和 L3 样本如何复用同一验证模型

## 实现约束

1. 本 FR 只提交规约文档，不提交运行时代码。
2. 不重定义 `FR-0017` 的候选能力描述对象。
3. 不把验证/重放对象扩写成导入、版本治理或分享协议。
4. 不把重放写成自动修复、重新学习或自愈流程。
5. 不建立第二套运行真相源；验证结果必须继续引用既有运行证据。

## 测试与验证策略

- 规约阶段门禁：
  - `bash scripts/docs-guard.sh`
  - `bash scripts/spec-guard.sh`
  - `git diff --check`
- 审查重点：
  - 验证与候选能力描述是否仍保持分层
  - 最近一次验证结果与失败大类是否足够稳定且用户可读
  - 是否把“验证通过”误写成“可交付”
- 实现前验证要求：
  - 验证请求对象可直接映射为实现入口
  - 健康视图可直接映射为 CLI / 能力界面的最小状态显示

## TDD 范围

- 本 FR 处于规约阶段，不直接新增实现测试。
- 后续实现 PR 默认纳入 TDD 的模块：
  - 验证请求解析与调度
  - 重放请求构造
  - 健康状态计算
  - 失败大类映射
- 不强制 TDD 的部分：
  - 与真实站点相关的手工验证证据

## 并行 / 串行关系

### 串行前置

- `FR-0017` spec review 未通过前，不进入本 FR 的实现 PR。
- 本 FR spec review 未通过前，不进入能力验证 / 重放实现 PR。

### 可并行

- `FR-0019` 的 L2 首次可用规约工作可并行进行，但不得绕过本 FR 自造验证结果对象。
- `FR-0021` 的下载能力方向可并行讨论，但其验证/重放边界仍要消费本 FR。

### 串行后置

- 任一能力若要显示最近一次验证结果，必须先消费本 FR 冻结的对象。
- 后续导入/交付线若要消费健康状态，也必须先继承本 FR 的最小判断模型。

## 进入实现前条件

1. FR-0018 规约 PR 完成 spec review 且无阻断项。
2. `contracts/ability-validation.md` 中的稳定对象语义无歧义。
3. `data-model.md` 已明确验证结果与运行证据的引用关系。
4. reviewer 明确认可本 FR 没有把“验证通过”提前扩张成“可交付/可分享”语义。
5. 后续实现 issue / PR 已明确引用：
  - FR-0018 owning Work Item
  - `FR-0017` 作为上游能力描述前置
