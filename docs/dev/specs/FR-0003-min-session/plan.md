# FR-0003 实施计划

## 实施目标

这次实施只交付“浏览器启动 + Named Profile + 最小身份 / 会话模型”的正式契约，不交付完整账号体系。

完成后应形成以下可审交付物：

- 指定 `profile` 的浏览器启动契约
- Profile 级最小状态机
- Profile 级代理粘性绑定契约
- Profile 独占锁语义
- `__webenvoy_meta.json` 的最小持久化边界
- 面向后续 `#145`、`#146`、`#148` 的稳定承载面

本 FR 的实现目标不是“把账号做全”，而是让后续命令可以在同一 Named Profile 上稳定复用会话与状态。

## 分阶段拆分

### 阶段 A：契约冻结与边界对齐

产出：

- `spec.md` 的状态机、命令语义和异常边界冻结
- `contracts/` 中的 Profile / runtime 生命周期契约冻结
- `data-model.md` 中的 Profile 最小持久化边界冻结
- `risks.md` 中的并发、代理与持久化风险冻结

依赖：

- `#141` 的 CLI 契约已经冻结
- `#143` 的场景、状态和代理边界已经对齐

### 阶段 B：Profile 启动与锁模型

产出：

- Named Profile 启动流程
- Profile 独占锁模型
- 启动失败、锁冲突、陈旧锁回收的语义

依赖：

- 阶段 A 完成

### 阶段 C：最小状态流转与代理绑定

产出：

- `uninitialized`、`starting`、`ready`、`logging_in`、`disconnected`、`stopping`、`stopped` 的状态流转
- `proxyUrl` 的首次绑定与粘性复用语义
- `runtime.status` 需要暴露的最小状态字段

依赖：

- 阶段 B 完成

### 阶段 D：最小持久化边界与恢复输入语义

产出：

- `__webenvoy_meta.json` 的写入、读取与恢复输入语义
- 首次登录从 `uninitialized` 直接进入 `runtime.login` 的最小语义
- 断连后 Profile 状态回读
- 手动登录结果持久化回 Profile 元数据
- `localStorageSnapshots` 的最小摘要回读语义（不含自动回写浏览器会话）

依赖：

- 阶段 C 完成

### 阶段 E：测试、验证与交接

产出：

- 状态机测试
- 锁测试
- 代理绑定测试
- 元数据读写测试
- 启动 / 断连 / 恢复的最小验证证据

依赖：

- 阶段 D 完成

最小闭环：

- 到阶段 D 结束时，应能证明“可按 Profile 启动、可复用会话、可稳定识别状态、可拒绝并发、可维持代理绑定”。

## 实现约束

- 必须保持最小身份 / 会话边界，不得扩张到账号矩阵、养号或长期运营。
- 不新增独立会话数据库；持久化只允许落在浏览器 UserDataDir 与 `__webenvoy_meta.json`。
- 不在本 FR 中扩张 `localStorageSnapshots` 自动回写浏览器会话链路。
- 不引入代理池、代理轮转或后台调度器。
- 不把 Profile 状态机和后续平台任务状态机混成一套。
- 不修改 FR-0001 已冻结的 CLI 外层契约。
- 不把浏览器启动逻辑写成单点脚本，必须保留明确的生命周期边界。

## 测试与验证策略

单元测试：

- Profile 状态机转移
- Profile 名称与路径校验
- `__webenvoy_meta.json` 序列化 / 反序列化
- 代理绑定与冲突判定
- 锁获取 / 释放 / 冲突 / 陈旧锁判定

集成 / 契约测试：

- 指定 `profile` 启动浏览器的完整链路
- `runtime.status` 能读回状态与代理绑定
- `runtime.login` 首次登录流程可以从未初始化 Profile 直接创建并回写最小元数据
- `runtime.login` 的手动登录确认流程
- `localStorageSnapshots` 写入元数据与摘要回读
- 并发启动同一 Profile 的冲突断言
- 浏览器崩溃后状态回落到 `disconnected`

人工验证：

- 本地 smoke 启动一个 Named Profile
- 手动完成一次登录并重启验证复用
- 修改代理参数并验证冲突拒绝
- 人工确认断连后的状态可回读

完成证据至少包括：

- Profile 生命周期测试通过
- 锁与代理绑定测试通过
- `docs-guard` 通过
- 本 FR 的 spec review 通过

## TDD 范围

默认先写测试的模块：

- Profile 状态机
- Profile 锁
- 元数据读写
- 首次登录状态流转
- 代理绑定与冲突判定
- 状态查询与错误分流

暂不强制先写测试的部分：

- 浏览器 UI 的人工登录步骤
- 真实站点上的手工验证流程
- 浏览器厂商差异导致的少量安装细节

这样划分的原因：

- 状态机、锁和元数据属于共享契约，后续 `#145`、`#146`、`#148` 都会依赖
- UI 手工步骤更适合后置 smoke，不值得在规约阶段强行 TDD 化

## 并行 / 串行关系

前绪阻塞：

- `#141` 的 CLI 契约必须先稳定
- Profile 状态机与代理绑定语义必须先在 spec 层冻结

可并行事项：

- `#142` 的通信 / 启动通道实现可以与本 FR 的规约并行推进
- `#144` 的基础运行记录可并行准备，但不能自行扩张成独立账号仓库
- `#145`、`#146`、`#148` 可以在本 FR 规约冻结后对接承载面

实现阶段串行关系：

- 阶段 A → B → C → D → E 为主串行链路
- 锁模型必须早于并发启动验证
- 代理绑定必须早于重启复用验证

## 进入实现前条件

只有在以下条件满足后，才允许进入 `feat/FR-0003-*` 的实现工作：

- 当前 spec-only 变更已经进入 Draft PR 或等价评审上下文
- spec review 结论为 `APPROVE`
- `ready_for_implementation = true`
- `contracts/`、`data-model.md`、`risks.md` 已随套件一起评审
- `TODO.md` 中的 spec review 阶段阻断项已清空

在这些条件满足前，明确禁止：

- 编写 FR-0003 的运行时代码
- 在 `#145`、`#146`、`#148` 中直接硬编码新的会话状态语义
- 把 Profile 持久化、代理绑定和并发锁拆成多个互不一致的实现口径
