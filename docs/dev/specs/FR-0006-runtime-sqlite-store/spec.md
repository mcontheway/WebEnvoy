# FR-0006 SQLite 最小持久化与运行记录基座

## 背景

Phase 1 需要先证明 CLI 到浏览器内执行链路可用，但如果运行信息只停留在进程内 stdout，就无法稳定回答三个问题：

1. 某次命令到底是否真正进入了浏览器内执行路径。
2. 失败时是参数问题、会话问题、通信问题还是页面问题。
3. 后续 `#154` 的结构化诊断与 `#159` 的能力壳如何复用一致的运行标识。

`#144` 的目标不是把完整业务数据模型一次性落地，而是建立一个最小、可审计、可复用的 SQLite 运行记录基座，作为 Phase 1 的证据层与调试层。

## 目标

1. 定义并冻结 Phase 1 所需的最小 SQLite 运行记录模型，用于记录命令运行事实与关键事件摘要。
2. 保证运行记录与 CLI 运行标识（`run_id`、`session_id`、命令名、时间窗口）可稳定关联。
3. 为 `#154` 的诊断输出提供可落库的最小结构，避免诊断只存在于瞬时响应。
4. 为 `#159` 的最小能力封装骨架提供统一的运行证据输入，不提前扩张到完整能力仓库。
5. 明确 SQLite 在 Phase 1 只承担“证据与审计记录”职责，不抢占 `#143` 的实时会话真相源角色。

## 非目标

- 不建立完整业务采集数据仓库，不承载平台详情、全文内容或媒体二进制。
- 不在本 FR 内定义完整迁移系统、复杂分区策略或长期归档系统。
- 不把 SQLite 作为运行时状态机的主真相源；实时状态仍由 `#143` 的 profile/session 路径承载。
- 不在本 FR 内扩张到能力市场、能力版本管理或多租户权限体系。
- 不改变 FR-0001 已冻结的 CLI 外层响应壳与退出码语义。

## 功能需求

### 1. 最小表结构

系统必须至少定义以下两类运行记录表：

- 运行主记录：描述一次命令运行的主事实（命令、profile、run_id、session_id、开始/结束时间、最终状态）。
- 运行事件记录：描述运行窗口内的关键阶段事件（阶段、组件、事件类型、摘要、结构化错误片段）。

表结构要求：

- 每条事件必须能关联到唯一运行主记录。
- 运行主记录必须支持按 `run_id` 唯一定位。
- 事件表必须支持按 `run_id + event_time` 读取时间线。

### 2. 写入边界与责任划分

- CLI 进程是 SQLite 写入入口，扩展侧与页面侧不直接写库。
- 写入仅接收“最小摘要”，不落完整请求体、完整响应体、Cookie、Token 或敏感头。
- 成功与失败都必须有可落库主记录，避免只记录失败导致统计偏差。

### 3. 与 `#143` 的边界

- SQLite 中可记录会话相关引用字段（如 `profile_name`、`session_id`），但不能替代 profile 元数据状态机。
- `runtime.status` 的实时判断仍由 `#143` 的 profile/session 路径给出，SQLite 只作为历史证据查询源。
- 不允许通过回放 SQLite 记录去直接驱动会话状态切换。

### 4. 与 `#154` 的边界

- 运行事件记录必须预留最小诊断字段（分类、失败位置、责任组件、摘要证据）。
- `#154` 输出的诊断结构要能映射到运行事件表，而不需要二次重建模型。
- 当诊断字段超出消息预算时，必须允许截断并标记，不得阻塞主记录写入。

### 5. 与 `#159` 的边界

- 运行主记录必须暴露能力层可复用的最小关联键（`run_id`、`command`、`profile_name`、`status`、`ended_at`）。
- 这套记录只提供“能力运行证据”，不定义能力目录、能力版本或能力发布语义。

### 6. 并发与一致性

- SQLite 必须运行在 WAL 模式。
- 同一 `run_id` 的主记录写入必须幂等，重复写入应更新有限字段而不是产生重复主记录。
- 事件写入允许追加，但必须保证事件关联的主记录存在。

## GWT 验收场景

### 场景 1：命令成功时写入完整主记录与最小事件
Given 一次 `runtime.ping` 命令成功完成  
When CLI 收敛输出并结束运行  
Then SQLite 中存在该 `run_id` 的唯一主记录  
And 至少有启动与完成两个关键事件  
And 主记录字段可用于后续按 `run_id` 查询

### 场景 2：命令失败时仍保留可诊断记录
Given 一次命令在执行链路中失败  
When CLI 返回结构化错误  
Then SQLite 中仍存在该 `run_id` 的主记录且状态为失败  
And 事件表中包含至少一条失败事件  
And 失败事件包含最小诊断字段而非空壳

### 场景 3：会话状态变化不依赖 SQLite 回放
Given 某 Profile 的运行状态由 `#143` 路径判定为 `disconnected`  
When 查询最近一次运行记录  
Then 查询结果只返回历史运行事实  
And 系统不会把 SQLite 记录当作实时状态真相源回写会话状态

### 场景 4：诊断摘要超长时可截断且不阻塞写入
Given 一次失败产生了超长诊断证据文本  
When CLI 尝试落库运行事件  
Then 事件记录中的证据字段按规则截断并标记  
And 主记录与关键失败事件仍可成功写入

### 场景 5：并发写入同一 run_id 不会产生重复主记录
Given 同一 `run_id` 触发了重复主记录写入请求  
When 写入层执行幂等更新  
Then 主记录表仍只保留一条该 `run_id` 记录  
And 事件表允许追加新事件但主记录不重复

## 异常与边界场景

- 数据库文件不存在时，系统应自动初始化最小 schema，不依赖人工建库。
- schema 版本不匹配时，必须返回结构化错误并阻断继续写入，不能静默跳过。
- 主记录写入成功但事件写入失败时，必须返回可定位错误并保留主记录，禁止整笔无声丢失。
- 敏感字段（Cookie、Authorization、Token、完整 URL query）必须在写库前脱敏或移除。
- 当数据库临时锁冲突出现时，写入层应按有限重试策略处理，超限后返回可重试错误。
- 运行记录查询接口只返回摘要，不返回潜在敏感原文载荷。

## 验收标准

1. `spec.md` 明确了 SQLite 在 Phase 1 的职责边界：记录证据，不承载实时会话真相。
2. 至少形成运行主记录与运行事件两类最小表，并具备稳定关联键。
3. 成功与失败路径都能落库，且失败路径包含最小诊断信息。
4. 文档明确了与 `#143`、`#154`、`#159` 的接口边界与依赖关系，避免职责重叠。
5. 写入策略满足最小并发一致性要求（WAL、幂等主记录、事件可追加）。
6. 套件未扩张到完整业务仓库、能力发布系统或上层调度系统。
