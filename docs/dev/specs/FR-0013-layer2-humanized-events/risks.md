# FR-0013 风险与回滚

## 风险 1：Layer 2 与 FR-0011 边界漂移

- 触发条件：实现或后续规约再次定义 `risk_state_machine`、`issue_action_matrix`、审批对象或审计对象。
- 影响：最小门禁与事件增强出现两套口径，后续实现无法判定以谁为准。
- 缓解：
  - 在 `spec.md`、`contracts/`、`data-model.md` 中显式声明继承关系。
  - 将“平行重定义 FR-0011 对象”设为 spec review 阻断项。
- 回滚：
  - 回到 FR-0011 作为唯一门禁真相源。
  - 删除 FR-0013 中越界对象后重新 review。

## 风险 2：Layer 2 膨胀成 Layer 3 session 行为引擎

- 触发条件：把长期 persona、跨页面节律、任务完成后闲逛等要求混入本 FR。
- 影响：Phase 2 边界失真，后续实现范围不可控。
- 缓解：
  - 将本 FR 的节奏范围限制为事件级窗口、滚动分段与最小长停顿。
  - 将完整 session 节律明确留给 Layer 3 独立事项。
- 回滚：
  - 移除所有 session 级要求，保留事件级最小边界。

## 风险 3：合成交互回退绕过最小风险门禁

- 触发条件：实现以“只是回退”为由，放行原本被 `FR-0011` 阻断的动作。
- 影响：高风险动作通过合成路径被偷偷执行，账号风险上升。
- 缓解：
  - 在契约中要求 Layer 2 选择发生在 `FR-0011` 门禁之后。
  - 要求策略选择结果显式回传阻断原因与回退原因。
- 回滚：
  - 禁用对应回退策略，恢复到真实交互或阻断。

## 风险 4：平台专用需求污染 Layer 2 通用边界

- 触发条件：把小红书完整发布细节、平台专属组件行为写进通用对象。
- 影响：通用 Layer 2 无法复用，FR 变成平台实现草稿。
- 缓解：
  - 套件中只冻结通用默认对象，并允许平台适配器覆盖。
  - 将平台覆盖值与完整发布闭环明确留到后续实现事项。
- 回滚：
  - 从通用规约中移出平台专用条目，转入平台实现 issue/PR。

## 风险 5：事件链定义过薄，后续实现仍靠猜

- 触发条件：只写“支持 focus/input/change/blur”，却不写不可缺失事件与状态收敛要求。
- 影响：实现和测试各写一套链路，回归不稳定。
- 缓解：
  - 在 `contracts/` 中冻结关键事件链对象与最小 trace 字段。
  - 在 `plan.md` 中要求事件链编排器纳入 TDD。
- 回滚：
  - 暂停实现 PR，先补齐事件链对象再继续。

## Stop-Ship 条件

- FR-0013 spec review 未通过就进入实现 PR。
- FR-0013 出现对 FR-0011 门禁/状态机/审批/审计对象的平行重定义。
- Layer 2 规约被扩写成 Layer 3 session 行为引擎。
- 合成交互被用来绕过 `FR-0011` 已阻断的动作等级。
- 平台完整发布闭环或 `#208` live 恢复被混入当前 FR。
