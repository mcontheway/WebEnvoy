# FR-0013 实施计划

## 实施目标

交付一套 implementation-ready 的正式规约输入，用于在 Phase 2 中实现 Layer 2 事件级拟人模拟增强。该输入要能被后续实现 PR 直接消费，覆盖：

- 真实交互优先与合成交互回退策略
- 事件链一致性要求
- 最小拟人化节奏与事件组合
- 读路径 / 写路径的接入边界
- 与 `FR-0011` 的继承关系和禁止重定义边界

## 分阶段拆分

### 阶段 A：定位与边界冻结

- 产出：`spec.md` 中的 Phase 2 定位、`#233` 挂载关系、与 `FR-0011` 的边界条款。
- 依赖：`vision.md`、`roadmap.md`、`anti-detection.md`、`FR-0011` 已存在。

### 阶段 B：稳定对象与调用链冻结

- 产出：`contracts/` 与 `data-model.md` 中的稳定对象、继承边界、调用顺序。
- 依赖：阶段 A 已明确本 FR 只做 Layer 2 事件增强，不改写门禁与状态机。

### 阶段 C：风险与实施前置冻结

- 产出：`risks.md`、`TODO.md`、进入实现前条件。
- 依赖：阶段 B 已明确哪些对象属于正式输入，哪些属于未来实现细节。

## 实现约束

1. 本 FR 只提交规约文档，不提交运行时代码。
2. 不重定义 `FR-0011` 已冻结的 `risk_state_machine`、`issue_action_matrix`、`write_interaction_tier`、`session_rhythm_policy`、`approval_record`、`audit_record`。
3. 不把 Layer 2 的事件增强扩写成 Layer 3 session 行为引擎。
4. 不把平台长期画像、Layer 4 或小红书完整发布闭环写入本 FR。
5. 不以“回退到合成事件”为理由绕过 `FR-0011` 的风险边界或动作等级阻断。
6. 不新增 CLI 外层契约，不并行引入第二套读写命令模型。
7. 不把 `#208`、`#227` 或任何实现 PR 的动作混入当前规约链路。

## 测试与验证策略

- 规约阶段门禁：
  - `bash scripts/docs-guard.sh`
  - `bash scripts/spec-guard.sh`
  - `git diff --check`
- 审查验证：
  - 对照 `spec_review.md` 检查是否满足高风险 FR 套件深度
  - 对照 `FR-0011` 检查是否只做继承和增量，不做平行重定义
  - 对照 `docs/dev/roadmap.md` 检查是否仍属于 Phase 2，而未提前混入 Layer 3/4
- 实现前验证要求：
  - 事件策略对象能直接映射为实现配置与测试夹具
  - 事件链对象能直接映射为单元测试用例
  - 调用链顺序能直接映射为集成测试断言

## TDD 范围

- 本 FR 本身处于规约阶段，不直接新增实现测试。
- 后续实现默认纳入 TDD 的模块：
  - 事件策略选择器
  - 事件链编排器
  - 焦点建立与状态收敛等待
  - 滚动/输入节奏生成器
  - 失败分类与回退原因映射
- 不强制 TDD 的部分：
  - 真实站点上的人工验证
  - 仅用于研究性参数校准的临时实验

## 并行 / 串行关系

### 串行前置

- `FR-0011` 已合并并作为最小前置存在。
- `FR-0013` spec review 未通过前，不进入 Layer 2 事件增强实现 PR。

### 可并行

- 与 `#233` 下其他延续项的规约工作可并行，例如 Layer 1 指纹补全、Layer 3 完整 session 节律引擎的独立立项。
- 与平台适配器或能力封装范围设计可并行讨论，但不得抢跑实现。

### 串行后置

- `#208` 的后续恢复判断如需消费 Layer 2 能力，必须先引用 FR-0013 的正式结论。
- 任何需要平台覆盖值的实现工作，必须以本 FR 的默认对象语义为前提。

## 进入实现前条件

1. FR-0013 规约 PR 完成 spec review 且无阻断项。
2. `contracts/layer2-humanized-events.md` 中的稳定对象语义无歧义，且 reviewer 明确认可被实现直接消费。
3. `data-model.md` 中的共享实体已明确哪些字段继承 `FR-0011`，哪些为 FR-0013 新增对象。
4. `risks.md` 中“回退绕过门禁”“Layer 2 膨胀成 Layer 3”“平台专用需求污染通用边界”等风险已有 stop-ship 与回滚策略。
5. 后续实现 issue / PR 明确使用 `Refs #236`，且不把 `#236` 误写为 `Fixes`。
