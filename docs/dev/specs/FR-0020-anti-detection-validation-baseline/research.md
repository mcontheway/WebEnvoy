## 结论

1. 当前仓库已经把反风控“做什么”写清了，但还没把“如何验证是否成立”冻结成正式对象。
2. `#239` 的真实角色更接近正式 FR，而不是长期保留的横切主线 issue。
3. `FR-0015` 只消费验证结果，不应拥有验证框架本体。
4. `FR-0016` 只定义 PR 级门禁，不应替代能力级 baseline 评估对象。

## 进入正式 FR 的原因

- 该事项会被多个 FR 共享消费。
- 该事项涉及稳定对象和长期诊断口径。
- 若继续停留在 umbrella / issue 层，后续实现很容易把探针、baseline、gate、run log 混写。
