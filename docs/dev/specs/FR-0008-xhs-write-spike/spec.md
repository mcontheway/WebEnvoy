# FR-0008 小红书 Spike A-Write：最小写链路与上传路径确认

## 背景

`#146` 的职责是为小红书写链路建立正式侦察输入，而不是直接承担 Phase 1 的阶段出站责任。

当前仓库已经把“Phase 1 至少一种最小页面交互动作正式验证”单独拆成 `#208`。因此，本 FR 只负责冻结以下问题的证据边界：

- 富文本编辑器的 DOM 结构与事件响应机制
- 图片上传路径与关键节点
- 最小写链路的前端交互路径与潜在降级路径
- 哪些侦察结论可以作为 `#208` 的候选输入，哪些仍只是研究背景

没有这层侦察输入，后续要么把完整写链路复杂度误混进 Phase 1 blocker，要么把一次性页面实验错误升级成正式验证结论。

## 目标

1. 冻结小红书最小写链路的证据矩阵，至少覆盖富文本输入、图片上传、提交前后关键反馈节点。
2. 明确写链路中的 `page primary + api fallback/candidate` 证据边界，避免把上传降级路径误写成默认实现承诺。
3. 为 `#208` 输出“最小页面交互动作候选集”，但不直接宣告其中任一动作已完成 Phase 1 正式验证。
4. 形成可复用的结构化侦察输出，供后续小红书写能力 FR 和 `#208` 直接引用。
5. 与 FR-0001/0002/0003/0004/0007 既有运行时与能力壳边界对齐，不引入第二套 CLI 或通信契约。

## 非目标

- 不实现完整小红书发布闭环。
- 不在本 FR 内交付 `publish`、`interact` 或媒体上传实现代码。
- 不把 `#146` 直接当作 `#208` 的关闭依据。
- 不把图片上传、富文本输入、多步骤发布确认全部膨胀成同一个 Phase 1 blocker。
- 不新增或修改持久化 schema。
- 不引入仓库外不可复核引用作为冻结结论依据。

## 功能需求

### 1. 写链路场景矩阵冻结

- 必须至少覆盖三个写链路场景：
  - 富文本编辑器输入
  - 图片上传
  - 最终提交前的最小页面交互动作候选
- 每个场景都必须显式标注：
  - `route_role`：`primary` 或 `fallback`
  - `path_kind`：`page` 或 `api`
  - `evidence_status`：`success`、`failed`、`candidate`
  - `evidence_maturity`：`observed_once`、`reproduced_multi_round`、`admission_ready`
- 只有达到后续实现前置条件的 `primary` 证据，才允许升级为 `admission_ready`。

### 2. 富文本编辑器证据边界

- 必须记录编辑器入口定位方式、DOM 结构特征、焦点获取路径、最小事件链、成功/失败信号。
- 必须区分：
  - 普通点击/输入可否用真实物理输入完成
  - 何时必须回退到 `CompositionEvent` / `InputEvent` 链
- 必须记录 `isTrusted`、React/Vue 受控更新、空值提交等潜在失败症状。
- 若当前只有单轮页面实验，只能标为 `observed_once` 或 `candidate`，不得越级写成“正式验证已通过”。

### 3. 图片上传路径证据边界

- 必须记录图片上传入口属于以下哪类：
  - 标准 `<input type=\"file\">`
  - 拖拽上传区域
  - 独立上传 API / OSS 预签名路径
- 必须冻结最小关键节点：
  - 文件注入点或拖拽落点
  - 前端触发上传的监听事件
  - 上传开始、上传完成、上传失败的页面信号
- 若存在 `page primary` 失败样本，必须保留对应 `api fallback` 或 `candidate` 路径，不得静默删除。

### 4. 给 #208 的 handoff 边界

- Spike 输出必须显式包含“可供 `#208` 选择的最小页面交互动作候选集”。
- `#208` 正式验证收口口径固定落在 `docs/dev/issue-208-min-page-interaction-validation-note.md`；本 FR 只提供候选输入，不替代该说明文档的关闭标准。
- 每个候选动作必须写清：
  - 运行前置
  - 执行动作
  - 成功信号
  - 失败信号
  - 最小复现实验
  - 当前是否已达到 `admission_ready`
- 该 handoff 仅代表“可供 `#208` 评估的输入”，不代表 `#208` blocker 已完成。

### 5. 双门禁结论

- 输出必须分别给出：
  - `fallback viability`
  - `implementation readiness`
- 不允许用 fallback 证据替代实现准入结论。
- 不允许把 `#146` 的 Spike 结论直接等同于 Phase 1 exit review 结论。

### 6. 与既有运行时契约对齐

- 侦察执行依赖 FR-0002/0003 的浏览器内链路与会话边界。
- 结果表达应与 FR-0004/0007 的观察与能力壳方向兼容。
- 不定义与 FR-0001 CLI 外层契约冲突的新输入输出壳。

## GWT 验收场景

### 场景 1：富文本编辑器交互被结构化冻结

Given Native Messaging、最小会话链路和观测壳已可用于页面内侦察
When 对小红书发布页或等价写入页面执行富文本编辑器侦察
Then 输出中能明确记录编辑器入口、最小事件链、成功信号与失败信号
And 能区分真实物理输入与合成事件链各自的适用边界
And 未达到多轮复现的证据不会被写成正式验证通过

### 场景 2：图片上传路径被拆成主路径与降级路径

Given 页面存在图片上传入口或上传触发链路
When 对页面上传流程执行侦察
Then 输出中能明确当前上传路径属于 `page primary`、`api fallback` 或 `candidate`
And 至少冻结一个文件注入点、一个前端触发事件和一组页面成功/失败信号
And 失败样本不会被成功结论覆盖

### 场景 3：#208 能直接消费最小交互动作候选

Given `#146` 的 Spike 输出已经完成
When 评审者检查 handoff 结构
Then 能直接识别一组供 `#208` 选择的最小页面交互动作候选
And 每个候选都包含前置、动作、成功信号、失败信号和最小复现实验
And 文档会明确说明这些候选输入不等于 `#208` 已关闭

### 场景 4：双门禁结论不被混写

Given 写链路侦察存在页面路径、API 降级路径和高风险写入边界
When 评审者检查 Spike 结论
Then 文档中能分别看到 fallback viability 与 implementation readiness
And fallback-only 证据不会被升级为实现准入
And `#146` 不会被描述为 Phase 1 正式出站项

## 异常与边界场景

### 1. 账号与风控异常

- 登录态失效、验证码、账号异常、上传限流必须单独归档，不得混写为普通交互失败。
- 若写链路侦察可能扩大账号风险，必须优先停在证据冻结，而不是继续放大实验。

### 2. 页面结构漂移

- 编辑器容器、上传入口、预览区、发布按钮名称或层级变化时，必须记录为结构漂移，不得静默套用旧路径。
- 若同类页面在不同入口出现分流，必须标注分流条件。

### 3. 空与极值

- 空正文、超长正文、空图片集、超量图片、异常文件类型、超大文件都必须有边界说明。
- 仅在单一 happy path 观测成功时，不得宣称写链路已可实施。

### 4. 运行时链路异常

- content-script 注入失败、会话断连、页面刷新导致节点丢失时，必须归类为运行时链路问题，而不是平台页面结论。
- 上传过程中若出现异步进度未收敛，必须记录等待策略缺口。

## 验收标准

1. 写链路至少形成富文本输入、图片上传、最小交互动作候选三类结构化结论。
2. 每类结论都显式标注 `route_role`、`path_kind`、`evidence_status`、`evidence_maturity`。
3. 输出中显式包含给 `#208` 的 handoff 候选集，并明确“输入不等于阶段出口”。
4. 文档分别给出 fallback viability 与 implementation readiness，不混写。
5. 风险、异常、失败样本与降级路径均被保留，未被成功样本覆盖。
6. 文档不引入与 FR-0001/0002/0003/0004/0007 冲突的契约，也不把本 FR 误写成实现 PR。
