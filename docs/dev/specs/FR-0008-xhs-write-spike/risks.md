# FR-0008 风险与回滚

## 风险 1：把写链路 Spike 误当 Phase 1 正式出口

- 触发条件：评审或后续 issue 直接把 `#146` 结论写成“最小交互动作已正式验证”
- 影响：`#208` 被绕过，Phase 1 blocker 边界失真
- 缓解：
  - 在 `spec.md`、`plan.md`、`contracts/` 中反复固定 `#146 -> #208` 的 handoff 关系
  - PR closing 语义使用 `Refs #146`，不使用 `Fixes #146`
- 回滚/降级：
  - 停止以 `#146` 作为阶段出口引用
  - 回到 `#208` 单独完成正式验证

## 风险 2：live 侦察放大账号与风控风险

- 触发条件：高频尝试输入、上传、提交，或在异常状态下继续试验
- 影响：账号异常、验证码、环境噪声污染侦察结论
- 缓解：
  - 把不可逆提交与高频上传视为高风险动作
  - 优先冻结证据边界，不在风险升高时继续放大实验
- 回滚/降级：
  - 立即停止当前会话侦察
  - 将当前结果降级为 `candidate` 或研究背景

## 风险 3：页面主路径与 API 降级路径被混写

- 触发条件：上传失败样本被静默删掉，或将 API fallback 写成默认主路径
- 影响：后续实现 FR 走错路径，范围膨胀
- 缓解：
  - 强制记录 `route_role`、`path_kind`、`evidence_status`
  - 双门禁中单独记录 fallback viability
- 回滚/降级：
  - 将当前结论回退为 `fallback-only` 或 `candidate`
  - 等待补充主路径复核后再进入实现

## 风险 4：只凭仓库基线就推进实现

- 触发条件：把架构文档和历史研究直接当成浏览器内第一手证据
- 影响：实现与真实页面行为脱节
- 缓解：
  - `research.md` 显式区分仓库基线和未来 live 证据
  - 在 TODO 中列出必须补齐的第一手复核项
- 回滚/降级：
  - 停止实现分支推进
  - 继续完成 Spike 复核，再单独起实现 FR
