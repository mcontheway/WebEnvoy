# Sprint 1 项目管理机制复盘（WebEnvoy）

## 1. 文档目的与范围

本文只复盘 Sprint 1 的**项目管理机制**，不复盘功能实现细节。目标是把已验证有效的管理做法沉淀为后续 Sprint 可直接执行的操作参考。

范围覆盖：

- 规划与立项机制
- Spec 评审与实现准入机制
- PR/Review/合并门禁机制
- 多线程执行与交接机制
- 证据管理与状态收敛机制

## 2. Sprint 1 做对了什么（可复用）

## 2.1 单向研发漏斗被执行到了位

从 `roadmap -> GitHub backlog -> FR 套件 -> spec review -> 实施 PR -> merge` 的单向漏斗在 Sprint 1 实际运行并完成收口，避免了“边做边改边补契约”的失控路径。

可复用价值：

- 范围控制更稳定，减少了未定义需求直接进入实现
- “先定契约、再承诺实现”的节奏被团队执行出来

## 2.2 “规约 PR 与实现 PR 分离”有效降低了返工成本

FR 规约调整（如 FR-0003、FR-0005）通过独立文档线先收口，再推进实现或后续动作，避免在实现代码中混入契约漂移。

可复用价值：

- 评审焦点清晰，不会把“是否该做”与“怎么实现”混在一次审查中
- 返工集中在文档层时，修正成本显著低于实现层回滚

## 2.3 多重门禁组合形成了“可放行”共识面

Sprint 1 并未把“checks 全绿”当作唯一放行条件，而是叠加了 `spec-review` 与本地 `pr-guardian` 的边界一致性判断，形成更稳的合并决策。

可复用价值：

- 能识别“CI 通过但边界仍冲突”的假通过状态
- 高风险文档与契约变更获得了更严格的准入控制

## 2.4 长链路任务采用 handoff 机制，降低了上下文丢失

Sprint 1 使用持续更新的 handoff 记录停点、阻塞、下一步、判定依据，使跨会话与多 worker 接力可继续推进而非重复探索。

可复用价值：

- 复杂事项不会因会话切换而回退到“重新读一遍”
- 决策路径可追溯，减少“为什么这么做”的认知断层

## 2.5 路线先裁决再改文档，避免无效迭代

在 FR-0005 后段，先做只读决策分析（路线 A/B），再决定是否改 FR 文档或先改 architecture 基线，避免了“在错误上位约束下反复修文案”。

可复用价值：

- 先解决上层边界冲突，再处理下层文档细节
- 降低“同一 PR 多轮小修仍不放行”的空转

## 3. Sprint 1 主要教训（需规避）

## 3.1 容易把“证据成立”误写成“实现准入”

Spike 阶段里，`candidate`、`fallback`、`admission_ready` 的语义边界一度混用，导致 review 反复阻断。

教训：

- 必须在契约里提前定义证据状态模型
- 证据可观测不等于实现可开工

## 3.2 架构基线与 FR 套件不一致会导致反复打回

当 architecture 文档与 FR 规约口径不一致时，FR 文档层面的修补会被持续上位阻断。

教训：

- 先判“冲突层级”，再决定改哪一层
- 上位基线未收口前，避免在下位文档高频迭代

## 3.3 PR 元数据（如 `Fixes #...`）会直接影响项目节奏

Spike 阶段错误使用 `Fixes #...` 可能提前关闭 issue，破坏真实状态流转。

教训：

- issue 关闭语义应与阶段语义绑定
- PR 描述属于项目管理资产，不是可忽略文本

## 3.4 本地环境噪音未形成固定清理节奏

Sprint 1 收尾时存在历史 worktree/临时目录噪音，虽不阻断交付，但增加后续操作风险和认知负担。

教训：

- 需要把“环境清理”纳入 Sprint DoD 收尾清单

## 4. 建议固化的机制（已作为本轮治理落地方向）

## 4.1 机制 A：三层边界判定优先

每次出现 review 反复打回，先做 10 分钟边界定位：

- 第 1 层：`vision/roadmap` 阶段边界是否冲突
- 第 2 层：`architecture/spec_review` 准入边界是否冲突
- 第 3 层：FR 套件内部表达是否冲突

执行规则：

- 未通过第 1/2 层，不进入第 3 层细修
- 先修上层，再修下层

## 4.2 机制 B：证据状态模型前置冻结

凡是 Spike/侦察类 FR，在 `spec.md + contracts/*` 中先冻结证据状态字段与判定语义，再允许写研究结论。

最小状态建议：

- `route_role`: `primary | fallback`
- `path_kind`: `api | page`
- `evidence_status`: `success | failed | candidate | blocked`
- `admission_ready`: `true | false`（必须有严格条件）

## 4.3 机制 C：规约线与实现线硬分离

继续保持“规约 PR 默认独立、实现 PR 默认独立”，仅允许 `TODO.md` 进度回写与实现同 PR（且不改正式契约语义）。

执行规则：

- 规约未通过 review，不得承诺实现 Sprint
- 发现契约漂移，优先回到规约线处理

## 4.4 机制 D：PR 放行采用“双门禁 + 一语义”

合并前同时满足：

- 门禁 1：GitHub checks 全绿
- 门禁 2：本地 review/guardian 无阻断结论
- 语义：Issue 状态语义与 PR 文案语义一致（`Fixes/refs` 使用正确）

## 4.5 机制 E：状态型 runtime 必须先冻结健康矩阵

凡是涉及会话、锁、controller/browser 活性、断连、恢复、孤儿进程回收的事项，在进入实现前先冻结：

- 健康判定矩阵
- 恢复 / 回收路径
- 最小回归矩阵

## 4.6 机制 F：Handoff 成为长链路事项的强制工件

符合以下任一条件即强制维护 handoff：

- 多 worker 并行
- 预计跨会话
- 存在路线分叉待裁决
- 连续两轮以上 review 未放行

handoff 最小字段：

- 当前状态
- 最新阻断
- 下一步顺序
- 决策备选项与推荐路线
- 证据与结论链接

## 4.7 机制 G：Sprint 收尾加入“环境清理 DoD”

Sprint 结束前，除业务与文档收口外，增加环境治理清单：

- 临时 worktree/clone 清理
- 本地噪音文件排查
- 未使用分支整理
- 下 Sprint 启动入口确认

## 6. 后续 Sprint 操作指引（简版）

1. Sprint 立项周：先校验候选项是否满足 Roadmap 阶段边界，再建 FR 套件。
2. Spec 周期：先冻结证据状态模型，再写研究结论与验收场景。
3. 实施前：必须通过 spec review，且规约 PR 已独立收口。
4. 合并前：同时满足 checks、guardian、Issue 语义一致。
5. 状态型 runtime 改动：先检查健康矩阵与恢复测试是否冻结。
6. Sprint 收尾：完成回顾文档与环境清理 DoD。

## 7. 依据与证据来源

本复盘结论基于以下仓库内材料：

- `vision.md`：产品边界、CLI-first、阶段主线与非目标
- `docs/dev/roadmap.md`：Phase 约束、阶段门与“状态以 GitHub 为准”
- `docs/dev/AGENTS.md`：研发漏斗、规约/实现分离、spec 套件要求
- `spec_review.md`：正式准入标准、必查维度、阻断条件
- Sprint 1 本地 handoff 记录：用于恢复执行停点、阻断收口、合并判定与收尾状态；该类记录属于本地恢复介质，不是仓库正式真相源
