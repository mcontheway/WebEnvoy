# Phase 1 Exit Review

> 日期：2026-03-21
> 关联事项：#201

## 1. 目的

本评审用于回答三件事：

1. Phase 1 是否已经达到正式出站条件。
2. 哪些事项已经满足，哪些只是“骨架已在”但证据还不够。
3. Phase 2 在什么条件下才能进入正式实施承诺，而不是继续停留在规划与前置澄清。

本文是阶段治理记录，不是正式契约，不替代 `docs/dev/specs/` 下的 FR 套件。

## 2. 结论

当前结论：**Phase 1 主要承诺已完成，但暂不宣布正式出站；Phase 2 也暂不应按“已满足前置条件”进入实施承诺。**

原因不是 CLI、通信、会话、诊断或能力壳不存在，而是首个平台的正式闭环仍停留在 `Spike + 壳层接入`，尚未达到 Roadmap 对 Phase 1 的两项硬要求：

- 首个平台最短路径必须成为可重复运行的命令闭环，而不是一次性侦察样本。
- 至少一种最小页面交互动作必须完成验证。

因此，当前更准确的阶段判断应为：

- **Go for close-out**：Phase 1 的治理、运行时基座和能力壳骨架已足够收口。
- **No-Go for exit**：首个平台真实 L3 命令闭环与最小交互验证尚未完成，Phase 1 不应被表述为“正式出站”。
- **No-Go for Phase 2 implementation commitment**：`#155`、`#156`、`#157` 这类 Phase 2 主线项，不应在当前状态下被表述为“已满足阶段前置，可直接进入完整实施承诺”。

## 4. 为什么不能宣布 Phase 1 正式出站

### 4.1 FR-0005 仍是侦察输入，不是实现闭环

FR-0005 当前冻结的是“小红书读链路侦察证据包”，不是正式 L3 读适配实现。PR `#184` 已明确写明：

- 详情页与用户主页的稳定成功 API 样本仍未补齐。
- `required_headers` 的最小必要实验矩阵仍未完成。
- 当前状态仍不足以宣布“实现就绪”。

这意味着仓库已经拥有了“往哪里实现”的证据，但还没有拿到“已经实现完成”的阶段证据。

### 4.2 `xhs.search` 仍是能力壳占位，不是首个平台闭环

当前 `src/commands/xhs.ts` 已接入能力壳校验和 `summary.capability_result` 输出骨架，但实际执行路径仍直接返回：

- `ERR_EXECUTION_FAILED`
- `reason: XHS_SEARCH_SPIKE_PENDING`

这证明 FR-0007 交付的是统一能力壳，不是首个平台真实读闭环本身。能力壳已就位，但平台命令仍未落地。

### 4.3 最小交互验证尚未完成

Roadmap 明确要求 Phase 1 至少验证一种最小页面交互动作。当前 `#146` 仍处于打开状态，且其职责就是把小红书最小写链路/上传路径的侦察与验证从读链路中拆出单独管理。

即使把 `#146` 视为“后段验证项”，当前仓库里也还没有其他正式证据证明“至少一种最小页面交互动作已完成验证”。因此，这一项不能判定为已满足。

## 5. 哪些结果可以视为已经收口

以下内容已经可以视为 Sprint 1 和 Phase 1 基座层的正式收口成果：

- CLI 外层契约、结构化错误、运行标识与帮助入口已冻结。
- Native Messaging 与浏览器启动 / profile 会话模型已形成稳定运行时底座。
- 最小 observability 与 SQLite 运行记录基座已到位，后续不是从零开始补诊断。
- 最小能力壳已经形成，后续 L3/L2 不需要再发明第二套外层输入输出模型。
- Sprint 1 的治理机制已经通过 retrospective 文档收口，不存在“流程没有沉淀”的缺口。

也就是说，**Phase 1 失败的不是基座建设，而是首个平台闭环还没从 Spike 跨过实现准入线。**

## 6. 进入 Phase 2 前必须补齐的约束

### 6.1 先补齐 Phase 1 剩余闭环，再谈 Phase 2 实施承诺

进入 Phase 2 正式实施承诺前，至少需要先满足以下条件：

1. 把首个平台从 Spike 结论推进到真实可重复命令闭环。
2. 让 `xhs.search` 不再返回 `XHS_SEARCH_SPIKE_PENDING`，而是具备真实平台执行结果。
3. 补齐至少一种最小页面交互动作的正式验证证据。
4. 补齐首个平台真实商业压力下的最小复核证据，使“能跑通一次”升级为“可重复运行”。

### 6.2 Phase 2 自身的前置也不能跳过

即使补齐了上述 Phase 1 缺口，Phase 2 仍应遵守现有上位约束：

- `docs/dev/architecture/system-design/reference.md` 已把 Spike D、Spike E 标记为 Phase 2 前置。
- `#155`、`#156`、`#157` 当前仍处于打开状态，说明 Phase 2 主线能力本身尚未完成 FR 与实施闭环。
- 在这些前置未冻结前，不应把 Sprint 2 表述成“Phase 2 已正式开工并进入稳定实施”。

## 7. 后续动作建议

建议将后续动作按以下顺序收口：

1. 先为首个平台 L3 读闭环建立正式实现 FR，承接 FR-0005 的 Spike 输出，而不是继续把 Spike 文档当实现替身。
2. 单独完成最小页面交互动作验证，并将其作为 Phase 1 剩余 blocker 显式关闭。
3. 在 Phase 1 真正出站后，再为 `#155`、`#156`、`#157` 开启 Phase 2 的正式实施承诺与 Sprint 语义。

在此之前，仓库对外的阶段表述应保持为：

- Phase 1 基座已基本完成。
- Phase 1 尚未正式 exit。
- Phase 2 可继续做前置澄清与规约准备，但暂不按已满足阶段门进入完整实施承诺。
