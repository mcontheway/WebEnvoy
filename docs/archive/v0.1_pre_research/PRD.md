> 版本：v0.1  
> 日期：2026年03月12日  
> 状态：草稿

---

## 一、产品概述

**WebEnvoy** 是一款面向 AI Agent 的浏览器自动化工具。它让 AI 能够像人类一样操作浏览器——浏览网页、采集数据、点击按钮、填写表单、发送消息，同时尽可能避免被目标网站识别为自动化程序。

### 核心理念

- **脑手分离**：AI 负责思考决策和行为链编排，WebEnvoy 负责执行单个动作并使其尽可能拟真，两者通过命令行指令通信
- **渐进式披露**：不一次性把所有信息塞给 AI，而是按需提供，避免信息过载
- **分层能力**：对重点平台提供精准高效的专用工具，对其他网站提供通用兜底能力
- **被动观察**：数据采集以拦截浏览器自然产生的请求和缓存为主，尽量不主动发起额外请求
- **技术指纹归零**：在技术指纹层面做到与真实用户的浏览器环境无法区分（无自动化标识、无调试端口、真实浏览器指纹），行为层面的拟真由 WebEnvoy 提供原子级工具，行为链的自然性由 AI 编排负责
- **三层架构解耦**：明确区分通信层（Native Messaging/WebSocket等）、执行层（OS级/插件/CDP）和感知层（语义树/网络拦截），各层可独立替换和组合
- **安全与效率可调**：提供多种执行层策略（从极致安全到高效优先），由用户根据场景自行选择
- **不重复造轮子**：在通用页面感知能力上，优先集成成熟开源方案（如 PinchTab、agent-browser），将核心精力聚焦于高风控平台的写操作和数据流转

### 角色定义

| 角色 | 职责 |
|---|---|
| **AI Agent** | 理解用户意图、规划任务步骤、组合调用工具、处理返回结果 |
| **命令行工具 (CLI)** | 接收 AI 指令、管理浏览器进程、协调数据流转、控制信息披露粒度 |
| **浏览器插件** | 在网页内部感知页面内容、执行具体操作、拦截数据 |
| **本地数据库** | 持久化存储采集到的数据，支持查询、去重、统计 |
| **平台规则库** | 提供独立可热更的配置（YAML/JSON），定义各平台的网络截获规则、风控水位和数据映射逻辑 |

## 二、用户场景与验收标准

以下使用 Gherkin（Given-When-Then）格式描述各业务场景及其验收条件。

---

### 场景组 A：环境准备

#### A1：首次初始化工作环境

```gherkin
场景: 用户首次使用 WebEnvoy
  假如 用户已安装 WebEnvoy
  当 用户执行初始化命令
  那么 系统应在本地创建一个专属的工作目录
  并且 工作目录中应包含浏览器配置空间和数据库文件
  并且 系统应自动准备好内置的浏览器插件（无需用户手动安装）
  并且 用户应看到初始化成功的确认信息
```

#### A2：为特定平台完成登录

```gherkin
场景: 用户需要让 AI 操作需要登录的平台（如小红书）
  假如 工作环境已初始化
  当 用户执行平台登录命令并指定平台名称
  那么 系统应打开一个可见的浏览器窗口
  并且 自动导航到该平台的登录页面
  并且 用户应能手动完成登录（扫码、输入密码等）
  并且 登录完成后，用户确认即可关闭窗口
  并且 该平台的登录状态应被永久保存，后续使用无需重复登录
```

#### A2-b：启动时校验登录状态

```gherkin
场景: AI 启动浏览器后系统自动校验已保存的登录状态
  假如 用户之前已为某平台完成登录并保存了状态
  当 AI 以该配置空间启动浏览器
  那么 系统应自动检测已保存的登录状态是否仍然有效
  并且 如果登录有效，AI 应收到"已登录"的确认及账号标识信息
  并且 如果登录已失效（Cookie 过期/被踢出），AI 应收到"登录已失效，需要重新登录"的明确提示
  并且 登录失效时不应阻塞浏览器启动，AI 可选择继续使用或先重新登录
```

#### A3：多账号隔离

```gherkin
场景: 用户需要同时管理同一平台的多个账号
  假如 用户已为平台 A 完成账号 1 的登录
  当 用户以不同的配置名称再次执行登录命令
  那么 系统应创建一个完全独立的浏览器配置空间
  并且 新账号的登录状态与已有账号互不干扰
  并且 AI 在执行任务时可以指定使用哪个账号
```

#### A4：启动与关闭浏览器

```gherkin
场景: AI 需要开始执行浏览器任务
  假如 目标平台已完成登录
  当 AI 发出启动浏览器的命令并指定配置名称
  那么 系统应在后台启动浏览器并加载对应的配置和登录状态
  并且 内置插件应自动加载并与命令行工具建立连接
  并且 AI 应收到"浏览器已就绪"的确认

场景: AI 完成任务需要关闭浏览器
  假如 浏览器正在运行
  当 AI 发出关闭命令
  那么 浏览器进程应被完全清理，不留残余
```

#### A5：选择执行策略与通信通道

```gherkin
场景: 用户需要根据目标平台的风控强度选择底层技术方案
  假如 工作环境已初始化
  当 用户为某个配置空间设置底层策略
  那么 系统应支持选择以下执行层策略：
    - "安全模式"：使用操作系统级别的鼠标和键盘模拟，实现完全真实的交互事件，但要求浏览器窗口可见
    - "标准模式"（默认）：使用浏览器插件执行操作，平衡安全性与效率
    - "效率模式"：使用底层浏览器调试协议 (CDP) 执行操作，速度最快但存在被检测的风险
  并且 系统应支持选择以下通信层通道：
    - "Native Messaging"（默认）：最安全稳定，无需端口
    - "WebSocket"：开发调试友好
  并且 用户可随时切换这些策略组合
  并且 切换策略不影响已保存的登录状态和采集数据

场景: 首选通信通道建立失败时自动降级
  假如 用户配置了 Native Messaging 作为通信通道
  当 浏览器启动后插件尝试通过 Native Messaging 建立连接但失败（如 Native Host 未注册）
  那么 系统应自动尝试通过 WebSocket 建立连接作为降级方案
  并且 通知 AI 当前使用的是降级通道及降级原因
  并且 如果所有通道均失败，返回明确的错误信息和排查建议
```

#### A6：查看可用配置空间

```gherkin
场景: AI 需要知道当前有哪些可用的配置空间
  假如 工作环境已初始化，且用户已创建过至少一个配置空间
  当 AI 执行列出配置空间命令
  那么 系统应返回所有配置空间的列表
  并且 每个条目应包含：配置名称、关联平台、创建时间、最近使用时间
  并且 AI 可据此选择目标配置空间来启动浏览器
```

### 场景组 B：数据采集（读操作）

#### B1：在特定平台搜索内容（专用工具）

```gherkin
场景: AI 需要在小红书搜索笔记
  假如 小红书账号已登录且浏览器已启动
  当 AI 执行小红书搜索命令，指定关键词和数量限制
  那么 系统应返回搜索结果的精简摘要（条数、预览信息）
  并且 完整的结构化数据应自动存入本地数据库
  并且 摘要中应包含实际采集条数与成功入库条数的比对（如"采集 20 条，入库 18 条，去重 2 条"）
  并且 AI 可以通过后续查询命令按需获取详细字段
  并且 不需要再次访问网页即可反复查询已采集的数据

场景: 平台返回的搜索结果少于 AI 请求的数量
  假如 AI 请求搜索 50 条笔记
  当 平台实际只返回了 15 条结果（关键词冷门或已到达末尾）
  那么 系统应返回实际采集到的 15 条数据
  并且 摘要中应明确标注"请求 50 条，实际获取 15 条，已无更多结果"
  并且 不应为了凑数而持续无效翻页
```

#### B2：获取特定平台内容详情（专用工具）

```gherkin
场景: AI 需要获取某篇小红书笔记的详细信息
  假如 AI 已知目标笔记的标识（如链接或 ID）
  当 AI 执行获取详情命令
  那么 系统应返回该笔记的核心信息摘要（标题、作者、互动数据）
  并且 完整数据（正文、评论、图片链接等）应存入数据库
  并且 返回信息应包含数据库中该笔记的记录标识，便于后续查询引用
  并且 如果该笔记已存在于数据库中，应提示"已更新已有记录"而非重复插入
  并且 AI 可按需查询任意字段而无需重新访问网页
```

#### B3：获取通用网页内容（兜底工具）

```gherkin
场景: AI 需要从一个没有专用适配的普通网站获取信息
  假如 浏览器已启动
  当 AI 执行通用网页读取命令并指定网址
  那么 系统应返回该页面的核心正文内容（去除广告、导航等干扰）
  并且 内容应以精简的 Markdown 格式返回
  并且 返回的文本量应控制在合理范围内，避免信息过载

场景: 目标页面要求登录才能查看内容
  假如 AI 使用通用网页读取命令访问一个需要登录的页面
  当 页面返回了登录墙或重定向到了登录页
  那么 系统应返回"页面需要登录"的提示，而非将登录页的内容当作目标内容返回
  并且 提示信息中应包含检测到的平台标识（如有），便于 AI 决定是否切换到已登录的配置空间
```

#### B4：查看页面上的可交互元素（兜底工具）

```gherkin
场景: AI 需要了解当前页面上有哪些按钮、输入框等可操作的元素
  假如 浏览器已打开某个页面
  当 AI 执行页面检视命令
  那么 系统应返回当前可见区域内所有可交互元素的列表
  并且 每个元素应包含描述性标签（如"搜索按钮"、"用户名输入框"）
  并且 每个元素应有一个可用于后续操作的标识符
```

#### B5：获取页面语义结构（兜底工具）

```gherkin
场景: AI 需要高效理解一个复杂页面的整体结构
  假如 浏览器已打开某个页面
  当 AI 执行获取页面语义结构命令
  那么 系统应通过集成的外部 Provider（如 PinchTab）提取页面的无障碍树 (Accessibility Tree)
  并且 语义树应只包含对 AI 决策有用的信息，剔除纯样式和布局噪声
  并且 返回的信息量应远小于完整页面源码（目标：压缩至原始页面的 5% 以内）
  并且 系统应支持在未来平滑替换该 Provider
```

#### B6：采集页面中的图片

```gherkin
场景: AI 需要保存页面中的图片到本地
  假如 页面中存在目标图片（如笔记配图、商品主图）
  当 AI 执行图片采集命令，指定图片标识或 URL 模式
  那么 系统应优先从浏览器缓存中提取图片文件（不产生额外网络请求）
  并且 将图片保存到本地指定目录
  并且 返回保存的文件路径列表
```

#### B7：采集页面中的视频

```gherkin
场景: AI 需要保存页面中的视频到本地
  假如 页面中正在播放或存在可播放的视频
  当 AI 执行视频采集命令
  那么 系统应通过安全的方式获取视频文件：
    - 优先从浏览器缓存中提取
    - 如缓存不完整，可通过加速播放使视频完整加载后再提取
  并且 将视频保存到本地指定目录
  并且 返回保存的文件路径和视频基本信息（时长、分辨率等）
```

#### B8：获取页面截图（视觉感知）

```gherkin
场景: AI 需要通过视觉模型（VLM）理解页面布局或确认操作结果
  假如 浏览器已打开某个页面
  当 AI 执行页面截图命令
  那么 系统应截取当前可见区域的图像
  并且 将截图保存到本地并返回文件路径
  并且 支持可选参数以截取整个长页面的全尺寸截图
```

### 场景组 C：页面交互（写操作）

#### C1：导航到指定页面

```gherkin
场景: AI 需要打开一个特定网址
  假如 浏览器已启动
  当 AI 执行导航命令并指定目标网址
  那么 浏览器应跳转到目标页面
  并且 系统应等待页面加载完成后再返回确认
  并且 返回信息应包含页面标题和当前网址
```

#### C2：点击页面元素

```gherkin
场景: AI 需要点击页面上的某个按钮或链接
  假如 AI 已通过检视命令获知目标元素的标识
  当 AI 执行点击命令并指定元素标识
  那么 系统应先确认该元素在页面中可见且可点击
  如果 元素不在可见区域，系统应先自动滚动使其可见
  并且 点击动作应尽可能模拟真实人类的操作方式
  并且 系统应返回点击后的页面变化情况

场景: 元素标识匹配到多个元素
  假如 AI 执行了点击命令并指定了元素标识
  当 页面中存在多个匹配该标识的元素
  那么 系统应返回"匹配到多个元素"的提示，并附带各匹配元素的简要描述（位置、文本内容）
  并且 系统不应自动选择其中一个执行点击
  并且 AI 可根据返回的描述信息，使用更精确的标识重新发起操作

场景: 目标元素在规定时间内未出现
  假如 AI 执行了点击命令
  当 目标元素在等待时限内未在页面中找到
  那么 系统应返回"元素未找到"的明确错误信息
  并且 AI 可决定是否重试或换用其他策略
```

#### C3：输入文字

```gherkin
场景: AI 需要在输入框中填写内容
  假如 页面上存在目标输入框
  当 AI 执行输入命令，指定目标元素和要输入的文字
  那么 系统应将文字逐步填入输入框
  并且 输入过程应模拟真实的人类打字行为：
    - 字符之间有随机的时间间隔（快慢不一）
    - 偶尔出现停顿（模拟思考）
    - 偶尔出现删除再重新输入（模拟纠错）
  并且 输入完成后页面应正确识别到内容变化（如发送按钮变为可用）

场景: AI 需要在富文本编辑器（contenteditable）中输入内容
  假如 页面上的目标输入区域是一个富文本编辑器（如小红书笔记正文编辑器）而非标准 input/textarea
  当 AI 执行输入命令
  那么 系统应自动识别输入区域类型（标准输入框 vs 富文本编辑器）
  并且 根据类型采用相应的输入策略，确保内容被正确写入并触发编辑器的状态更新
  并且 支持输入包含换行的多段落文本
```

#### C4：上传文件

```gherkin
场景: AI 需要上传本地文件（如发布带图笔记）
  假如 页面上存在文件上传入口
  当 AI 执行上传命令，指定本地文件路径
  那么 系统应将文件传递到浏览器中
  并且 页面应像用户手动选择文件一样正常接收并显示预览

场景: 页面使用拖拽上传而非标准文件选择器
  假如 页面上的文件上传区域不是标准的 <input type="file">，而是拖拽上传区域
  当 AI 执行上传命令
  那么 系统应自动检测上传组件类型
  如果 是标准文件选择器，通过注入文件到 input 元素完成上传
  如果 是拖拽上传区域，通过模拟拖放事件（dragenter/dragover/drop）将文件传递给页面
  并且 上传完成后应等待页面显示文件预览或上传进度后再返回确认
```

#### C5：页面滚动

```gherkin
场景: AI 需要浏览更多内容（如加载瀑布流列表）
  假如 页面内容超出当前可见区域
  当 AI 执行滚动命令
  那么 页面应按指定方向和距离滚动
  并且 滚动过程应平滑自然，模拟人类浏览习惯
  并且 系统应返回滚动后的页面状态（是否到底、是否有新内容加载）
```

#### C6：特定平台的高级操作（专用工具）

```gherkin
场景: AI 需要在小红书发送私信
  假如 小红书账号已登录
  当 AI 执行小红书发送私信命令，指定目标用户和消息内容
  那么 系统应自动完成导航到对话页面、输入消息、点击发送的完整流程
  并且 每一步操作之间应有合理的等待时间
  并且 发送结果（成功/失败/需要验证）应返回给 AI

场景: AI 需要在小红书发布笔记
  假如 小红书账号已登录
  当 AI 执行发布笔记命令，指定标题、正文和图片路径
  那么 系统应自动完成上传图片、填写标题和正文、点击发布的完整流程
  并且 发布结果应返回给 AI，包括：成功/失败/审核中状态、笔记 ID 或 URL（如可获取）、发布时间

场景: AI 需要获取小红书用户的公开资料
  假如 小红书账号已登录
  当 AI 执行获取用户主页命令，指定目标用户
  那么 系统应返回该用户的公开资料和笔记列表摘要
  并且 完整数据应存入数据库供后续查询

场景: AI 需要在小红书笔记下发布评论
  假如 小红书账号已登录
  当 AI 执行发布评论命令，指定目标笔记和评论内容
  那么 系统应自动完成导航到笔记页面、输入评论、提交的完整流程
  并且 评论结果（成功/失败/需要验证）应返回给 AI

场景: AI 需要查看小红书私信列表
  假如 小红书账号已登录
  当 AI 执行获取私信列表命令
  那么 系统应返回当前账号的私信会话列表摘要
  并且 完整的消息内容应存入数据库供后续查询
```

#### C7：管理多标签页

```gherkin
场景: AI 需要在不丢失当前页面状态的情况下打开新页面（如查阅参考资料）
  假如 浏览器已打开页面 A
  当 AI 执行新建标签页命令并导航至页面 B
  那么 系统应在后台或新标签页中加载页面 B
  并且 AI 可随时执行切换标签页命令回到页面 A
  并且 页面 A 的滚动位置和输入状态应保持不变
```

### 场景组 D：数据管理

#### D1：查询已采集的数据

```gherkin
场景: AI 需要从已采集的数据中筛选特定信息

  当 AI 执行数据查询命令，指定筛选条件（如"点赞数大于1000的笔记"）
  那么 系统应返回符合条件的结果摘要
  并且 结果数量应可控（支持分页或数量限制）
  并且 AI 可进一步细化查询条件

场景: AI 需要查看某条数据的完整原始信息
  假如 AI 已知目标数据的标识
  当 AI 执行详情查询命令，指定标识和需要的字段
  那么 系统应仅返回 AI 指定的字段内容
  并且 不应一次性返回所有原始数据
```

#### D2：操作去重与防重复

```gherkin
场景: AI 即将执行一个可能重复的操作（如再次给同一用户发私信）
  假如 数据库中记录了历史操作
  当 AI 执行操作前查询是否已执行过
  那么 系统应准确返回该操作是否已有记录

#### D3：数据清理

场景: 长期使用后本地数据占用过多磁盘空间
  假如 数据库中积累了大量历史数据
  当 用户执行数据清理命令，指定保留时间范围
  那么 系统应删除超出时间范围的原始采集数据
  并且 用户主动标记为"保留"的数据不应被清理
```

#### D4：数据标记与导出

```gherkin
场景: 用户希望保护特定数据不被自动清理
  假如 数据库中存有已采集的数据
  当 用户对指定数据执行"标记为保留"命令
  那么 该数据在后续自动清理时应被豁免
  并且 保留标记可随时取消

场景: AI 需要将查询结果导出为文件
  假如 数据库中存有已采集的数据
  当 AI 执行数据导出命令，指定查询条件和输出格式（CSV/JSON）
  那么 系统应将符合条件的数据写入本地文件
  并且 返回导出的文件路径和记录数量
```

### 场景组 E：异常处理与容错

#### E1：遇到验证码

```gherkin
场景: 操作过程中目标平台弹出验证码
  假如 AI 正在执行自动化任务
  当 插件检测到页面出现验证码（如滑块、图片选择）

  并且 通知 AI"需要人工介入处理验证码"
  并且 如果浏览器窗口可见，用户可手动完成验证
  并且 验证通过后，系统应能恢复并继续执行后续任务
```

#### E2：浏览器意外关闭

```gherkin
场景: 浏览器进程意外崩溃或被关闭
  假如 AI 正在执行任务
  当 系统检测到浏览器进程不再存活
  那么 系统应通知 AI"浏览器已断开"
  并且 AI 可选择重新启动浏览器继续任务
  并且 之前已存入数据库的数据不应丢失
```

#### E3：页面加载异常

```gherkin
场景: 目标页面加载超时或返回错误
  假如 AI 发出了导航或操作命令
  当 页面在合理时间内未完成加载，或返回了错误页面
  那么 系统应返回明确的错误类型（超时/404/被拦截等）
  并且 AI 可决定是否重试
```

#### E4：操作频率保护

```gherkin
场景: AI 在短时间内发出过多操作请求
  假如 系统内置了各平台的安全操作频率规则
  当 AI 的操作请求超出安全频率
  那么 系统应主动延迟执行，而非立即拒绝
  并且 通知 AI 当前正在等待冷却
  并且 达到安全间隔后自动恢复执行
```

#### E5：网络连接中断

```gherkin
场景: 操作过程中本机网络断开
  假如 AI 正在执行需要网络的操作（如导航、数据采集）
  当 系统检测到页面加载失败且原因是本机网络不可用（而非目标服务器返回错误）
  那么 系统应返回"网络不可用"的明确错误类型，区别于页面自身的 4xx/5xx 错误
  并且 AI 可决定是否等待网络恢复后重试
  并且 网络恢复后系统不应残留异常状态，应能正常接受后续命令
```

#### E6：账号被平台封禁或限制

```gherkin
场景: 操作过程中检测到当前账号被平台封禁或限制
  假如 AI 正在执行写操作（如发私信、发评论）
  当 插件检测到平台返回账号受限或封禁的提示（如"账号已被限制操作"、"违规封禁"）
  那么 系统应立即停止当前操作序列，不再自动重试写操作以避免加重封禁
  并且 返回错误信息中包含平台提示原文
  并且 暂停该配置空间的后续写操作，直到用户手动确认解除（由用户判断封禁性质）
  并且 AI 可查询封禁状态但不可绕过暂停限制
```

#### E7：专用工具因平台变更而失效

```gherkin
场景: 专用工具依赖的平台页面结构或 API 发生变更
  假如 AI 使用小红书专用工具执行搜索或获取详情
  当 插件未拦截到预期的网络请求，或拦截到的数据格式与预期不匹配
  那么 系统应返回"专用工具执行异常，可能因平台更新导致"的明确错误
  并且 附带诊断信息（如"预期拦截 URL 模式 X，实际未匹配到"）
  并且 提示 AI 可降级使用兜底工具（如通用网页读取、页面检视）替代
  并且 不应因专用工具失效而导致系统崩溃或卡死
```

  那么 系统应优先返回一份精简摘要（总数、前几条预览）
  并且 完整数据应存入数据库供后续按需查询
  并且 AI 的上下文中不应被灌入大量原始数据

  假如 数据库中已存有完整的采集数据

### 场景组 G：边界情况

#### G1：重复初始化工作环境

```gherkin
场景: 用户对已初始化的环境再次执行初始化命令
  假如 工作目录已存在且包含配置空间和数据库
  当 用户再次执行初始化命令
  那么 系统应检测到工作目录已存在
  并且 跳过初始化，返回"工作环境已存在，无需重复初始化"的提示
  并且 不覆盖、不删除任何已有数据
  并且 提供可选的 --force 参数用于强制重新初始化
  并且 强制重新初始化前应明确警告用户将丢失所有配置空间和数据
```

#### G2：对已运行的浏览器再次发出启动命令

```gherkin
场景: AI 对同一配置空间重复发出启动浏览器命令
  假如 该配置空间的浏览器已在运行且连接正常
  当 AI 再次发出启动命令并指定相同的配置空间
  那么 系统不应启动第二个浏览器实例
  并且 返回"浏览器已在运行"的提示，附带当前实例信息（PID、打开的标签页数）
  并且 AI 可直接继续发出操作命令

场景: AI 对不同配置空间发出启动浏览器命令
  假如 配置空间 A 的浏览器已在运行
  当 AI 发出启动命令并指定配置空间 B
  那么 系统应为配置空间 B 启动独立的浏览器实例
  并且 两个浏览器实例互不干扰
  并且 AI 在发出命令时需明确指定目标配置空间，系统据此路由命令
```

#### G3：输入内容包含特殊字符

```gherkin
场景: AI 输入的文字包含 emoji、换行符等特殊字符
  假如 页面上存在目标输入框
  当 AI 执行输入命令，内容包含 emoji 表情、换行符、零宽字符或其他特殊 Unicode 字符
  那么 系统应正确传递所有字符，输入框中的内容应与指定内容完全一致
  并且 拟真输入对于 emoji 等多字节字符应一次性输入（而非拆字节）
  并且 如果目标输入框不支持某些字符（如纯数字框收到字母），应返回输入未被接受的提示

场景: AI 输入的文字超出平台字数限制
  假如 目标平台对输入内容有字数上限（如小红书标题限 20 字、正文限 1000 字）
  当 AI 执行输入命令，内容超出平台允许的长度
  那么 系统应在输入完成后检测页面是否出现字数超限提示
  并且 将超限信息返回给 AI（如"已输入 1200 字，平台限制 1000 字"）
  并且 由 AI 决定是否截断或修改内容
```

#### G4：元素标识匹配到多个元素

```gherkin
场景: AI 指定的元素标识在页面中对应多个元素
  假如 AI 通过检视命令获取了元素标识
  当 AI 执行点击或输入命令，但该标识在当前 DOM 中匹配到多个可见元素
  那么 系统应返回"元素标识不唯一"的错误
  并且 附带匹配到的元素数量和各元素的简要描述（如位置、文本片段）
  并且 AI 可据此用更精确的标识重新操作
```

#### G5：写操作的目标不可达

```gherkin
场景: AI 尝试给不可达的用户发送私信
  假如 小红书账号已登录
  当 AI 执行发送私信命令，但目标用户存在以下情况之一：
    - 用户已注销或不存在
    - 用户已将当前账号拉黑
    - 用户关闭了私信功能
    - 用户设置了仅允许粉丝私信
  那么 系统应返回具体的失败原因（如"用户不存在"/"已被对方拉黑"/"对方未开启私信"）
  并且 返回"目标用户不可达"的错误类型及平台提示原文
  并且 该次操作应记录为"发送失败"而非"未执行"

场景: AI 尝试评论已关闭评论区的笔记
  假如 小红书账号已登录
  当 AI 执行发布评论命令，但目标笔记的评论区已关闭或笔记已被删除
  那么 系统应返回"目标内容不可评论"的错误类型及原因
```

#### G6：上传文件异常

```gherkin
场景: AI 指定的本地文件不存在
  假如 页面上存在文件上传入口
  当 AI 执行上传命令，但指定的本地文件路径不存在
  那么 系统应在尝试上传前就返回"文件不存在"的错误
  并且 不向浏览器发送任何上传请求

场景: AI 上传的文件格式不被平台支持
  假如 页面的上传组件限制了文件类型（如只允许 jpg/png）
  当 AI 执行上传命令，指定了一个不被接受的文件类型
  那么 系统应在页面拒绝文件后返回"文件格式不被支持"的错误
  并且 附带平台支持的文件类型列表（如果页面有提供）

场景: AI 上传的文件超出平台大小限制
  假如 平台对上传文件有大小限制
  当 AI 上传了超出限制的文件
  那么 系统应在页面显示超限提示后将此信息返回给 AI
```

#### G7：页面包含 iframe

```gherkin
场景: 目标交互元素位于 iframe 内部
  假如 页面包含 iframe（如嵌入的编辑器、第三方登录框）
  当 AI 执行检视命令获取可交互元素列表
  那么 同源 iframe 内的可交互元素应被包含在返回列表中，并标注其所属 iframe 标识
  并且 对于同源 iframe 内的元素，AI 可通过"iframe 标识 + 元素标识"的方式进行操作
  并且 跨域 iframe 内的元素不可访问，应在列表中标注"存在跨域 iframe，内部元素不可操作"
```

#### G8：配置空间命名规则

```gherkin
场景: 用户创建配置空间时使用了非法名称
  假如 工作环境已初始化
  当 用户执行创建配置空间命令，名称包含以下情况之一：
    - 包含文件系统不允许的特殊字符（如 / \ : * ? " < > |）
    - 长度超过 64 个字符
    - 与已有配置空间重名
    - 为空字符串或纯空格
  那么 系统应返回具体的命名错误原因
  并且 提供命名规则说明（允许中英文、数字、短横线、下划线，长度 1-64 字符）
  并且 不创建任何目录或文件
```

#### G9：超大页面处理

```gherkin
场景: 当前页面 DOM 结构极其复杂（如长列表、瀑布流已加载大量内容）
  假如 页面中可交互元素超过 500 个
  当 AI 执行页面检视命令 (NAV-03) 或获取无障碍树命令 (NAV-07)
  那么 系统应设置合理的元素数量上限（默认返回最多 200 个元素）
  并且 返回信息中标注"页面共有 N 个可交互元素，当前返回前 200 个"
  并且 支持通过参数指定区域（如仅当前视口内）或偏移量来获取后续元素
```

#### G10：媒体资源不可用

```gherkin
场景: AI 尝试采集的图片或视频资源不可用
  假如 页面中存在图片/视频元素，但实际资源已失效
  当 AI 执行媒体采集命令，目标资源存在以下情况之一：
    - 资源在浏览器缓存中不存在或已被清除
    - 资源 URL 返回 404（已被删除）
    - 资源被防盗链机制拦截（返回 403 或替代图片）
    - 资源 URL 已过期（签名 URL 超时）
  那么 系统应返回每个资源的采集结果（成功/失败及具体原因）
  并且 对于缓存未命中，建议 AI 先刷新页面或播放视频使资源加载到缓存后重试
  并且 对于防盗链拦截，提示 AI 优先使用缓存提取方式而非直接请求
  并且 成功采集的文件正常保存，失败的不生成空文件
  并且 返回的汇总信息应包含"成功 N 个，失败 M 个"
```

### 场景组 H：空状态处理

#### H1：数据库无数据时查询

```gherkin
场景: AI 在数据库中没有任何数据时执行查询
  假如 数据库刚初始化，尚未进行过任何数据采集
  当 AI 执行数据查询命令
  那么 系统应返回统一的空结果格式：{ total: 0, items: [], message: "暂无数据" }
  并且 不应返回错误或异常
  并且 提示 AI 可先执行采集命令获取数据

场景: AI 查询条件无匹配结果
  假如 数据库中已有数据，但不满足 AI 指定的筛选条件
  当 AI 执行数据查询命令（如"点赞数大于100万的笔记"）
  那么 系统应返回：{ total: 0, items: [], message: "无匹配结果" }
  并且 附带当前数据库中该类型数据的总量信息（如"数据库中共有 500 条笔记数据"），帮助 AI 判断是条件过严还是确实无数据
```

#### H2：搜索结果为空

```gherkin
场景: AI 在平台上搜索但没有任何结果
  假如 浏览器已启动且账号已登录
  当 AI 执行搜索命令，但平台返回零结果（关键词无匹配或关键词被屏蔽）
  那么 系统应返回：{ total: 0, items: [], message: "搜索无结果" }
  并且 不向数据库写入任何记录
  并且 如果平台页面上显示了搜索建议或提示信息，应一并返回给 AI

场景: AI 指定的搜索数量超过平台实际返回数量
  假如 AI 请求搜索 50 条结果
  当 平台实际只返回了 12 条（关键词较冷门）
  那么 系统应返回实际采集到的 12 条数据
  并且 摘要中明确标注"请求 50 条，实际获取 12 条（已到达平台结果末尾）"
```

#### H3：用户主页无内容

```gherkin
场景: AI 获取的目标用户没有发布过任何笔记
  假如 小红书账号已登录
  当 AI 执行获取用户主页命令，目标用户存在但未发布过笔记
  那么 系统应返回该用户的基本资料（昵称、简介、粉丝数等）
  并且 笔记列表部分返回空：{ notes_count: 0, notes: [] }
  并且 用户资料仍应正常存入数据库
```

#### H4：私信列表为空

```gherkin
场景: AI 获取私信列表但当前账号没有任何私信记录
  假如 小红书账号已登录（新账号或已清空私信）
  当 AI 执行获取私信列表命令
  那么 系统应返回：{ total: 0, conversations: [], message: "暂无私信记录" }
  并且 不应返回错误
  并且 不向数据库写入空记录
```

#### H5：页面无可交互元素

```gherkin
场景: AI 检视的页面没有任何可操作的元素
  假如 浏览器打开了一个纯静态展示页面（如纯文本页面、图片页面、错误页面）
  当 AI 执行页面检视命令
  那么 系统应返回：{ total: 0, elements: [], message: "当前页面无可交互元素" }
  并且 建议 AI 可尝试获取页面正文内容 (NAV-04) 或页面语义结构 (NAV-07) 来理解页面
```

#### H6：导出无匹配数据

```gherkin
场景: AI 导出数据时查询条件无匹配记录
  假如 数据库中存有数据，但不满足导出的筛选条件
  当 AI 执行数据导出命令
  那么 系统应返回提示："无匹配数据，未生成导出文件"
  并且 不创建空的 CSV/JSON 文件
  并且 附带当前数据库中可导出数据的总量信息
```

#### H7：缓存中无目标媒体文件

```gherkin
场景: AI 采集媒体文件时浏览器缓存中不存在目标资源
  假如 页面上显示了图片/视频，但浏览器缓存中没有对应的文件（如页面刚打开、缓存已被清理）
  当 AI 执行媒体采集命令
  那么 系统应返回"缓存中未找到目标文件"的提示
  并且 对于图片：建议 AI 先滚动页面使图片完整加载后重试
  并且 对于视频：建议 AI 先播放视频使其完整缓存后重试
  并且 不应自动发起额外的网络请求去下载资源（遵守"被动观察"原则）
```

#### H8：首次使用时查看配置空间列表

```gherkin
场景: AI 在未创建任何配置空间时查看列表
  假如 工作环境刚初始化，尚未创建任何配置空间
  当 AI 执行列出配置空间命令
  那么 系统应返回：{ total: 0, profiles: [], message: "暂无配置空间" }
  并且 提示 AI 需要先执行平台登录命令来创建配置空间
```

### 场景组 I：数据一致性保障

#### I1：采集中断时的部分数据处理

```gherkin
场景: 批量采集过程中浏览器崩溃或网络中断
  假如 AI 发起了一个批量采集任务（如搜索 50 条笔记）
  当 采集进行到第 30 条时浏览器崩溃或网络中断
  那么 已成功写入数据库的前 30 条数据应完整保留，不被回滚
  并且 该采集批次应被标记为"未完成"状态，记录已采集数量和中断原因
  并且 AI 重新发起相同搜索时，系统应告知"上次采集未完成（已采集 30/50）"
  并且 AI 可选择从中断处继续采集或重新开始
```

#### I2：媒体文件与数据库记录的同步

```gherkin
场景: 媒体采集过程中文件保存与数据库记录需保持一致
  假如 AI 执行了批量图片采集命令，目标包含 10 张图片
  当 系统逐张保存图片到本地并写入数据库记录
  那么 每张图片应先保存文件再写入数据库记录（文件为主，记录为辅）
  并且 如果文件保存成功但数据库写入失败，文件应保留，系统返回"文件已保存但记录未入库"的警告
  并且 如果文件保存失败，不应在数据库中创建该条记录
  并且 最终返回结果中应分别列出"文件保存成功 N 张，记录入库成功 M 条，不一致 K 条"

场景: 数据库记录指向的媒体文件已不存在
  假如 数据库中记录了某张图片的本地路径
  当 AI 查询该图片信息，但对应的本地文件已被手动删除或移动
  那么 系统应在返回信息中标注"文件缺失"状态
  并且 不应因文件缺失而报错或阻塞查询
```

#### I3：写操作记录与实际执行结果的一致性

```gherkin
场景: 写操作执行成功但操作记录写入失败
  假如 AI 执行了发送私信命令
  当 私信在平台上实际发送成功，但操作记录写入本地数据库时失败（如数据库被锁定）
  那么 系统应优先返回操作的实际结果（发送成功），并附带警告"操作已执行但记录未入库"
  并且 不应因记录写入失败而向 AI 报告操作失败
  并且 系统应在下次数据库可用时尝试补写该操作记录

场景: 写操作执行失败时操作记录需正确反映
  假如 AI 执行了发送私信命令
  当 操作在平台上执行失败（如被拒绝、超时）
  那么 操作记录应准确记录失败状态和失败原因
  并且 AI 后续查询操作历史时应能看到该失败记录，而非认为操作从未执行过
```

#### I4：数据去重的并发安全

```gherkin
场景: 多个数据采集几乎同时触发了对同一内容的入库
  假如 AI 在标签页 A 和标签页 B 同时浏览了包含相同笔记的页面
  当 两个标签页的数据拦截几乎同时传回 CLI
  那么 系统应保证同一业务 ID 的数据只入库一次（不产生重复记录）
  并且 后到达的数据应触发更新而非报错
  并且 不应因并发写入导致数据库锁死或数据损坏
```

#### I5：数据清理的级联完整性

```gherkin
场景: 按时间范围清理数据时关联资源应同步清理
  假如 数据库中存有笔记数据，且笔记关联了本地保存的图片/视频文件
  当 用户执行按时间范围清理数据的命令
  那么 被清理的数据库记录所关联的本地媒体文件应一并删除
  并且 清理结果应汇报：删除了 N 条数据库记录和 M 个媒体文件，释放了 X MB 磁盘空间
  并且 标记为"保留"的数据及其关联的媒体文件均不受影响

场景: 删除配置空间时的关联数据处理
  假如 用户决定删除某个配置空间
  当 用户执行删除配置空间命令
  那么 系统应提示即将删除的关联数据范围：浏览器配置、该空间采集的数据库记录数、关联的媒体文件数和总大小
  并且 用户确认后，系统应完整删除配置空间目录、关联的数据库记录和媒体文件
  并且 不影响其他配置空间的任何数据
```

### 场景组 J：故障恢复

#### J1：批量任务断点续传

```gherkin
场景: AI 希望从上次中断的位置继续执行批量采集
  假如 上次执行"搜索小红书笔记 100 条"的任务在第 40 条时因浏览器崩溃而中断
  当 AI 重新启动浏览器后，再次执行相同的搜索命令
  那么 系统应检测到存在未完成的同类批次
  并且 提示 AI"存在未完成的采集批次（40/100），是否从断点继续？"
  并且 如果 AI 选择继续，系统从第 41 条开始采集
  并且 如果 AI 选择重新开始，系统从头采集并将之前的批次标记为"已废弃"
```

#### J2：数据库损坏检测与恢复

```gherkin
场景: 系统检测到数据库文件异常
  假如 用户执行了任意需要访问数据库的命令
  当 系统检测到 SQLite 数据库文件损坏（integrity check 失败）
  那么 系统应返回"数据库文件损坏"的错误
  并且 如果存在自动备份文件，提示用户"检测到最近的备份（日期：X），是否恢复？"
  并且 如果没有备份，提示用户可执行"重建数据库"命令（将丢失所有已采集数据，但保留配置空间和登录状态）
  并且 系统不应在数据库损坏的状态下尝试继续写入数据

场景: 系统定期自动备份数据库
  假如 系统正常运行且数据库中有数据
  当 满足自动备份条件（如每日首次启动时，或累计写入超过 1000 次后）
  那么 系统应自动创建数据库的备份文件
  并且 备份文件命名应包含时间戳
  并且 只保留最近 N 个备份（默认 3 个），自动清理更早的备份
  并且 备份操作不应阻塞正常的命令执行
```

#### J3：CLI 意外退出后重新接管浏览器

```gherkin
场景: CLI 进程被意外终止后重新启动
  假如 CLI 进程在运行过程中被 kill 或崩溃，但浏览器进程仍在运行
  当 用户重新启动 CLI 并执行启动浏览器命令（指定相同配置空间）
  那么 系统应检测到该配置空间已有运行中的浏览器进程
  并且 尝试与已有浏览器中的插件重新建立通信连接
  并且 如果重连成功，返回"已重新接管浏览器"的确认，附带当前浏览器状态
  并且 如果重连失败（如插件已断开且无法恢复），提示用户手动关闭残留浏览器后重新启动
```

#### J4：页面卡死检测与恢复

```gherkin
场景: 当前页面因 JavaScript 错误或资源耗尽而无响应
  假如 AI 发出了操作命令（如点击、输入）
  当 页面在合理时间内（如 30 秒）没有任何响应
  那么 系统应检测到页面无响应状态
  并且 返回"页面无响应"的错误类型
  并且 建议 AI 可尝试以下恢复策略：刷新页面、关闭并重新打开标签页、或重启浏览器
  并且 不应无限等待页面恢复
```

#### J5：限流冷却状态的持久化

```gherkin
场景: CLI 重启后需要恢复限流冷却计时器
  假如 系统因某平台的操作频率过高而进入冷却等待状态（如距下次可操作还需 5 分钟）
  当 CLI 在冷却期间被重启
  那么 系统应从持久化存储中恢复该平台的冷却截止时间
  并且 如果冷却时间尚未到期，继续等待而非立即允许操作
  并且 如果冷却时间已过期，正常允许操作
```

### 场景组 K：状态管理与并发控制

#### K1：浏览器生命周期状态机

```gherkin
场景: 系统维护每个浏览器实例的完整生命周期状态
  假如 每个浏览器实例的生命周期包含以下状态：
    - "未启动"：初始状态，浏览器进程不存在
    - "启动中"：浏览器进程已拉起，等待插件连接
    - "就绪"：插件已连接，可接受命令
    - "执行中"：正在执行一条操作命令
    - "已暂停"：因验证码或登录失效而暂停，等待人工介入
    - "关闭中"：正在清理资源和关闭进程
    - "异常断开"：浏览器崩溃或插件失联
  当 AI 发出任何操作命令
  那么 系统应根据当前状态判断命令是否可执行：
    - "就绪"：接受所有命令
    - "执行中"：拒绝新的操作命令（查询类命令除外），返回正在执行的命令信息
    - "已暂停"：拒绝操作命令，返回暂停原因（验证码/登录失效）
    - "启动中"/"关闭中"：拒绝操作命令，提示等待状态转换完成
    - "未启动"/"异常断开"：拒绝操作命令，提示需要先启动或重启浏览器

  合法的状态转换路径：
    未启动 → 启动中（执行启动命令）
    启动中 → 就绪（插件连接成功）
    启动中 → 异常断开（启动超时或插件连接失败）
    就绪 → 执行中（收到操作命令）
    执行中 → 就绪（命令执行完成或失败）
    执行中 → 已暂停（检测到验证码或登录失效）
    已暂停 → 就绪（人工介入完成）
    就绪/执行中 → 异常断开（浏览器崩溃或插件失联）
    就绪/已暂停 → 关闭中（收到关闭命令）
    关闭中 → 未启动（清理完成）
    异常断开 → 未启动（确认残留进程已清理）
    异常断开 → 启动中（执行重启命令，先尝试重连已有浏览器，重连失败后清理残留再重新启动）

场景: AI 查询浏览器当前状态
  假如 CLI 正在运行
  当 AI 执行状态查询命令
  那么 系统应返回浏览器的当前状态名称
  并且 附带与状态相关的上下文信息：
    - "执行中"：正在执行的命令名称、已执行时长
    - "已暂停"：暂停原因（验证码/登录失效）、暂停时间
    - "异常断开"：断开时间、断开前的最后状态、最后执行的命令
    - "就绪"：当前打开的标签页数量、所在页面 URL
```

#### K2：通信连接状态机

```gherkin
场景: CLI 与插件的通信连接维护独立的状态生命周期
  假如 通信连接的状态包含：
    - "未连接"：浏览器未启动或连接尚未建立
    - "连接中"：正在尝试建立连接（含通道降级尝试，参见 A5）
    - "已连接"：双向通信正常
    - "重连中"：检测到断线，正在自动重连（如 MV3 Service Worker 被浏览器回收）
    - "已断开"：所有重连尝试失败
  当 连接从"已连接"变为"重连中"
  那么 浏览器状态应临时冻结，不接受新的操作命令
  并且 重连期间收到的新操作命令应排队等待（而非直接失败），队列上限为 5 条
  并且 如果重连在 30 秒内成功，浏览器状态恢复为重连前的状态，排队命令按序执行
  并且 如果重连超时，浏览器状态转为"异常断开"，排队命令全部返回"连接已断开"错误
  并且 通信连接状态的变化应同步反映在浏览器状态查询结果中
```

#### K3：登录状态生命周期

```gherkin
场景: 系统跟踪并持久化每个配置空间的登录状态
  假如 每个配置空间维护独立的登录状态：
    - "未登录"：配置空间刚创建，从未登录过
    - "已登录"：登录状态有效（最后验证时间：X）
    - "已失效"：Cookie 过期或被平台踢出（失效时间：Y）
  当 登录状态发生变更
  那么 新状态及变更时间应立即持久化到配置空间元数据中
  并且 如果是从"已登录"变为"已失效"，应主动通知 AI 并说明原因
  并且 登录状态不因浏览器关闭/重启而丢失（登录态保存在持久化的配置空间中）

场景: 操作过程中检测到平台要求重新登录
  假如 AI 正在执行需要登录的操作（如发私信、发评论）
  当 插件检测到平台返回了"请重新登录"的页面、弹窗、或接口 401 响应
  那么 系统应将该配置空间的登录状态更新为"已失效"
  并且 浏览器状态从"执行中"转为"已暂停"
  并且 当前操作返回失败，错误类型为"登录已失效"
  并且 AI 可选择：(a) 触发重新登录流程（打开可见窗口）或 (b) 切换到其他配置空间
```

#### K4：命令串行执行模型

```gherkin
场景: 同一浏览器实例的操作命令严格串行执行
  假如 浏览器处于"执行中"状态（正在执行一条操作命令）
  当 AI 发出一条新的操作命令（点击、输入、导航、采集等涉及浏览器的命令）
  那么 系统应立即拒绝新命令，返回信息包含：
    - 错误类型："命令冲突"
    - 正在执行的命令名称和参数摘要
    - 已执行时长
  并且 不将新命令排队（避免 AI 误以为命令已被接受）

场景: 查询类命令不受串行限制
  假如 浏览器正在执行操作命令
  当 AI 发出以下类型的命令：
    - 数据库查询（QRY-*）
    - 浏览器状态查询（ENV-06）
    - 数据导出（QRY-04）
    - 配置空间列表（ENV-05）
  那么 这些命令应能立即执行并返回结果
  并且 不影响正在执行的浏览器操作，不改变浏览器状态
```

#### K5：配置空间独占锁

```gherkin
场景: 同一配置空间在同一时刻只能被一个 CLI 进程使用
  假如 CLI 进程 A 已以配置空间 X 启动了浏览器
  当 另一个 CLI 进程 B 试图以配置空间 X 启动浏览器
  那么 进程 B 应收到"配置空间 X 已被占用"的错误
  并且 错误信息包含占用进程的 PID 和启动时间
  并且 进程 B 不应等待或重试，而是立即返回

场景: 占用进程已不存在时的锁自动清理
  假如 配置空间 X 存在锁文件，但记录的 PID 对应的进程已不存在
  当 新的 CLI 进程尝试使用配置空间 X
  那么 系统应检测到锁是残留的（进程已死亡）
  并且 自动清理残留锁文件
  并且 允许新进程正常使用该配置空间
  并且 如果存在残留的浏览器进程（属于已死亡的 CLI），应先终止残留浏览器再继续
```

#### K6：多浏览器实例路由

```gherkin
场景: 多个浏览器实例同时运行时的命令路由
  假如 系统同时运行了配置空间 A 和配置空间 B 的浏览器实例
  当 AI 发出操作命令
  那么 命令中必须包含目标配置空间标识
  并且 系统根据标识将命令路由到对应的浏览器实例
  并且 如果命令未指定目标配置空间且当前有多个实例运行，系统应返回错误并列出所有可用实例
  并且 如果只有一个实例运行，可自动路由到该实例（无需显式指定）

场景: AI 向不存在的浏览器实例发出命令
  假如 系统中运行着配置空间 A 的浏览器
  当 AI 发出命令但指定了未启动的配置空间 B
  那么 系统应返回"配置空间 B 的浏览器未启动"的错误
  并且 附带当前运行中的实例列表
```

#### K7：执行策略热切换

```gherkin
场景: 浏览器运行期间切换执行策略
  假如 浏览器正在运行
  当 用户切换执行策略（如从"标准模式"切到"安全模式"）
  那么 如果浏览器处于"就绪"状态，切换立即生效
  并且 如果浏览器处于"执行中"状态，等待当前命令完成后自动切换
  并且 如果浏览器处于"已暂停"/"异常断开"等状态，策略设置应被保存，在恢复到"就绪"后生效
  并且 切换不需要重启浏览器，不影响已保存的数据

场景: 切换到安全模式时的前置检查
  假如 当前浏览器窗口不可见（最小化或被遮挡）
  当 用户切换到安全模式（要求窗口可见以进行 OS 级输入模拟）
  那么 系统应提示"安全模式要求浏览器窗口可见，请确认"
  并且 如果无法使窗口可见，切换应失败并保持原策略
```

### 场景组 L：性能约束

#### L1：命令响应延迟要求

```gherkin
场景: AI 发出的命令应在可接受的时间内返回结果
  假如 浏览器已启动且连接正常
  当 AI 发出任何命令
  那么 纯本地命令（数据库查询、状态查看）应在 1 秒内返回
  并且 页面操作命令（点击、输入、导航）应在 30 秒内返回结果或超时错误
  并且 数据采集命令（搜索、详情获取）应在命令发出后 5 秒内返回首条中间反馈（如"正在采集..."），最终结果在合理时间内返回
  并且 超过预期时间未完成的命令应支持 AI 主动取消
```

#### L2：无障碍树提取性能

```gherkin
场景: 复杂页面的无障碍树提取应在合理时间内完成
  假如 浏览器打开了一个 DOM 节点极多的复杂页面（如已加载数百条内容的瀑布流）
  当 AI 执行获取页面语义结构命令
  那么 提取过程应在 5 秒内完成
  并且 如果页面过于复杂导致提取超时，应返回已提取的部分结果并标注"结果不完整，页面结构过于复杂"
  并且 提取过程不应导致页面卡顿或浏览器无响应
```

#### L3：数据库查询性能

```gherkin
场景: 数据库长期使用后查询仍保持可接受的速度
  假如 数据库中已积累了大量数据（10 万条以上记录）
  当 AI 执行条件查询命令
  那么 带索引字段的查询（平台、数据类型、业务 ID、采集时间）应在 1 秒内返回
  并且 全文搜索或复杂条件组合查询允许更长时间但不应超过 5 秒
  并且 系统应在初始化数据库时为高频查询字段创建索引
```

#### L4：通信通道消息大小控制

```gherkin
场景: CLI 与插件之间传输大体量数据
  假如 CLI 与插件通过 Native Messaging 通信
  当 单条消息的数据量接近或超过 Chrome Native Messaging 的 1MB 限制
  那么 系统应首选通过压缩、降质、摘要等方式将消息体控制在 1MB 以内
  并且 确实无法压缩到 1MB 以下时，作为兜底方案自动将大消息拆分为多条分片传输，接收端自动重组
  并且 WebSocket 通道不受此限制，但单条消息大小也应控制在 10MB 以内
```

#### L5：磁盘空间监控

```gherkin
场景: 长期使用导致本地存储空间不足
  假如 系统在数据写入时检测到磁盘可用空间不足
  当 可用空间低于 500MB
  那么 系统应在命令返回中附带"磁盘空间不足"的警告
  并且 建议用户执行数据清理命令
  当 可用空间低于 100MB
  那么 系统应停止所有媒体文件采集操作（数据库写入仍允许）
  并且 警告信息中附带当前工作目录的磁盘占用明细（数据库大小、媒体文件大小）
```

#### L6：拟真输入的性能预期

```gherkin
场景: AI 需要输入一段较长的文本（如 500 字以上的笔记正文）
  假如 AI 执行输入命令且开启了拟真输入模式
  当 输入内容较长，拟真输入预计耗时超过 3 秒
  那么 系统应在开始输入后 3 秒内返回"正在输入..."的中间状态及预计完成时间
  并且 AI 可通过可选参数关闭拟真模式以加速输入（牺牲自然性换取速度）
  并且 即使在拟真模式下，实际输入时间不应超过文本字数 × 200ms（如 500 字不超过 100 秒）
```

### 场景组 M：安全保护

#### M1：CLI 与插件的通信认证

```gherkin
场景: 防止非授权程序冒充 CLI 连接到浏览器插件
  假如 插件在浏览器中运行并等待 CLI 连接
  当 有程序尝试与插件建立通信连接
  那么 Native Messaging 通道应通过 Chrome 原生的 allowed_origins 机制限制只有注册的 CLI 程序可连接
  并且 WebSocket 通道应在连接建立时进行令牌验证（CLI 启动时生成一次性令牌，插件验证后才接受连接）
  并且 未通过验证的连接应被立即拒绝，不返回任何系统信息
```

#### M2：查询注入防护

```gherkin
场景: AI 通过查询命令检索数据库
  假如 AI 执行数据查询命令
  当 查询条件中包含特殊字符或 SQL 关键字
  那么 系统应使用参数化查询（Prepared Statement），禁止直接拼接用户输入到 SQL 语句中
  并且 如果提供 SQL 模式查询，应限制可用的 SQL 操作为 SELECT，禁止 INSERT/UPDATE/DELETE/DROP 等写操作
  并且 对单次查询返回的结果数量设置硬上限（最多 1000 条），防止全表扫描导致性能问题
```

#### M3：批量写操作安全上限

```gherkin
场景: AI 需要连续执行大量写操作（如批量发送私信）
  假如 AI 在一次任务中连续发出多个同类型写操作命令
  当 同一类型写操作的累计次数接近或超过安全上限
  那么 系统应在达到上限前发出预警（如"已发送 45/50 条私信，即将达到单日上限"）
  并且 达到上限后拒绝同类操作并返回"已达到单日操作上限"的明确错误
  并且 不同操作类型的默认上限（每配置空间独立计数，目的是保护单个账号不触发平台风控）：
    - 私信：50 条/日
    - 评论：100 条/日
    - 发布笔记：20 篇/日
  并且 用户可通过配置文件调整这些上限（需手动确认了解风险）
  并且 上限计数器每自然日零点重置
```

#### M4：配置空间数据的文件系统保护

```gherkin
场景: 保护配置空间中的登录状态和采集数据不被其他程序轻易读取
  假如 系统创建工作目录和配置空间目录
  当 目录和文件被创建时
  那么 工作目录和配置空间目录的权限应设置为仅当前用户可读写（Unix: 700，Windows: 对应 ACL）
  并且 数据库文件权限设为仅当前用户可读写（Unix: 600）
  并且 临时文件（如截图、导出文件）在使用完毕后应由 AI 决定保留还是清理
```

#### M5：恶意网页隔离

```gherkin
场景: 浏览器访问了一个试图干扰插件行为的恶意网页
  假如 插件在页面中注入了 Content Script
  当 恶意网页尝试通过 DOM 操作或消息伪造干扰插件
  那么 插件的 Content Script 应运行在 Chrome 的隔离执行上下文中（Isolated World），与页面 JavaScript 互不可见
  并且 Content Script 与 Service Worker 之间的消息传递应包含不可预测的内部标识，忽略不含有效标识的消息
  并且 插件不应向页面上下文暴露任何可被检测到的全局变量或 DOM 修改
```

#### M6：敏感数据管理

```gherkin
场景: 系统采集到了包含个人信息的数据（如用户资料、私信内容）
  假如 数据库中存储了他人的个人信息
  当 数据超过保留期限（默认 90 天）且未被标记为"保留"
  那么 自动清理流程应将其删除（同 D3 数据清理逻辑）
  并且 清理时应同步删除关联的媒体文件
  并且 系统应在首次初始化时向用户提示："本工具采集的数据仅供个人合法用途，请遵守相关法律法规和平台使用协议"
```

#### M7：操作审计日志

```gherkin
场景: 用户需要事后审查系统执行了哪些操作
  假如 系统在运行过程中执行了各类写操作
  当 用户或 AI 查看审计日志
  那么 所有写操作（发私信、发评论、发布笔记等）应记录：操作类型、目标标识、执行时间、使用的配置空间、执行结果（成功/失败及原因）
  并且 审计日志与采集数据分开存储，不受数据清理命令影响
  并且 审计日志保留期限为 180 天，超期自动清理
  并且 AI 可通过命令查询审计日志（只读，不可修改或删除）
```

#### M8：macOS 辅助功能权限引导

```gherkin
场景: 用户首次使用安全模式（OS 级输入模拟）在 macOS 上运行
  假如 用户将执行策略设置为安全模式
  当 系统尝试执行 OS 级鼠标或键盘模拟
  那么 系统应先检测当前是否已获得 macOS 辅助功能权限
  并且 如果未授权，返回明确的授权引导步骤（"请在系统设置 > 隐私与安全性 > 辅助功能中允许 WebEnvoy"），而非返回通用的权限错误
  并且 权限授予后自动验证可用性并返回确认
  并且 Windows/Linux 上如需类似权限，也应提供对应的引导信息
```

### 场景组 N：规则引擎与热更新

#### N1：加载与应用平台规则

```gherkin
场景: 系统启动时加载平台的适配规则
  假如 每种支持的平台都有独立的规则配置文件（包含 API 匹配规则、数据映射规则、DOM 选取器和限流水位等）
  当 CLI 启动并加载配置空间
  那么 系统应读取本地的平台规则库
  并且 内置插件根据规则库中的特征规则开启网络监听，而无需在底层代码硬编码任何平台特定的 API 地址
```

#### N2：规则库存量内容的独立热更新

```gherkin
场景: 目标平台的 API 发生变更导致专用工具失效
  假如 平台 X 修改了搜索接口的返回 JSON 结构或 API 请求地址
  当 系统的数据目录中的平台规则文件被更新（无论是手动覆盖还是自动拉取）
  那么 系统无需重启或重新编译核心组件，即可在下一次执行指令时应用新的规则
  并且 成功恢复对该平台的数据采集与特种操作能力
```

#### N3：规则库的在线订阅与自动同步

```gherkin
场景: AI 期望系统始终保持对各大平台的兼容性
  假如 用户在配置中启用了"同步远程规则库"功能，并配置了规则源地址（如 GitHub 仓库或官方服务器）
  当 CLI 引擎启动，或距离上次检查由于系统调度已满配置的更新周期（如 24 小时）
  那么 系统应在后台无感向远程源检查是否有新版规则
  并且 如果发现更新，系统将安全的将远端差异规则合并更新至本地仓库
  并且 该同步与更新过程不应该阻塞和干扰此时用户与 AI 正在执行的浏览器会话，更新会在下一次触发工具或新建会话时热生效
```

## 三、约束与边界

### 不做的事

1. **核心逻辑不依赖云端**：浏览器的自动化控制、拟真执行和本地数据存储等核心组件均在用户的物理机器上运行。除可选的平台规则库订阅/更新等元数据同步功能外，不提供且不强制依赖任何中心化的远程服务器
2. **不自动破解验证码**：遇到验证码时暂停并请求人工介入
3. **不存储用户密码**：登录过程由用户在可见浏览器中手动完成
4. **不编排行为链**：WebEnvoy 只负责单个动作的拟真执行，操作之间的顺序、间隔、穿插浏览等行为策略由 AI Agent 自行编排
5. **不主动构造 API 请求**：不伪造请求签名去直接调用平台内部接口，数据采集以拦截浏览器自然产生的流量和缓存为主

### 支持的运行环境

- macOS、Windows、Linux
- Chrome / Chromium 浏览器

## 四、术语表

| 术语 | 含义 |
|---|---|
| 配置空间 | 一个独立的浏览器数据目录，包含登录状态、偏好设置等，不同配置空间互不影响 |
| 平台规则库 | 独立于核心代码的外置配置文件集合序列，定义不同平台的网络拦截特征、限流规则及数据解析映射等，支持热更新 |
| 专用工具 | 针对特定平台（如小红书）深度适配的命令，返回高质量的结构化数据 |
| 兜底工具 | 适用于任何网页的通用命令，通过分析页面结构来完成操作 |
| 渐进式披露 | 先返回摘要，AI 按需再请求细节，避免一次性返回过量信息 |
| 冷却等待 | 系统主动降速，在两次操作之间插入合理的等待时间以规避风控 |
| 执行策略 | 系统执行浏览器操作的引擎档位：安全模式（OS 级模拟）、标准模式（插件执行）、效率模式（CDP 协议） |
| 通信通道 | CLI 与浏览器/插件之间传递消息的网络或进程间协议（如 Native Messaging, WebSocket） |
| 感知层 | 系统理解页面内容的方式（如解析 DOM、提取无障碍树、拦截网络请求） |
| 被动观察 | 通过拦截浏览器自然产生的网络请求和缓存来获取数据，而非主动构造请求 |
| 无障碍树 | Accessibility Tree，屏幕阅读器使用的精简语义结构，能大幅压缩页面信息并保留核心语义 |
| 生命周期状态 | 浏览器实例从创建到销毁经历的一系列状态（未启动/启动中/就绪/执行中/已暂停/关闭中/异常断开），状态之间有严格的合法转换路径 |
| 串行执行 | 同一浏览器实例在同一时刻只能执行一条操作命令，新命令须等待前一条完成后才可执行；查询类命令不受此限制 |
| 配置空间锁 | 通过锁文件实现的互斥机制，确保同一配置空间同一时刻只能被一个 CLI 进程使用，防止多进程竞争导致数据损坏 |
| 命令路由 | 多浏览器实例同时运行时，CLI 根据命令中指定的配置空间标识，将命令分发到对应的浏览器实例 |
