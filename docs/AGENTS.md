# WebEnvoy 研发协作与需求管理规范 (For Agents & Humans)

本文档旨在为参与 WebEnvoy 项目的 AI Agents（如 Cursor, Claude Code, Codex, Antigravity 等）及人类开发者提供统一的需求管理与协作规范，确保项目在高速迭代下保持「需求清晰」与「进度一致」。

## 核心心法：蓝图 vs 流水线

*   **本地文档 (`docs/FR.md`, `docs/PRD.md`) 是产品的「蓝图」**：它描述的是系统完整的架构形态、功能全景和验收标准。
*   **远端库 (GitHub Issues & Projects) 是工地的「流水线看板」**：它描述的是当前的开发进度、任务分配和具体的落地状态。

---

## 一、何时改文档？何时改 Issue？

### 场景 1：需求"内容"发生变化 ➡️ 永远优先修改本地文档
**触发条件：**
* 需要新增一个新的功能模块或验收条件。
* 现有设计的实现路径被推翻，需要修改需求细节。
* 在评估后决定取消某个功能。

**操作规范：**
1. **优先更新本地**：在遇到业务或逻辑变更时，人类/AI 必须优先在本地代码库中修改 `docs/FR.md` 或 `docs/PRD.md` 的内容。
2. **提交与记录**：修改完成后提交代码（如 `docs: 移除 ACT-09 鼠标轨迹拟真需求`）。
3. **绝对底线**：不要试图仅在 GitHub Issue 的评论里讨论并修改产品逻辑，而不将定稿同步到本地文档。本地文档必须始终反映真实需要构建的最终形态，它是 AI Agent 每次启动工作时的唯一宪法。

### 场景 2：需求"进度状态"发生变化 ➡️ 永远只在 GitHub 流转
**触发条件：**
* 准备开始编写某个特定需求的代码。
* 遇到技术阻塞（Blocker），需要暂停并记录原因。
* 代码编写完成并准备合并进主干。

**操作规范：**
1. **状态流转**：在 GitHub 看板（Projects）上流转 Issue 卡片状态（如 `Todo` -> `In Progress`）。
2. **自动闭环**：在提交合并代码 (Pull Request / Commit) 时，必须使用 GitHub 的关联语法（如 `Fixes #23`、`Resolves #45`），使得代码合入时自动关闭对应的 Issue 变成 `Done`。
3. **隔离进度**：不要在本地 `FR.md` 中手动添加“已完成/待开发”这类进度标记字段。本地文档不需要充当进度的打点工具，保持其干净和纯粹的需求属性。

## 二、双端数据自动化同步机制设计

为了避免双头维护的痛苦和产生割裂状态，项目采用**「本地驱动远端 (Docs to Issues)」**的机器同步更新策略：

### 1. 唯一事实来源 (SSOT)
`docs/FR.md` 中的需求列表和表格是项目所有被编号（例如 `ACT-01`、`ENV-02`）功能的唯一基准来源。远端的所有 Issue 都是由它衍生且依附于它的影子。

### 2. 同步工作流策略
项目将引入同步脚本（如构建基于 Node/Python 的 CLI 工具或 GitHub Actions）来缝合这两个世界：
*   **增量下发建站**：当在本地 `FR.md` 新增了一行如 `[XXX-01]` 时，同步脚本会自动解析并调用 GitHub API 创建对应的待办 Issue，同时解析原表格附带对应的 Priority Tag（如 `P0`, `P1`）。
*   **内容单向映射更新**：如本地 `FR.md` 中对应 `XXX-01` 的描述被修改了，脚本将单向覆盖远端 GitHub Issue 的正文描述，但绝不会去干扰它在看板上的已流转状态和历史讨论。
*   **状态淘汰与废弃**：如本地蓝图中直接删除了某个需求行，远端关联的 Issue 将被脚本自动标记打上 `Obsolete` (已废弃) 标签并关闭。

## 三、AI Agent 协同流程指引

当任何 AI Agent 需要协助开发 WebEnvoy 时，请遵循以下标准的行为框架：

1. **认知全景**：在新建会话并开始任何写代码的操作前，优先读取本地的 `docs/PRD.md` 和 `docs/FR.md` 以获取最新和最权威的系统设计理念及术语表。**不要盲猜系统架构。**
2. **领取任务**：如果配备了适当的工具权限（如 gh CLI 或 GitHub MCP），请通过读取 GitHub 上状态为 `Todo` 且分配到当前里程碑（Sprint/Milestone）的 Issue，作为本轮迭代的开发切入点。
3. **闭环提交与状态清理**：每完成一个独立代码功能模块（解决了一个 Issue）后，在 Git Commit Message 中务必显式声明关联的 Issue 编号（例：`feat(cli): Add init logic, closes #12`）。

## 四、分支管理与 PR 机制 (Branching & PRs)

作为一个由 AI 驱动快速迭代的项目，保持主干（`main`/`master`）的绝对稳定是第一要务。我们采用 **GitHub Flow** 的轻量级分支策略：

1. **绝对禁止直推主分支**：所有的开发工作（包含修改本文档、编写 `FR.md` 等）都必须在独立的功能分支（如 `dev`、`feature/xxx`、`fix/xxx`）上进行。
2. **基于 Issue 创建分支**：在开始开发一个新的需求前，从最新的保护分支拉取新的开发分支。推荐的分支命名格式为 `类型/Issue编号-简短描述`（例如：`feat/42-init-workspace`）。
3. **中文与规范化提交 (Conventional Commits)**：提交信息（Commit Message）**必须使用中文**，且符合 Angular 规范（如 `feat: 新增核心环境初始化 logic (fixes #12)`、`docs: 更新 API 文档`）。这有助于项目文档的统一并在日后自动生成可读性高的 Changelog。
4. **通过 PR 合并**：开发完成及本地验证通过后，通过向目标开发/集成干道提交 Pull Request (PR) 来合并代码。不接受任何通过 Git CLI 直接进行的 `--force` 覆盖强推。
5. **主干合并必须 Squash (压缩合并)**：合入主分支（`main`/`master`）的操作必须采用 **Squash and Merge** 策略。将功能分支上散落、琐碎的开发足迹（如 `fix typo`、`update info`）压缩为一条干净整洁并附带详细说明的记录，这是保证主干分支不被“提交垃圾”堵塞的核心法则。
6. **CI 拦截与验收**：PR 必须通过自动化 CI 检查（如有）并在人类开发者确认（Code Review）后才能合并入主干。这样可以确保即使 AI 产生了幻觉或错误代码，也不会立刻污染可用版本。
