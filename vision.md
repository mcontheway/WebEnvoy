# WebEnvoy 愿景 (Vision)

---

## 一、 一句话 (Elevator Pitch)

**WebEnvoy 是专为 AI Agent 打造的高速、高隐匿的无人值守 Web 执行引擎**，让 Agent 像调用本地函数一样无障碍地畅游整个互联网（包括稳定穿透最严苛的高防平台风控），完成结构化数据抓取与复杂交互——而不是靠视觉截图低效漫游页面。

## 二、 问题陈述 (The Problem)

当前的 AI Agent（如基于浏览器/多模态视觉的自动化方案）在尝试无障碍畅游真实商业 Web 环境时，往往会遭遇两座难以逾越的大山——碎片化的长尾 DOM 结构与严密的反爬风控，显得极其脆弱、低效且昂贵：

1.  **速度与成本的灾难**：让大模型通过"获取无障碍树/截图 -> 推理按钮位置 -> 等待页面加载 -> 再次验证"的闭环操作网页，速度极慢，且会海量吞噬宝贵的 Token 上下文窗口。
2.  **脆弱的系统稳定性**：强依赖目标前端的 UI 结构，一旦网页改版或元素位移，基于视觉坐标或硬编码选择器的脚本将立即崩溃。
3.  **极高的风控拦截率与账号损耗**：直接调用传统的无头浏览器（Headless Browser）或 CDP 原生协议，极易触发目标网站（如小红书、抖音、Cloudflare）的风控探针，被识别为机器人并导致封号。
4.  **成功经验难以沉淀与复用**：现有的自动化工具往往停留在“一次跑通”，缺乏把成功路径快速封装、持续维护、分享导入的能力。网站一旦改版、接口一旦变化，用户往往只能重新摸索、重写脚本，难以把网页操作真正沉淀成长期可用的能力资产。

随着大语言模型能力的爆发，AI 智能体与真实互联网交互的**行动力（Action Capability）不足**已成为制约其落地的核心瓶颈。

## 三、 项目愿景 (The Vision)

WebEnvoy 的目标是将整个真实互联网（从长尾普通站点到高防封闭生态）的复杂交互，收敛为 AI Agent 可直接调度的**结构化执行接口**，并将执行链路的完整控制权交还给上层智能体。

我们致力于实现：**AI 只需表达业务意图，WebEnvoy 负责在全网任意节点择优执行并实时反馈链路细节。** 我们通过提供分层执行策略，赋予大模型真正「无障碍畅游」互联网的行动力。

### 愿景分层与兑现顺序

为避免把终局能力误当作首阶段主线，WebEnvoy 的愿景按以下层级逐步兑现：

1. **第一层：先成为 Agent 真正好用的上网工具（当前主线）**
   先把最核心的体验做对：上层 AI 只管说清目标，WebEnvoy 就能稳定地去网页里读取、触发必要的基础交互，并把结果清楚地带回来。它在路线图上的第一步，是先把“读取做稳、最小交互可验证、错误能看清”这件事做成；填写、提交等更完整的写入闭环，会在后续阶段继续补齐。用户首先要感受到的是“它真的能替我上网做事”，而不是一堆复杂配置。
2. **第二层：让用户能很快把一次成功经验封装起来（当前主线）**
   当用户在一个网站上跑通了一次流程，不应该只能靠记忆或临时脚本复用。WebEnvoy 应该让用户像整理 Skills 一样，把这次成功经验快速封装成一个可反复使用的小能力，留给自己下次直接调用。
3. **第三层：让这些能力真正能交付、能分享、能持续可用（近期主线）**
   用户不只需要“做出来”，还需要“拿得出去、用得住”。别人导入后能直接使用，网站变化时能尽快发现，出问题时知道哪里坏了，升级后知道会不会受影响，必要时还能快速修回来。WebEnvoy 在这一层交付的是可用能力，而不只是一个文件。
4. **第四层：形成分享、导入、管理能力的使用网络（后层能力）**
   最终，WebEnvoy 不只是一个人自己用的工具，而是一个能力可以被分享、导入、更新、管理的网络。用户可以保留自己的私有能力，也可以导入别人分享的能力；官方可以提供示例和打样，但重点不是“官方维护全部能力”，而是让用户之间的能力流动真正成立。

### 与上层系统的关系

WebEnvoy 的长期定位，是成为可被其他系统调用的网页执行底座。

- 身份管理、账号矩阵、任务调度、风控运营等产品可以把 WebEnvoy 当作底层网页执行引擎来调用
- WebEnvoy 自己不以同时承载这些上层系统职责为目标

### 当前阶段的集成方式

在当前阶段，WebEnvoy 默认以 **CLI 工具** 的形态提供集成能力。

- CLI 本身就是最天然的第一层集成面：跨语言、低门槛、适合 Agent、脚本、工作流与其他产品直接调用
- 当前阶段**不以单独封装 SDK / API 作为主线交付**。在网页执行能力、能力封装与能力交付机制尚未稳定前，过早包装 SDK / API 只会放大未稳定边界
- 这并不意味着 CLI 可以随意设计。相反，CLI 必须按“可被上层系统可靠调用”的标准来设计：命令语义稳定、输入输出机器可读、错误结构化、事件可追踪、运行标识清晰
- 只有当未来出现明显的长期集成压力，例如高频调用、长会话复用、实时事件流、复杂并发调度时，再评估是否演进出本地服务、RPC 或其他更重的集成方式

### 当前阶段的产品策略

WebEnvoy 当前虽然以执行底座为主线，但这不意味着“先做基础设施、以后再考虑用户价值”。相反，当前阶段必须同时满足两条约束：

- **底层先行**：优先把网页执行、身份 / 会话承载、观察 / 诊断、能力沉淀这些长期护城河做稳
- **价值可感知**：每个阶段都必须交付用户能明确感受到的小闭环，而不是只在内部完成一组底层模块

这意味着 WebEnvoy 当前阶段的产品判断应固定为：

- 不追求先把产品面铺得很宽，再回头补治理
- 不把“支持更多命令 / 更多站点”作为当前阶段的主要成功标准
- 每个 Phase / Sprint 都必须回答：用户这轮新获得了什么明确价值
- 当前阶段的产品节奏应是：**先把一个闭环做稳，再逐步扩大覆盖面**

### 当前阶段的差异化判断

WebEnvoy 当前最值得押注的差异化之一，不只是“能执行网页动作”，而是：

- 在真实商业 Web，尤其是高防、高价值平台上，仍然具备足够的拟人性、隐身性、稳定性和可持续使用能力

这项差异化的重要性不在于“技术更炫”，而在于它直接决定：

- 用户和 Agent 是否敢把关键流程长期托付给 WebEnvoy
- 工具能否从“偶尔跑通”的实验品，升级为“可进入生产环境”的执行能力
- 一次成功经验能否在高风险场景下被重复使用，而不是频繁因风控和环境漂移失效

因此，高防与生产级可持续执行能力应被视为 WebEnvoy 的核心竞争力之一，而不是附加卖点。

### 当前阶段的能力策略

WebEnvoy 的目标不是成为“官方维护海量适配器的命令仓库”。当前阶段更合理的方向是：

- 官方优先提供 **协议**
  - 定义什么样的能力可以被保存、验证、重放、升级和交付
- 官方优先提供 **工具**
  - 帮助用户把一次成功经验更快整理成候选能力，并完成验证、诊断与重跑
- 官方优先提供 **治理机制**
  - 让用户知道什么只是草稿，什么已经达到正式能力的准入门槛
- 官方优先为未来 **社区生态** 预留空间
  - 让用户、团队和社区能够共享能力，而不是把增长完全压在官方适配器产能上

换句话说，WebEnvoy 当前阶段更应该成为“帮助用户和社区生产、验证、维护能力的平台”，而不是“官方自己做完全部能力的产品”。

### 最小身份 / 会话能力的边界

当前仓库只保留**为了完成一次可靠网页执行而必需**的最小身份 / 会话能力。

- 包括：指定执行身份、恢复既有登录态、在执行期间保持隔离、避免同一身份被并发抢占、在执行结束后回传运行状态
- 不包括：账号池调度、长期养号、跨身份任务编排、健康度系统、配额策略、矩阵运营等独立上层系统能力

**我们的核心系统哲学：**

1.  **全网漫游的三层引擎兜底（L3/L2/L1）**：我们提供完整的读写闭环，不是只为高防平台服务，而是为「畅游互联网」服务。采用 **L3 / L2 / L1** 三层执行模型。**L3**（专用引擎，极速）：针对高防阵地或高频站点，在平台页面内直接发起 API 请求或直读内存状态（`window.__INITIAL_STATE__`）；**L2**（通用引擎，通用稳健）：针对海量长尾站点，通过 AX Tree 语义定位 + 合成事件通用读写，不依赖特定 API 适配；**L1**（物理层兜底，最高保真度）：VLM 视觉定位 + CDP/OS 级物理输入，应对纯 Canvas 界面或 DOM 手段失效的极端场景。始终以满足意图所需的**最低代价**通用或专用路径作为首选。
2.  **链路完全透明与深度介入（核心竞争力）**：拒绝黑盒化。WebEnvoy 开放网络层与 DOM 层的**侦察与调度原语**。允许主控 AI 像指挥特战小队一样，实时监控交互节点（如流量特征、JS 变量、风控探针），并在关键时刻执行微观干预（如修改请求指纹、拦截异常上报），而不是死等成功或失败的最终结果。
3.  **从高防攻坚反哺通用漫游**：我们对高防平台的特殊关注，是为了给大模型扫清最棘手的障碍。通过对头部高防平台的深度行为分析，我们沉淀出最顶级的抗检测工程底座（指纹混淆、行为建模、长尾操作分布），然后将这些降维打击能力赋能给 L2/L1 通用执行层，使得 WebEnvoy 在任何普通网站上运行都具有碾压级的安全稳健性。
4.  **能力可沉淀、可分享、可持续维护**：WebEnvoy 不仅要让用户在未知网站上先做成一次，还要让这次成功经验能被快速留下、反复调用、分享给别人，并在网站变化后仍然有办法继续维护。
5.  **边界清晰，先把该做的事做好**：身份管理、账号矩阵、任务调度、长期运营等能力有其独立价值和独立复杂度。WebEnvoy 应优先把网页执行这一件事做到足够强、足够稳，并为上层系统留下清晰的集成边界，而不是把所有问题都包进自己。
6.  **先把 CLI 做成可靠契约，再决定是否升级集成形态**：当前阶段默认以 CLI 作为集成面，不急于承诺 SDK / API；但 CLI 必须具备稳定语义、机器可读输出和结构化错误，能够被上层系统可靠调用。
7.  **每一阶段都必须让用户感受到新价值**：WebEnvoy 不接受“长期埋头做基础设施，用户无感”的推进方式。每个阶段都必须交付一个用户能明确感知的小闭环，并用这个闭环反向约束 Sprint 范围与优先级。
8.  **官方优先做协议、工具和治理，不以适配器数量取胜**：WebEnvoy 的增长不应建立在“官方内置尽可能多的网站能力”之上，而应建立在“帮助用户和社区更高效地产出、验证、诊断、升级能力”之上。

## 四、 核心原则与价值观 (Core Principles / Values)

当我们在架构设计或功能开发面临取舍时，应遵循以下最高原则：

1.  **效率优先**：以最小代价（Token 消耗、推断时间）换取最精确的结构化结果。抵制会滥用上下文的重型通信模式。
2.  **先把重点平台做深，也要让长尾网站能快速变成现成能力**：先打通高价值平台，保障高成功率；面对未知网站，先让用户获得首次可用性，再尽快把成功经验沉淀成可复用能力。不为形式上的"通用化"牺牲实战可用性，也不把一次性成功停留在一次性。
3.  **链路完全可控**：系统不是黑盒。主控 AI 或开发者必须能在链路各节点（网络层、DOM 层）介入侦察和调度。
4.  **交付用户可直接拿来用的能力**：我们的目标不只是证明“技术上可行”，而是让用户能把能力留下、分享、导入、升级，并在网站变化后继续用下去。
5.  **集成优先从轻，不从重**：当前阶段优先把 CLI 做成可靠的集成契约，而不是提前扩张到 SDK / API / 服务化体系；只有在真实使用压力出现后，再升级集成形态。
6.  **账号安全与痕迹消除**：对于高风险平台，必须尽可能保障用户身份与会话安全，深度抹除自动化痕迹（指纹特征、行为规律等），将降低风控风险视为与任务成功率同等重要的核心指标。
7.  **边界清晰，能力分层**：WebEnvoy 专注网页执行、能力沉淀与能力流动；身份管理、矩阵调度、长期运营等上层问题，应通过清晰边界与集成方式解决，不在当前产品中混成一团。
8.  **代码资产 100% 自主**：可借鉴开源竞品的攻防思路与业务数据字典，但核心机制必须用自主技术栈重写，确保合规与独立。
9.  **差异化要落在“可托付”而不是“看起来功能多”**：面对 OpenCLI、agent-browser 等产品，WebEnvoy 的优势应建立在真实商业 Web、尤其是高防场景下的拟人性、隐身性、稳定性与可持续使用能力，而不是单纯依赖功能面扩张。

## 五、 目标受众 (Target Audience)

*   **主要用户**：构建 AI 智能体应用、自动化工作流或网页操作能力的开发者与产品团队。他们需要一个稳定、不被风控轻易阻断、能把网页操作变成可复用能力的底层工具。
*   **次要用户**：需要把某个网站流程快速封装起来、私有使用、团队共享或持续维护的个人用户、小团队与运营角色。
*   **延伸用户**：构建身份管理、账号矩阵、任务调度、风控运营等上层系统的团队。他们不是来让 WebEnvoy 替自己完成全部系统建设，而是把 WebEnvoy 作为网页执行层接入自己的产品。

## 六、 非目标 (Non-Goals / Out of Scope)

划定清晰的边界，防止需求无止境的"范围蔓延"（Scope Creep）。**以下内容我们坚决不做**：

1.  **我们不做自主决策的 Agent（核心底线）**：WebEnvoy 将自身的定位严格界定为供智能体调用的"外挂执行工具（Tool）"。所有的任务规划、意图理解判断、长篇图文润色等复杂逻辑推理都应交由外部的 AI 大脑去完成，我们只解决"如何最快、最稳地跨越网络风控并送回极简干净结构化数据"的苦差事。
2.  **我们不造大模型或聊天前端**：WebEnvoy 不提供面向终端大众的 Chatbot 交互界面。我们只做大语言模型背后的那双"执行系统的手"。
3.  **我们不把“多账号管理平台”直接做进当前产品里**：身份管理、账号矩阵、任务调度、长期运营是独立问题域。WebEnvoy 会为这些系统提供执行底座与集成边界，但不把它们全部内化成当前仓库的主线交付。
4.  **我们不盲目追求虚幻的"100% 零配置的通用化"**：面对高频高防护的头部阵地，我们承认并积极维护特定站点的适配方案，绝不为了追求形式上的通用而牺牲 95% 以上的实战可用性。

## 七、 成功标准 (Success Criteria)

任何 AI Agent 或开发者在面临架构取舍时，可以用以下标准来衡量 WebEnvoy 是否走在正确轨道上：**在任意互联网目标（从长尾普通站点到主流高防平台）上，AI Agent 能以调用本地函数的成本完成读写闭环，且成功率、速度与账号安全性均达到工业级生产标准。**

按阶段衡量时，应至少区分以下里程碑：

1. **先变成好用工具**：Agent 已能稳定调用 WebEnvoy 完成网页读取与必要的最小交互，结果清楚，出错可查，失败可定位。
2. **再变成可复用能力**：用户可以把一次成功经验快速留下，自己反复使用。
3. **再变成可交付能力**：这些能力可以分享、导入、验证、升级，并在网站变化后快速修复。
4. **再变成能力网络**：能力能够在用户之间流动，形成持续分享、导入、管理与演进的网络。

具体阶段划分与交付计划，请参阅 [`docs/dev/roadmap.md`](./docs/dev/roadmap.md)。
