## 协作原则

欢迎参与 WebEnvoy 的开发。为了保持架构演进与需求进度的清晰，且控制 AI 的上下文载入量（渐进式披露），请严格遵守以下协作纪律。

### 核心法则

1. **认知全景在前**：在开始任何工作之前，请结合任务目标，主动查阅 `docs/PRD.md` 和 `docs/FR.md` 获取最新的需求定义。
2. **唯一事实来源 (SSOT)**：所有关于"要做什么/做成什么样"的定义，永远以 `docs/FR.md` 和 `docs/PRD.md` 为唯一基准。任何需求的增删改，必须先体现在对这两个本地文档的修改上。
3. **隔离状态与蓝图**：**绝不要**在本地 Markdown 文档中记录"开发中/已完成"等进度状态。进度流转仅通过系统远端（GitHub Issues/Projects）和 Commit Message（如 `fixes #12`）来进行。
4. **分支与 PR 机制**：**严禁直推主分支**！所有的需求开发和文档修改必须通过拉取独立的分支（如 `dev`、`feat/xxx`）完成，并通过 Pull Request 合入主干。
5. **持续整洁的提交线**：提交信息（Commit Message）**必须使用中文且符合 Conventional Commits 规范**。向主分支合入 PR 时，必须采用 **Squash（压缩）合并**，以保持主干历史日志的绝对清晰与干净。

> 💡 **完整的工作流与双端自动化同步机制指引**，请务必在需要时查阅这篇完整文档： 👉 [docs/AGENTS.md](./docs/AGENTS
